**[BrowserTab.tsx:306-321] `"failed"` event from backend is silently swallowed**

The Electron main process (`browser-views.ts:139-145`) sends three `event` values on the `browser:page-load` channel: `"started"`, `"finished"`, and `"failed"`. The BrowserTab IPC listener (`BrowserTab.tsx:306-321`) only handles `"started"` and `"finished"`. The `"failed"` branch is entirely absent — the `did-fail-load` signal is received but discarded.

The practical effect: when a page fails to load (DNS failure, connection refused, HTTP 5xx that triggers Chromium's error page, etc.) the tab stays in `loading: true` forever. The full-screen loading spinner never clears. The user sees an infinite Loader2 with no error message and no way to know the page failed. The `tab.error` field is never set, so the error overlay (`BrowserTab.tsx:744-764`) never renders.

Fix: add an `else if (eventType === "failed")` branch that sets `{ loading: false, error: data.error?.description ?? "Page failed to load" }` and clears `completingLoad`.

### 6. **Workspace Layout Store (Persistence)**

File: `/apps/web/src/features/workspace/store/workspaceLayoutStore.ts`

**Browser-Specific State:**
```ts
interface WorkspaceLayoutState {
  browserTabs: PersistedBrowserTab[];     // Array of { id, url, title }
  activeBrowserTabId: string | null;     // Which tab was last active
  // ... other content tabs, selected file, sidebar state, terminal, simulator state
}
```

**Persistence Flow:**
1. BrowserPanel maintains local React state for tabs
2. On tab change/creation/URL navigation, debounced 300ms persist to layout store
3. `workspaceLayoutActions.setLayout(workspaceId, { browserTabs, activeBrowserTabId })`
4. Store persists to localStorage automatically (via Zustand persist middleware)
5. On workspace switch: BrowserPanel clears old webviews synchronously before loading new tabs
6. New tabs hydrated from persisted state via `hydratePersistedTab()`

**Key Lifecycle:**
- Tabs: Only URL + title persisted (not history, not webview state)
- Webviews: Destroyed on unmount, recreated on remount with new BrowserView instances
- Session partition: Each webview gets isolated partition `persist:browser-${label}` for cookies/storage

### 9. **Element Inspection Flow**

**User Action:** Click inspect/selector button in nav bar

1. **BrowserTab.tsx** calls `tabRefs.current.get(tab.id)?.toggleElementSelector()`
2. **BrowserTab (forwardRef)** injects `INSPECT_MODE_SETUP` IIFE into webview
3. IIFE sets up `window.__deusInspect` object with:
   - `enable()` — Listen for clicks, return element metadata on select
   - `disable()` — Stop listening
   - `isActive()` — Check status
   - `drainEvents()` — Return buffered events as JSON

4. **Inspect event drain loop** (every 500ms):
   - Executes `INSPECT_MODE_DRAIN_EVENTS` JavaScript
   - Gets buffered `ElementSelectedEvent` objects
   - Dispatches `onElementSelected(tabId, event)` callback

5. **BrowserPanel.tsx** receives event:
   - `handleElementSelected()` serializes element metadata
   - Calls `chatInsertActions.insertElement(workspaceId, { ref, tagName, path, styles, props, attributes, innerHTML, reactComponent })`
   - Chat input receives and inserts formatted element info

**Element Metadata Captured:**
- `tagName`, `id`, `className`, `innerText`, DOM path
- `rect` (position/size), `styles` (CSS vars for local, visual props for external)
- React component name, file, line number
- Whitelisted HTML attributes (data-testid, href, type, role, aria-label)
- Shallow innerHTML (capped 500 chars)

### 10. **Detached Window Architecture**

**Detached Window Store:**
- `useBrowserWindowStore` tracks `detachedWindowOpen` + `detachedWorkspace` context
- Only one physical detached window at a time (runtime state, not persisted)

**Detached Window Lifecycle:**

1. User clicks "Detach" button in BrowserTabBar
2. `useBrowserDetach().detach()`:
   - Marks as detached in store
   - Calls `native.browserViews.createDetachedWindow()` with URL params
   - Sends `BROWSER_WORKSPACE_CHANGE` IPC to new window

3. New window loads `/` with `?window=browser-detached&workspaceId=...` params
4. `App.tsx` detects param and renders `DetachedBrowserWindow` component
5. `DetachedBrowserWindow`:
   - Parses workspace context from URL params
   - Renders BrowserPanel full-screen
   - Listens for `BROWSER_WORKSPACE_CHANGE` to sync workspace switches
   - Bridges `chatInsertStore` payloads via `CHAT_INSERT` IPC back to main window

6. When user closes detached window:
   - Main process sends `BROWSER_DETACHED_CLOSED` IPC
   - `useBrowserDetach` effect resets detached state
   - Main window UI switches back to showing BrowserPanel inline

## Critical Issues

### [apps/desktop/preload/index.ts] `browser_view_exists` missing from `ALLOWED_INVOKE_CHANNELS` — the entire view parking recall is dead

The core innovation of this rewrite — parking native WebContentsViews alive across workspace switches and recalling them without a page reload — is non-functional in production.

`createWebviewIfNeeded` in `BrowserTab.tsx` (line 134) calls `native.browserViews.viewExists(webviewLabel)`, which resolves to `invoke("browser_view_exists", { label })`. The generic `invoke()` bridge in the preload checks `ALLOWED_INVOKE_CHANNELS` before forwarding to the main process. `"browser_view_exists"` is registered as a handler in `browser-views.ts` (line 582) but is absent from the allowlist set.

The call rejects with `IPC invoke channel "browser_view_exists" is not allowed`. The `catch {}` at line 148 of `BrowserTab.tsx` silently swallows the rejection and falls through to `create()`, which always creates a fresh view with a full page reload. The parking recall is dead code.

Fix: add `"browser_view_exists"` to `ALLOWED_INVOKE_CHANNELS` in `apps/desktop/preload/index.ts` alongside the other browser webview channels.

### [apps/web/src/features/browser/ui/BrowserPanel.tsx:202] Workspace switch effect reads stale `tabs` and `activeTabId` from closure — silently persists wrong tabs

The `useEffect` that runs on `workspaceId` change (line 191) captures `tabs` and `activeTabId` from the render closure at the time of the last render. The deps array is suppressed to `[workspaceId]` via `eslint-disable`. This is intentional for the workspace-swap logic, and is correct under normal conditions because React commits current state before the effect fires.

However: between `setTabs(newTabs)` at line 244 and the next render completing, if a tab operation (e.g. `handleUpdateTab` from an in-flight IPC load event) fires mid-effect, the `persistTabs` call at line 205 may write a snapshot that is one tick stale. Mitigation: snapshot `tabs` and `activeTabId` into local consts at the top of the effect body, using the ref-based copies (`tabInfoRef.current`, `activeTabIdRef.current`) which are always fresh.

## Warnings

### [apps/web/src/features/browser/ui/BrowserPanel.tsx:263] `closeTab` has `tabs` in deps — callback remakes on every tab state change

`closeTab` is a `useCallback` with `[tabs, activeTabId, persistTabs]` as deps (line 297). `tabs` is an array that changes on every `handleUpdateTab` call (title changes, URL changes, load events). This means `closeTab` is a new function reference on every page load event, causing all consumers (BrowserTabBar's `onTabClose` prop) to receive a new prop on every tick while a page is loading.

`closeTab` itself only uses `tabs.find(...)` to look up the closing tab's `webviewLabel` before the functional `setTabs(prev => ...)` form. The `tabs.find` could be moved inside `setTabs(prev => ...)` where `prev` is always fresh, removing the `tabs` dep entirely:

```tsx
const closeTab = useCallback(
  (closingTabId: string) => {
    setTabs((prev) => {
```

The "belt-and-suspenders" guard effect at line 146 reads `tabInfoRef.current` (always fresh) but is only triggered by `activeTabId` and `panelVisible` changes. ### [apps/desktop/main/browser-views.ts:319] `ipc-message` arity bug still present

`view.webContents.on("ipc-message", (_event, channel, data) => {...})` — the `ipc-message` event signature is `(event, channel, ...args)`. `data` captures only `args[0]`, which works for the current single-argument sends from `browser-preload.mjs`, but is formally incorrect. The 200ms fixed delay between `navigateToUrl("about:blank")` and `navigateToUrl(restoreUrl)` is a timing guess. On slower machines or large pages, the blank-page navigation may not complete in 200ms, so the restore URL loads before `about:blank` finishes. This results in the previous page's cookies being sent with the initial request of the restore load, defeating the purpose of the clear.

Fix: use a one-shot `BROWSER_PAGE_LOAD` event listener with `event === "finished"` on `about:blank` before firing the restore navigation, or expose a `reloadAfterClear` imperative method on the BrowserTab handle.

### [apps/web/src/features/browser/ui/BrowserTab.tsx:132] `createWebviewIfNeeded` could log the allowlist rejection separately from real network failures

When `viewExists` throws because the channel is not in the allowlist (the current broken state), the `catch {}` at line 148 is completely silent — no log, no indicator. ### [apps/desktop/main/browser-views.ts:594] Cookie `browser:cookies:set` handler iterates in a serial `for` loop — use `Promise.all` for parallel set

```ts
for (const cookie of cookies) {
  await ses.cookies.set(cookie);
}
```

`cookies.set()` is an async operation on the cookie store. Serial awaiting means N cookies take N × round-trip time. Use `await Promise.all(cookies.map(c => ses.cookies.set(c)))`.

### [apps/web/src/features/browser/ui/BrowserPanel.tsx] Parked view memory is unbounded

Views park on workspace switch but are never evicted (no `close()` call on switch, only `hide()`). After switching through N workspaces, N × M views remain alive in the main process `views` Map. An LRU eviction of views for workspaces not visited in the last 5 switches would bound memory use. This is a known design tradeoff per the comments, but worth tracking.

### [apps/desktop/main/browser-views.ts:690] `loadFile` in production build loses the path component of the URL

```ts
detachedWindow.loadFile(join(__dirname, "../renderer/index.html"), {
  search: url.includes("?") ? url.split("?")[1] : "",
});
```

`loadFile`'s options accept `pathname`, `search`, and `hash` separately. If `url` is `/browser?tab=x`, the `pathname` (`/browser`) is discarded — only `tab=x` is passed as `search`. The SPA router in the detached window sees no path, so it renders whatever the default route is. The dev path uses `loadURL(rendererURL + url)` which correctly includes the path. This is an asymmetry between dev and production for detached windows with path-based routes.
