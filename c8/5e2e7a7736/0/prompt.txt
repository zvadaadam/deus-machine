## Code Review: Workspace Workflow Status

**Branch:** `zvadaadam/workspace-status`
**Summary:** Adds a `status` field to workspaces (`backlog` → `in-progress` → `in-review` → `done` / `canceled`) with auto-progression logic, backfill migration, and frontend optimistic updates.

---

### Validated Issues

### **#1 Backfill runs on every app startup and overrides user-set status**

`backfillWorkspaceStatus()` is called from `initDatabase()` on every startup with no migration guard or "already ran" flag. It updates workspaces matching `status = 'in-progress'` based on heuristics (e.g., `pr_url IS NOT NULL → in-review`). Once users can explicitly set status, restarting the app silently overrides their choice. For example: user sets a workspace with a PR back to `in-progress` → next restart pushes it to `in-review` again. The `autoProgressStatus()` function has proper sticky/rank guards, but the backfill bypasses all of them via raw SQL.

**Fix:** Add a migration guard (e.g., a one-time flag in a metadata table, or move this into a versioned migration) so it only runs once.

**File:** `apps/backend/src/db/queries.ts` (backfillWorkspaceStatus) / `apps/backend/src/lib/database.ts:53`

---

### **#2 Unvalidated `status` value in WebSocket mutation allows arbitrary strings in DB**

The `updateWorkspaceStatus` mutation in `query-engine.ts` reads `status` via `readStringParam` and casts with `as WorkspaceStatus` (compile-time only). No runtime Zod validation. The HTTP PATCH route validates via `PatchWorkspaceBody` → `WorkspaceStatusSchema`, but the WebSocket `q:mutate` path does not. Any arbitrary string sent via WebSocket would be written directly to the `status` column, causing: (1) the workspace to become invisible in stats counts, (2) `STATUS_RANK[current]` returning `undefined`, making `<=` comparisons always `false` and bypassing the regression guard.

**Fix:** Add `WorkspaceStatusSchema.parse(status)` before calling `setWorkspaceStatus`.

**File:** `apps/backend/src/services/query-engine.ts` (lines ~493-502)

---

### **#3 Hardcoded Tailwind color classes instead of semantic CSS variable tokens**

`WORKFLOW_STATUS_CONFIG` uses `text-amber-500`, `text-purple-500`, and `text-green-500` — raw Tailwind palette utilities. The existing `STATUS_CONFIG` in the same file consistently uses semantic tokens (`text-destructive`, `text-status-unread`, `text-primary`, `text-muted-foreground`). CLAUDE.md explicitly states: *"Don't hardcode colors: `text-green-500 dark:text-green-400` — Create semantic variables: `--color-status-success` → `text-[color:var(--status-success)]`"*

**Fix:** Define CSS variables in `global.css` (e.g., `--status-in-progress`, `--status-in-review`, `--status-done`) and reference them here.

**File:** `apps/web/src/features/sidebar/lib/status.ts` (lines ~113-119)

### Validated Issues

### **#1 Stats query adds 3 more separate COUNT subqueries instead of consolidating**

Lines 407-409 of `apps/backend/src/db/queries.ts` add 3 new `SELECT COUNT(*) FROM workspaces WHERE status = ...` subqueries to `getStats()`, bringing the total from 8 to 11 separate full-table scans in one query. CLAUDE.md explicitly addresses this:

> "Consolidate count queries -- GET /stats currently runs 8 separate COUNT(*) full table scans. Aggregate queries should combine into a single query where possible, and results should be cached with short TTL (5-10s) for frequently polled endpoints."

While the existing 8 subqueries are pre-existing, the 3 **new** ones are introduced code that worsens the violation. These three status counts could be consolidated with the existing workspace counts into a single table scan using conditional aggregation:

```sql
-- Instead of 6 separate subqueries on `workspaces`, use one:
SELECT
  COUNT(*) as workspaces,
  COUNT(CASE WHEN state = 'ready' THEN 1 END) as workspaces_ready,
  COUNT(CASE WHEN state = 'archived' THEN 1 END) as workspaces_archived,
  COUNT(CASE WHEN status = 'backlog' AND state != 'archived' THEN 1 END) as workspaces_backlog,
  COUNT(CASE WHEN status = 'in-progress' AND state != 'archived' THEN 1 END) as workspaces_in_progress,
  COUNT(CASE WHEN status = 'in-review' AND state != 'archived' THEN 1 END) as workspaces_in_review
FROM workspaces
```

The 5s TTL cache mitigates the performance impact, but new code should follow the stated consolidation rule.

**File:** `apps/backend/src/db/queries.ts:407-409`
