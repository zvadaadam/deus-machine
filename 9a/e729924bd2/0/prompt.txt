## Analysis and Implementation Plan: Speed Ramping and Time Compression

### Part 1: Root Cause of the Duration Bug

The video being 5+ minutes for a 57-second session stems from the `generateFfmpegFilter` function in `/Users/zvada/Developer/agent/box-ide/.conductor/monrovia-v1/packages/screen-studio/src/recorder/encoder.ts`, specifically line 141:

```ts
const fps = durationSec > 0 ? Math.max(1, Math.round(timeline.length / durationSec)) : 30;
```

This computes the "fps" by dividing total timeline frame count by the duration in seconds. The zoompan filter then uses this fps to interpret the input frames. Here is the problem:

**The timeline frames are captured event-driven, not at a steady frame rate.** In `session-manager.ts` line 221-230, `timelineRecorder.captureFrame()` is only called inside the `event()` method -- meaning one timeline frame per agent event. If 22 events occur over 57 seconds, that gives `fps = Math.round(22 / 57) = 0`, which gets clamped to `Math.max(1, 0) = 1`. So the zoompan filter interprets the input as 1 frame per second.

But the zoompan filter's `d=1` means "display each input frame for 1 output frame", and `fps=1` means 1 output frame per second. The raw capture runs at 10fps, producing ~570 frames over 57 seconds. Zoompan with `fps=1` then stretches those 570 input frames to 570 seconds of output. That is approximately 9.5 minutes -- which is in the ballpark of the reported issue.

There is a **fundamental mismatch**: the zoompan filter's `fps` parameter should match the raw capture's actual frame rate (10fps for stream capture), not the timeline's event-driven "frame rate". The timeline is used for camera positions only; the raw video's frame rate is independent.

**The stream recorder captures frames at a fixed rate (~10fps)** in `/Users/zvada/Developer/agent/box-ide/.conductor/monrovia-v1/packages/screen-studio/src/mcp/stream-capture.ts`, piped to ffmpeg with `-framerate 10`. The raw MP4 is encoded at 10fps. The post-processing zoompan filter should also use that same fps, but instead it computes a bogus fps from timeline frame count.

### Part 3: Speed Ramping Design

Here is a concrete design for Cursor-style speed ramping adapted to our 10fps stream capture pipeline.

#### 3.1 Concept: Render Plan

A **RenderPlan** is a list of segments, each specifying a time range in the source recording and how it should be played:

```ts
interface RenderSegment {
  /** Source time range (ms since session start). */
  sourceStartMs: number;
  sourceEndMs: number;
  /** Playback rate: 1.0 = normal, 2.0 = 2x speed, etc. */
  playbackRate: number;
  /** Whether this is an action window or compressed gap. */
  type: "action" | "gap" | "trim";
}

interface RenderPlan {
  segments: RenderSegment[];
  /** Total output duration after speed ramping (seconds). */
  outputDuration: number;
  /** Frames trimmed from start (leading dead time). */
  leadTrimMs: number;
  /** Frames trimmed from end (trailing dead time). */
  trailTrimMs: number;
}
```

#### 3.2 Render Plan Generation Algorithm

The algorithm takes the recorded events (`AgentEvent[]`) and produces a RenderPlan:

**Step 1: Identify action windows.**
For each event, create an action window: `[event.t - leadInMs, event.t + trailOutMs]` at 1x playback. Merge overlapping action windows.

**Step 2: Identify gaps.**
Gaps are the regions between consecutive action windows. Apply speed ramping to each gap:
- If gap duration < `minGapToSpeedUp` (800ms): keep at 1x
- Otherwise: compute playback rate = `gapDuration / targetGapOutputMs`, clamped to `maxPlaybackRate` (8x), with output capped at `maxGapOutputMs` (2s)

**Step 3: Trim dead time.**
- Leading dead time: everything before the first event minus `leadInMs` is trimmed entirely (rate = infinity / skip)
- Trailing dead time: everything after the last event plus `trailOutMs` is trimmed

**Config constants (matching Cursor's tuning as a starting point):**

```ts
const SPEED_RAMP_CONFIG = {
  leadInMs: 800,        // Show 800ms before each action
  trailOutMs: 600,      // Show 600ms after each action
  minGapToSpeedUp: 800, // Gaps shorter than 800ms stay at 1x
  targetGapOutputMs: 1200, // Ideally compress gaps to ~1.2s output
  maxGapOutputMs: 2000,  // No gap renders longer than 2s output
  maxPlaybackRate: 8,    // Maximum speedup factor
  leadTrimPaddingMs: 500, // Keep 500ms before very first action
  trailTrimPaddingMs: 500, // Keep 500ms after very last action
};
```

**Example for the 57-second session with 22 events:**
If events are clustered around 5 action bursts with 10s average gaps between them, the plan might produce:
- 5 action windows at 1x (~2-3s each = ~12s)
- 4 gaps compressed from ~10s each to ~1.5s = ~6s
- Leading/trailing trim: remove 5-10s
- Total output: ~18-20 seconds instead of 57 seconds (or 5+ minutes with the bug)

#### 3.3 Implementation in ffmpeg

The speed ramping needs to happen in the ffmpeg post-processing step. There are two approaches, and I recommend approach B.

**Approach A: setpts filter with segment concat (complex, reliable)**

Generate a sequence of ffmpeg commands that:
1. Split the raw video into segments using `-ss` and `-t` for each RenderSegment
2. Apply `setpts=PTS/{playbackRate}` to each segment
3. Concat all segments using the concat protocol or filter

This is precise but requires many intermediate files or a complex filter graph.

**Approach B: Single-pass setpts with a piecewise expression (recommended)**

ffmpeg's `setpts` filter supports expressions. We can build a piecewise-linear PTS mapping that applies different speeds to different time regions, all in one pass:

```
setpts='if(lt(T,gap1_start), PTS-STARTPTS,
        if(lt(T,gap1_end), (gap1_output_start + (T-gap1_start)/gap1_rate)/TB,
        if(lt(T,gap2_start), ...,
        ...
        )))'
```

However, this gets unwieldy for many segments and ffmpeg expressions have recursion limits. A better variation:

**Approach B': Pre-computed PTS map via timeline resampling**

Instead of fighting ffmpeg expressions, we resample the timeline at the raw capture's actual frame rate, applying speed ramping in our TypeScript code, then output a simple zoompan filter with the correct mapping. This is the cleanest approach and leverages code we already have.

The idea:
1. Generate the camera timeline at the raw video's fps (10fps) covering the full session duration -- this is `CameraEngine.processTimeline(10, sessionDurationSec)`. This gives us one camera position per raw video frame.
2. Build the RenderPlan from events.
3. **Resample the timeline according to the RenderPlan**: walk through source frames, skip trimmed regions, duplicate frames for slowed sections, drop frames for sped-up sections. The output is a new timeline where:
   - Action window frames are kept 1:1
   - Gap frames are subsampled (keep every Nth frame for Nx speed)
   - Trimmed regions are omitted entirely
4. Feed the resampled timeline to `generateFfmpegFilter` (or `generateCropScaleFilter`). Since frames in the resampled timeline are at the output rate, `d=1` and `fps=outputFps` work correctly.

**For the raw video frame selection**, we use ffmpeg's `select` filter to pick which input frames to keep, combined with `setpts` to fix timestamps:

```
select='between(t,0.0,2.5)+between(t,12.0,15.0)+...',setpts=N/FRAME_RATE/TB
```

This selects specific time ranges from the raw video (corresponding to action windows at full frames, gaps at subsampled frames) and reassigns timestamps so the output plays smoothly.

#### 3.5 ffmpeg Filter Strategy (Detailed)

The final ffmpeg filter_complex will chain three filters:

```
[0:v] select='FRAME_SELECT_EXPR',setpts=N/FPS/TB, ZOOMPAN_OR_CROP_FILTER [out]
```

Where:
- `FRAME_SELECT_EXPR` picks frames from the raw video based on the render plan. For action windows, pick all frames. For gaps, pick every Nth frame (where N = playbackRate). For trimmed regions, pick nothing.
- `setpts=N/FPS/TB` reassigns timestamps sequentially to the selected frames
- The zoompan/crop filter applies camera movements using the resampled timeline keyframes

The `select` filter expression pattern:

```
select='
  between(t,0.5,3.2)    # action window 1 (all frames)
 +between(t,3.2,12.0)*not(mod(n\,4))  # gap (keep every 4th frame = 4x speed)
 +between(t,12.0,15.5)  # action window 2
 +between(t,15.5,30.0)*not(mod(n\,8))  # gap (8x speed)
 +between(t,30.0,33.1)  # action window 3
'
```

However, the `select` approach with `mod(n,N)` for subsampling gaps is approximate because `n` is the global frame counter, not relative to the gap. A more reliable method:

**Alternative: concat demuxer approach**

For each segment in the render plan:
1. Extract the segment with `-ss` and `-t`
2. Apply `setpts=PTS/rate` for speed-up
3. Concat all segments

This is cleaner but requires temp files. For our use case (short recordings, ~50-100 segments max), the overhead is acceptable.

**Simplest approach (recommended for v1):**

Since we already have the resampled timeline controlling which camera positions to use, the simplest approach is:

1. Generate the camera timeline at the raw fps
2. Apply the render plan to produce a mapping: `outputFrame[i] -> sourceTimeMs[i]`
3. Use ffmpeg's `select` filter to pick source frames closest to the mapped times
4. Apply zoompan from the resampled timeline keyframes
5. Output at the desired fps

For v1, an even simpler approach: use the zoompan filter's `d` parameter per-keyframe to control how long each frame displays. If a gap frame should play at 4x speed, set `d=1`. If an action frame should play at 1x, set `d=1`. The fps of the zoompan filter controls the output rate. By dropping frames during gaps in our timeline resampling, we effectively achieve speed ramping without needing ffmpeg expressions.

**This is the recommended v1 approach:**

1. Generate full timeline at rawFps (e.g., 10fps) for the entire session
2. Walk through the render plan, building a new "output timeline" that:
   - Copies action-window frames 1:1
   - Skips N-1 out of every N gap frames (for Nx speedup)
   - Omits trimmed frames entirely
3. The output timeline has fewer frames. Compute `outputFps = outputTimeline.length / desiredOutputDuration`
4. Pass to `generateFfmpegFilter(outputTimeline, sourceSize, outputSize)` with the correct outputFps
5. The zoompan filter now produces a video of the correct length

### Part 5: Potential Challenges

1. **Frame alignment**: The raw video has frames at fixed intervals (every 100ms at 10fps). The render plan works in event timestamps. When subsampling gap frames, the nearest-frame selection needs to be correct -- off-by-one errors here will cause visual glitches.

2. **Camera continuity during speed ramps**: When skipping frames in a gap, the camera position should still transition smoothly. Since we generate the full timeline first and then subsample, the camera positions at the kept frames will already be in their settled positions (gaps typically have the camera holding still), so this should work naturally.

3. **Audio**: Not currently in scope (no audio capture), but if audio is added later, speed ramping requires audio pitch correction or muting during fast segments.

4. **zoompan expression length limits**: ffmpeg has practical limits on expression complexity. With 100+ keyframes, the `if(between(...))` chain can get very long. The existing `step` subsampling (every 10th frame, capped at 100 keyframes) handles this, but after resampling, the output timeline might be shorter. Verify the step computation still produces manageable expressions.

5. **Frame count mismatch**: The zoompan filter expects `d=1` (one output frame per input frame). If the resampled timeline has a different number of entries than the raw video has frames after select filtering, there will be a mismatch. The solution is to ensure the zoompan keyframe count matches the select-filtered frame count exactly.

### Key Constraint Differences

1. **10fps vs 60fps**: Our stream delivers ~10fps JPEGs. Cursor gets 60fps from x11grab. This means our interpolation between frames must be much more aggressive to produce smooth output. At 10fps we have 100ms gaps between frames; at 60fps Cursor has 16.6ms.

2. **No cursor in raw capture**: Cursor explicitly uses `-draw_mouse 0` to suppress the native cursor then renders a spring-physics cursor via their Rust renderer. Our WebSocket JPEG stream from agent-browser similarly has no cursor (headless Chromium). So both pipelines need cursor overlay, but we currently only draw cursors in the `CanvasRenderer` which is not part of the ffmpeg post-processing pipeline.

3. **No Rust renderer**: Cursor's `polished-renderer.node` (NAPI) handles compositing at GPU speed. We need to achieve similar effects using ffmpeg filters, Canvas 2D (via `@napi-rs/canvas`), or a WebCodecs-based renderer.

4. **Existing infrastructure is strong**: Our `CameraEngine`, `IntentClassifier`, `ShotPlanner`, spring physics, dead zone, `Compositor`, and `CanvasRenderer` are all production-ready and well-tested (15 test files). The `render-plan.ts` speed ramping is already Cursor-parity quality.

### 2. What Needs to Be Built or Rewritten

**a) CRITICAL: Canvas-based frame renderer to replace zoompan filter**

The current pipeline uses ffmpeg's `zoompan` filter for camera effects. This is fundamentally limited:
- Cannot overlay cursor, click ripples, spotlight glow, device frames, or keystroke overlays
- ffmpeg expressions cap at ~80 keyframes (the codebase already has this limit)
- Cannot do smooth sub-frame interpolation between 10fps input frames
- Cannot render different playback speeds per-segment natively

What Cursor does with their Rust renderer, we need to do with a **Canvas-based frame-by-frame renderer** using the existing `Compositor` + `CanvasRenderer`:

For each output frame:
1. Decode the source JPEG from raw video (or interpolate between frames for smooth 30fps from 10fps input)
2. Compute camera transform from the timeline
3. Compute cursor position from the timeline (spring-interpolated)
4. Call `compositor.computeFrame(camera, cursor)` to get `RenderInstructions`
5. Call `canvasRenderer.renderFrame(sourceImage, instructions, background)` to draw the composited frame
6. Encode the canvas output to the final video

This replaces the `generateFfmpegFilter()` + ffmpeg post-process approach with a much richer pipeline.

**b) Frame interpolation for 10fps to 30fps output**

At 10fps input, adjacent frames are 100ms apart. Simply repeating frames produces jerky output. We need either:
- **Frame blending**: Cross-dissolve between consecutive JPEG frames (easy with Canvas `globalAlpha`)
- **Hold-and-transition**: Hold each frame for most of its duration, quick cross-fade to next (like Cursor's approach with variable speed)
- **Motion interpolation via camera only**: Keep the same source frame but animate the camera pan/zoom smoothly between keyframes (this is what `CameraEngine` already does naturally -- it updates the crop window every frame even when the source image hasn't changed)

The third approach is actually what our system already supports -- the camera moves smoothly at 30fps even when source frames only change at 10fps. The result is smooth zoom/pan over occasionally-changing screenshots, which matches Cursor's behavior closely (their 60fps video has lots of static frames between actions too).

**c) Speed ramping integration into the render pipeline**

`render-plan.ts` is ready algorithmically but disconnected from the actual rendering. The select filter approach (line 8 of session-manager: "disabled until select filter is debugged") needs to be replaced with the canvas-based approach anyway. In the canvas renderer, speed ramping becomes trivial: for each output frame at output timestamp T, map T back to source timestamp via `outputToSourceTime()`, find the corresponding source frame and camera state, render.

**d) Keystroke overlays**

Cursor renders keystroke overlays (showing typed text). Our `AgentEvent` already captures `meta.text` for type events. The `CanvasRenderer` needs a new method to draw keystroke pills/badges. This is a purely additive feature on top of the existing renderer.

**e) Brand tag / intro/outro**

Cursor appends a 2s logo. We can do this via ffmpeg concat (trivial) or render it as part of the canvas pipeline. Low priority.

### 4. Key Architectural Decisions

**Decision 1: Canvas rendering vs. ffmpeg filters**

The zoompan approach is limited to crop+scale. Moving to canvas-based rendering unlocks cursor overlay, click effects, device frames, keystroke overlays, and gradient backgrounds -- all of which the `CanvasRenderer` already implements. The cost is performance: canvas rendering at 1920x1080 @ 30fps is ~57k frames/minute. With `@napi-rs/canvas`, each frame takes ~5-15ms to render, so a 60-second video takes ~90 seconds to render. This is acceptable for offline post-processing.

**Decision 2: Frame decode approach**

Rather than decoding the entire raw.mp4 into memory, use ffmpeg as a frame server: pipe frames out on demand. For speed-ramped sections, we can skip decoding frames we do not need (gap segments at 8x only need every 8th frame). This keeps memory bounded.

**Decision 3: Output encoding**

Pipe canvas output (as raw RGBA or MJPEG) into ffmpeg for H.264 encoding. This gives us hardware acceleration where available and proper codec handling. The pipeline becomes:

```
raw.mp4 --[ffmpeg decode]--> JPEG buffers --[Canvas render]--> RGBA pixels --[ffmpeg encode]--> final.mp4
```

**Decision 4: 10fps handling**

Do NOT attempt optical flow or AI frame interpolation. Instead, rely on the camera engine's smooth animation: even when the source frame is static (same screenshot), the camera pan/zoom changes every frame, creating smooth motion. When the source does change (new screenshot), the transition happens naturally. This matches how screen recordings actually work -- most of the visual interest comes from cursor movement and zoom changes, not from frame-level content changes.
