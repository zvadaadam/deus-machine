**Philosophy:** Connection state is infrastructure — it should be felt before it is read. The interface communicates through behavior change, not decoration.

## The System

A single component, `ConnectionBanner`, owns all four states. It lives at the top of the main content area (not the sidebar), spanning the full width. It uses vertical space as its primary signal: no space when healthy, a thin 32px bar for transient issues, a 48px bar for persistent problems, and the full viewport for terminal states. The sidebar remains untouched in all cases — it is navigation, not status.

A secondary indicator — a 6px circle in the sidebar footer next to the connection label — provides ambient awareness but never carries the primary communication burden.

## State 1: Transient Disconnect (auto-resolves in <30s)

**Trigger:** WebSocket closes or misses 2+ heartbeats. Delay rendering by 2 seconds — if reconnection succeeds within that window, the user sees nothing.

**Visual treatment:**

A 32px banner slides down from the top of the content area. Background: `oklch(0.25 0.02 70)` — a barely warm neutral, darker than the app background but not alarming. No border. No icon.

Single line of text, `font-size: 13px`, `oklch(0.65 0.01 70)` (muted foreground): **"Reconnecting…"**

A subtle progress animation: a 2px-tall bar at the bottom edge of the banner, `oklch(0.55 0.08 70)` (desaturated amber), translating left-to-right in a 2.5s loop using `translateX` with `ease-in-out-sine` (`cubic-bezier(0.37, 0, 0.63, 1)`).

## State Machine & Timing

The `ConnectionBanner` component is driven by a state machine with explicit timing thresholds:

```
CONNECTED ──[ws close/timeout]──→ GRACE_PERIOD (2s, render nothing)
GRACE_PERIOD ──[reconnected]──→ CONNECTED
GRACE_PERIOD ──[2s elapsed]──→ RECONNECTING (show thin banner)
RECONNECTING ──[reconnected]──→ CONNECTED (fade out banner)
RECONNECTING ──[30s elapsed]──→ DISCONNECTED (escalate banner)
DISCONNECTED ──[reconnected]──→ CONNECTED (fade out banner, restore opacity)
DISCONNECTED ──[retry click]──→ RECONNECTING (attempt + de-escalate banner)

NEVER_CONNECTED (first load fail) → SERVER_OFF (full screen)
SERVER_OFF ──[ws connects]──→ CONNECTED (cross-fade to app)

CONNECTED ──[401/unpaired]──→ UNPAIRED (full screen)
UNPAIRED ──[pairing success]──→ CONNECTED (cross-fade to app)
```

---

## Three Key Decisions

**1. No red anywhere in the transient or extended states.**

Red implies user error or danger. A network blip is neither. Amber and warm neutrals communicate "working on it" without triggering alarm. Red appears only in the sidebar dot during extended outage, and even there it is `oklch(0.65 0.18 25)` — a diagnostic tone, not an emergency. This keeps the emotional register proportional to the actual severity. Users managing multiple AI agents cannot afford to be startled by infrastructure noise.

**2. Content dims but never hides during disconnection.**

The user may have sent an agent to refactor a module and is reviewing the streamed output when the connection drops. Hiding or blurring that content would destroy their context. Dimming to `opacity: 0.6` with disabled interaction clearly communicates "this is stale, do not trust it" without removing the information. The moment the connection returns, full interactivity restores instantly. This respects the user's attention, which is the most expensive resource in the system.

**3. Full-viewport takeover only for terminal states (server off, unpaired).**

States 3 and 4 share one trait: there is nothing meaningful the user can do in the app. No sidebar, no workspace list, no chat — all of it depends on the backend. Showing an empty shell with a banner would be worse than showing nothing, because it implies the app is partially functional when it is not. A centered, minimal screen with a single clear action (wait, or pair) eliminates ambiguity. It also avoids the uncanny valley of a populated-but-broken UI.

---

## One Acknowledged Tradeoff

**The 2-second grace period means users with genuinely flaky connections will experience a delayed notification.** If a disconnect lasts 2.5 seconds, the user sees the banner for only 0.5 seconds before it disappears — which may feel like a flash or glitch. The alternative (showing the banner immediately) would create constant visual noise for users on unstable networks, which is worse. The 2-second threshold was chosen because WebSocket reconnection in a local-network desktop relay typically succeeds within 1-1.5 seconds. For users with genuinely poor connections, the banner will persist past the grace period and the escalation to State 2 will provide sustained, clear communication. The tradeoff favors quiet confidence over anxious transparency.</result>

## Three Key Decisions

## Three Key Decisions

**1. No toasts for connection state — ever.**

Toasts are for actions the user initiated. Connection drops are infrastructure events. A toast for "Disconnected" forces the user to acknowledge something they cannot act on. The amber pulse in State 1 and the content bar in State 2 are **ambient** — they exist in peripheral vision, not in the notification stack. This keeps the toast system trustworthy: when a toast appears, it means something the user did had a result.

**2. The 30-second threshold before any visible interruption to the content area.**

Most WebSocket reconnects resolve in under 10 seconds. Showing a banner at 2 seconds trains users to see and then ignore warnings — "cry wolf" conditioning. At 30 seconds, the disruption is real: the user may have tried to send a message to an agent, or is watching an agent work and seeing no updates. The bar appears precisely when the user would start to wonder "is something wrong?" — not before. This restraint is what makes the signal trustworthy when it finally appears.

**3. States 3 and 4 use the same full-page layout shell with no sidebar.**

When the app cannot show workspace data (server off, user unpaired), rendering an empty sidebar frame with a loading skeleton would be dishonest — it implies content is coming momentarily when it may not be. A clean full-page state with the warm illustration communicates completeness: "This is the right place. Here is what needs to happen." It also means the transition into the full workspace (sidebar sliding in, content populating) feels like an arrival — a moment of satisfaction rather than incremental loading.

## 3 Key Decisions

**1. Temperature as the primary communication channel, not iconography or severity colors.**

Traditional UX uses green/yellow/red traffic-light coding. This creates a jarring, clinical feel that clashes with the product's warm illustration aesthetic. By mapping connection health to a warm-to-cool temperature gradient, we extend the existing brand language (the orange tangled cord) into a coherent chromatic system. A user who sees the app cooling down understands intuitively that warmth equals life — without needing to parse a status badge. This also avoids the accessibility pitfall of relying solely on red/green distinction, since the shift is along the warm/cool axis (hue 50-70 vs 250-280) which is perceptible even to most color-vision-deficient users.

**2. Progressive escalation with timed phase transitions rather than instant state changes.**

State 1 waits 2 seconds before showing anything. State 2 introduces the ambient stripe at 30 seconds and the notification bar at 45 seconds. State 3 is immediate (it is the initial load). This staggered approach prevents the "boy who cried wolf" problem — if every brief hiccup showed a banner, users would learn to ignore connection warnings entirely. By the time we show a notification bar, the user can trust that it is meaningful. The timing values (2s, 30s, 45s) are calibrated to WebSocket reconnection patterns: most auto-reconnects resolve within 5-15 seconds.

**3. The app shell remains visible in states 2 and 4 but is desaturated or overlaid, not replaced.**

Replacing the entire UI with an error page (as state 3 does) is appropriate only when there is genuinely nothing to show. For states 2 and 4, the user's workspace, their agent sessions, and their recent work should remain visible — even if non-interactive. This preserves spatial context ("I can see my sessions are still there") and reduces anxiety. The desaturation overlay in state 2 and the frosted glass in state 4 communicate "paused" rather than "gone," which is the accurate mental model.

---

## 1 Acknowledged Tradeoff

**The temperature metaphor requires onboarding through experience, not explanation.** Unlike a red badge labeled "Disconnected," the chromatic temperature shift is ambient and non-verbal. A first-time user will not immediately understand that "the sidebar dot turned blue-gray" means "server unreachable." We accept this tradeoff because: (a) the notification bar in state 2 and the full-page in state 3 provide explicit textual fallbacks for serious outages, so no critical information is lost; (b) over repeated use, the temperature association becomes internalized faster than reading labels — it becomes a peripheral sense rather than a conscious check; and (c) the product's audience (semi-technical users managing AI agents) is sophisticated enough to build this association within a few sessions. The warmth language is learned through the body, not the manual. That is the point.</result>

# Ambient Presence

**Philosophy:** Connection state is not an alert — it is a quality of the environment. The way a room feels dim when a lamp goes off, then brightens when it returns. The user should never "read" the connection state; they should *feel* it through the interface's warmth, responsiveness, and visual density.

## State 1: Brief Flaky Disconnect (auto-resolves within ~5 seconds)

**The user should not notice this at all.** This is a micro-disruption the software absorbs.

### Visual treatment

**No new UI element appears.** The existing interface continues to render with its cached data (React Query cache remains intact). The only change is an ambient, nearly subliminal desaturation of interactive elements.

**Concrete implementation:**

A CSS custom property `--connection-vitality` transitions from `1` to `0.85` when the WebSocket disconnects. This property modulates:

- `--primary` saturation: the cool-rose interactive accent (`oklch(0.78 0.09 345)`) shifts to `oklch(0.78 0.04 345)` — the chroma drops by half, making buttons and links feel slightly muted without appearing broken
- The sidebar's `CircularPixelGrid` animation speed slows by 30% (multiply `requestAnimationFrame` delta by `0.7`), creating a subtle "holding breath" effect
- Message input placeholder text shifts from `--text-muted` to `--text-disabled`

**Timing:**

- Desaturation begins after a **1.5-second grace period** (covers most Wi-Fi blips). Uses `transition: 800ms cubic-bezier(0.165, 0.84, 0.44, 1)` on the custom property propagation via JS.
- On reconnect, saturation returns over **400ms** — faster recovery than onset, because "restored" should feel crisp.

**Message input behavior:**

- The input remains fully interactive. Messages typed during the blip are queued and sent when the WebSocket reconnects (the existing `sendCommand` already rejects with "WebSocket not connected" — instead, queue in a local buffer and flush on `onConnectionChange(true)`).
- No toast. No banner. No dot. Nothing.

**Component structure:**

No new components. The `BackendStatusIndicator` gains a `graceMs` prop (default `1500`) and suppresses rendering during the grace window. The CSS variable modulation lives in a `useConnectionVitality()` hook that sets `document.documentElement.style.setProperty('--connection-vitality', ...)`.

```tsx
// useConnectionVitality.ts — sets CSS variable on <html>
function useConnectionVitality() {
  useEffect(() => {
    let graceTimer: ReturnType<typeof setTimeout> | null = null;
    
    return onConnectionChange((connected) => {
      if (!connected) {
        graceTimer = setTimeout(() => {
          document.documentElement.style.setProperty('--connection-vitality', '0.85');
        }, 1500);
      } else {
        if (graceTimer) clearTimeout(graceTimer);
        document.documentElement.style.setProperty('--connection-vitality', '1');
      }
    });
  }, []);
}
```

**Copy:** None. Silence is the design.

## 3 Key Decisions

**1. Grace period before any visual feedback (1.5 seconds).**

Most WebSocket disconnects in local development and even relay mode are transient — a Wi-Fi re-association, a macOS sleep/wake cycle, a backend hot-reload. Showing any indicator for these sub-second events would train users to associate the IDE with fragility. The 1.5-second grace period absorbs the majority of flaky disconnects silently. The user's experience is: "It just works." The 1.5s threshold was chosen because the existing backoff starts at 500ms and doubles — a disconnect that auto-resolves within 1.5s will have completed at most one reconnect attempt.

**2. Ambient environmental response instead of discrete alerts.**

Rather than adding toasts, banners, or modal overlays for connection issues, the design modulates *existing* visual properties — color saturation, animation speed, surface luminance. This follows Fukasawa's principle: the interface's "mood" shifts, not its structure. A toast demands attention and creates cognitive overhead ("Should I dismiss this? Is my work lost?"). An environment that feels slightly dimmer creates awareness without anxiety. The user's autonomic response is correct — "something feels different" — but they don't need to act on it.

**3. Separate the relay "server disconnected" state from local disconnection.**

In relay mode, the WebSocket between browser and relay stays open even when the desktop app goes down. The user's mental model is different: they're connected to a service, but their computer went away. The copy and behavior must reflect this distinction. Showing reconnection attempts (which the relay manages transparently) or a "Retry" button (which would do nothing useful) would violate the user's expectations. Instead, the interface says "waiting for your computer" — accurate, calm, and action-clear ("go open the desktop app").

## 1 Acknowledged Tradeoff

**The ambient desaturation approach (CSS variable modulation) is invisible to users who are not visually attentive.** Someone focused on reading a chat transcript during a 5-second disconnect will not notice the chroma shift from `0.09` to `0.04` on the primary accent. This is intentional — for brief outages, we *want* invisibility — but it means that if a user sends a message during the 1.5-8 second window (after grace period, before status bar), the message will silently queue and the user has no feedback that it didn't send immediately. The mitigation is that messages appear instantly in the optimistic React Query cache, and the actual send latency (when the WS reconnects seconds later) is indistinguishable from normal agent processing time. The risk is low, but not zero — a user sending a time-sensitive "stop" command during a 6-second outage would experience a ~6-second delay with no indication it hasn't been delivered yet. For a v2 iteration, a per-message delivery indicator (subtle checkmark on sent messages) could close this gap, but adding that now would violate the "design dissolving into behavior" principle by introducing an element that is 99.9% redundant in normal operation.</result>

### #1 — `useConnectionState` facade hook defeats Zustand selector discipline

**Severity:** CLAUDE.md violation (low practical impact)

**File:** `apps/web/src/features/connection/hooks/useConnectionState.ts`

The facade hook creates 5 subscriptions internally:

```ts
const state = useConnectionStore((s) => s.state);
const disconnectedAt = useConnectionStore((s) => s.disconnectedAt);
const sendAttemptFailed = useConnectionStore((s) => s.sendAttemptFailed);
const retry = useConnectionStore((s) => s.retry);
const markSendAttemptFailed = useConnectionStore((s) => s.markSendAttemptFailed);
```

Every consumer of `useConnectionState()` subscribes to **all 5 fields**, regardless of what they actually use. `ConnectionOrb` and `MainContent` only need `state` but will re-render when `disconnectedAt` or `sendAttemptFailed` changes.

CLAUDE.md rule (verbatim):
> **Zustand selector discipline** — never subscribe to an entire store. Always use individual selectors:
> ```tsx
> // ❌ BAD — re-renders on ANY store change
> const { collapsedRepos, toggleRepoCollapse } = useSidebarStore();
> ```

**Fix:** Have consumers use `useConnectionStore((s) => s.state)` directly when they only need one field. The facade hook is fine for `ConnectionBanner` which needs 3 fields.

---

**Practical impact is low** — the store has only 3 mutable fields, `retry`/`markSendAttemptFailed` are stable function refs, and state changes happen rarely (only on disconnect/reconnect). But the pattern sets a precedent that could be more costly in larger stores.
