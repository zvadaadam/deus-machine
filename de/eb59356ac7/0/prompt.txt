## CLI Installation & First-Run UX Patterns Research

### 1. Claude Code

**Installation:**
```bash
npm install -g @anthropic-ai/claude-code
claude
```

**First-run experience:**
- On first launch, Claude Code detects no authentication and initiates an **OAuth browser flow**. It opens your default browser to `console.anthropic.com` where you log in with your Anthropic account.
- After authenticating in the browser, a callback redirects back to the CLI, which receives and stores an OAuth token locally (in `~/.claude/` or the platform equivalent).
- There is no traditional API key prompt by default -- it uses OAuth so the user never copies/pastes a key. However, it also supports `ANTHROPIC_API_KEY` environment variable as an alternative for CI/headless environments.
- After auth, it drops you directly into the interactive REPL -- no setup wizard, no project config step. It auto-detects the current directory as context.
- The CLI displays a welcome message with the model being used and available slash commands.
- **Permissions model:** On first use of tools (file edits, shell commands), it asks for explicit permission with an accept/reject prompt. It remembers allowed permissions per project in `.claude/settings.json`.

**Key pattern:** OAuth-first with env-var fallback. Zero-config start -- just authenticate and go. Progressive permission prompts (not upfront).

### 2. Cursor

**Installation:**
- Cursor is primarily a **desktop app** download (`.dmg` / `.exe` / `.AppImage`) from `cursor.com/downloads`.
- The CLI (`cursor`) is installed **from within the desktop app**: Command Palette > "Install 'cursor' command". This is the exact same pattern as VS Code's `code` command.
- No `npm install` path exists. The CLI is a thin launcher that opens files/folders in the desktop app.

**First-run experience (desktop app):**
- Download and open the app. First launch presents a **setup wizard**:
  1. **Import settings** -- offers to import from VS Code (extensions, keybindings, settings). This is a key onboarding step since Cursor is a VS Code fork.
  2. **Sign in / Sign up** -- creates a Cursor account (email + password, or GitHub/Google OAuth). This gates access to the AI features.
  3. **Choose plan** -- Free tier vs Pro ($20/mo). Free gives limited "fast" requests.
  4. **Privacy preferences** -- toggle for whether your code is used for training (privacy mode).
- After setup, it opens directly as an editor with the AI chat panel ready.

**CLI behavior:**
- `cursor .` opens the current directory in the desktop app.
- `cursor --install-extension` installs extensions.
- The CLI has no standalone mode -- it always launches the GUI app.

**Key pattern:** Desktop-first with CLI as a secondary launcher. Setup wizard for import/migration. Account creation gates AI features.

### 3. Supabase CLI

**Installation:**
```bash
# Via npm (most common)
npx supabase init        # Initialize a project
npx supabase login       # Authenticate

# Via brew
brew install supabase/tap/supabase

# Via scoop (Windows)
scoop bucket add supabase https://github.com/supabase/scoop-bucket.git
scoop install supabase
```

**First-run experience:**
- `supabase login` opens a browser to `app.supabase.com` and generates a **personal access token**. The browser shows the token, and the CLI prompts you to paste it back into the terminal. The token is stored in `~/.supabase/access-token`.
- Alternative: `supabase login --token <token>` for non-interactive/CI use.
- `supabase init` creates a `supabase/` directory with `config.toml` -- this is the project configuration step. It asks no interactive questions; it generates defaults.
- `supabase link --project-ref <ref>` connects a local project to a remote Supabase project. The `project-ref` is found in the Supabase dashboard URL.
- `supabase start` spins up a full local Supabase stack using Docker (Postgres, Auth, Storage, etc.). This is the "wow" moment -- a complete backend running locally.
- No setup wizard. Each command is independent and self-documenting.

**Key pattern:** Token-based auth with browser generation + terminal paste. Explicit project linking step. Local-first development with Docker. Commands are modular, not a linear wizard.

### 4. Vercel CLI

**Installation (confirmed from docs):**
```bash
npm i -g vercel      # or pnpm/yarn/bun
npx vercel           # one-shot without install
```

**First-run experience:**
- Running `vercel` (the deploy command) or any command without auth triggers `vercel login` automatically.
- `vercel login` presents an **interactive prompt** with multiple auth methods:
  1. **Continue with GitHub** -- opens browser for GitHub OAuth
  2. **Continue with GitLab** -- opens browser for GitLab OAuth
  3. **Continue with Bitbucket** -- opens browser for Bitbucket OAuth
  4. **Continue with Email** -- sends a verification email with a confirmation code
  5. **Continue with SAML SSO** -- for enterprise
- After browser-based auth completes, the CLI receives and stores a token in `~/.vercel/auth.json`.
- For CI, you use `--token` flag with a token created at `vercel.com/account/tokens`.
- After login, running `vercel` in a project directory triggers a **project link flow**:
  1. "Set up and deploy?" [Y/n]
  2. "Which scope?" (if multiple teams)
  3. "Link to existing project?" [Y/n]
  4. If new: "What's your project's name?" with auto-detected default
  5. "In which directory is your code located?" with auto-detected default
  6. Auto-detects framework (Next.js, Vite, etc.) and shows build settings
  7. Deploys immediately
- This interactive flow is skipped if a `.vercel/project.json` already exists (project is linked).

**Key pattern:** Multi-provider OAuth with interactive selection. Auto-detection of framework/settings. Project linking creates a local `.vercel/` directory. First deploy is part of first-run. `--token` flag for CI.

### 5. Railway CLI

**Installation:**
```bash
# Via brew
brew install railway

# Via npm
npm install -g @railway/cli

# Via shell script
curl -fsSL https://railway.app/install.sh | sh

# Via scoop (Windows)
scoop install railway
```

**First-run experience:**
- `railway login` presents two options:
  1. **Browser login** (default) -- opens browser to `railway.app` for OAuth (GitHub or email). After auth, a callback sends the token to the CLI automatically via a localhost callback server. No manual paste needed.
  2. **Token login** (`railway login --browserless`) -- for headless/SSH environments. Generates a pairing URL, you visit it on any device, authenticate, and the CLI detects the auth automatically via polling.
- Token is stored in `~/.railway/config.json`.
- After login, `railway link` connects to a project interactively:
  1. Shows a list of your Railway projects (interactive fuzzy-searchable list)
  2. Select project
  3. Select environment (production, staging, etc.)
  4. Creates `.railway/` local config
- `railway run <command>` injects all environment variables from your Railway project into the command's environment.
- `railway up` deploys the current directory.

**Key pattern:** Browser-first OAuth with automatic token callback (no paste). Browserless fallback with URL + polling for SSH/headless. Interactive fuzzy-searchable project/environment selection. Environment variable injection is the killer feature.

### 6. ngrok

**Installation:**
```bash
# Via brew
brew install ngrok

# Via npm
npm install -g ngrok     # unofficial wrapper

# Via direct download
# Download binary from ngrok.com/download, unzip, add to PATH

# Via snap (Linux)
snap install ngrok

# Via chocolatey (Windows)
choco install ngrok
```

**First-run experience:**
- ngrok uses a **two-step pairing model**:
  1. **Sign up** at `dashboard.ngrok.com` -- creates an account (GitHub OAuth, Google OAuth, or email)
  2. **Copy authtoken** from the dashboard (`dashboard.ngrok.com/get-started/your-authtoken`)
  3. **Paste into CLI**: `ngrok config add-authtoken <YOUR_TOKEN>`
- This stores the token in `~/.config/ngrok/ngrok.yml` (or `~/Library/Application Support/ngrok/ngrok.yml` on macOS).
- After that, `ngrok http 3000` immediately creates a tunnel. No project setup, no linking.
- The authtoken is a long-lived credential that ties the CLI to your dashboard account.
- Without an authtoken, ngrok still works but with severe limitations (sessions expire after 2 hours, no custom domains, rate limited). This is a deliberate freemium gate.
- The ngrok dashboard shows all active tunnels, request inspection, traffic replay -- it's a **live companion** to the CLI, not a one-time setup destination.
- `ngrok http 3000` prints a status panel in the terminal showing:
  ```
  Session Status    online
  Account           user@email.com
  Forwarding        https://abc123.ngrok-free.app -> http://localhost:3000
  Web Interface     http://127.0.0.1:4040
  ```
- The local web inspector at `http://127.0.0.1:4040` is started automatically, letting you inspect all tunneled HTTP traffic.

**Key pattern:** Dashboard-first account creation, then copy-paste token to CLI. Works without auth (degraded). Authtoken is a persistent pairing mechanism. Dashboard is a live companion, not just onboarding. Local web inspector for debugging.

## Summary: Pattern Taxonomy

| Pattern | Used By | How It Works |
|---|---|---|
| **OAuth browser flow (automatic callback)** | Claude Code, Railway, Vercel | CLI opens browser, auth completes, token returned automatically via localhost callback |
| **OAuth browser flow (manual paste)** | Supabase | Browser generates token, user pastes it into terminal |
| **Copy authtoken from dashboard** | ngrok | Sign up on web, copy token, paste into CLI config command |
| **Desktop app installs CLI** | Cursor | CLI is a launcher for the desktop app, installed from within it |
| **Env var fallback** | Claude Code, Vercel | `ANTHROPIC_API_KEY` / `--token` flag for CI/headless |
| **Browserless fallback** | Railway | Generates URL for auth on another device, polls for completion |
| **Auto-detection** | Vercel, your Deus CLI | Detect framework, platform, display availability |
| **Progressive permission** | Claude Code | Don't ask everything upfront; prompt when features are first used |
| **Freemium gate** | ngrok, Cursor | Works without auth but with limitations |
| **Setup wizard** | Cursor | Multi-step onboarding (import, auth, plan, privacy) |
| **Project linking** | Vercel, Railway, Supabase | Separate `link` command that creates local config file |
