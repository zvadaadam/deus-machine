## Review: `packages/screen-studio/`
This is a new standalone package implementing a "Screen Studio-style" camera engine for agent screen recordings. It takes agent browser actions (click, type, scroll, navigate) and produces smooth auto-zoom/pan camera movements via spring physics, then outputs either canvas render instructions or ffmpeg filter strings for polished MP4s.
**Architecture:** Camera engine → Compositor → Canvas renderer, with MCP server integration, CLI, and ffmpeg post-processing.

### Issues Found
### **#1 DOM injection via unsanitized `ref` in `RESOLVE_ELEMENT_JS`**
`src/adapter/mcp-adapter.ts:204` — The `ref` parameter is concatenated directly into a CSS selector string: `'[data-deus-ref="' + ref + '"]'`. The function is invoked via string concatenation (`RESOLVE_ELEMENT_JS + '("@e5")'`), meaning a crafted ref value could break out of both the selector and the JS expression. While refs currently come from the agent's own snapshot system (`@eN` format), this is an inherently unsafe pattern.
**Fix:** Pass `ref` as a parameter to `page.evaluate((ref) => {...}, ref)` instead of string concatenation, or sanitize `"`, `\`, and `]` characters before interpolating.
File: `packages/screen-studio/src/adapter/mcp-adapter.ts`
### **#2 Division by zero in `smoothstep`/`smootherstep` when `edge0 === edge1`**
`src/interpolation/smoothstep.ts:7,17` — Both functions compute `(x - edge0) / (edge1 - edge0)`. When `edge0 === edge1`, this produces `NaN` (via `clamp(NaN, 0, 1)` → `NaN`), which silently corrupts camera positions. The sibling function `inverseLerp` (line 33) already guards against this with `if (Math.abs(b - a) < 1e-10) return 0`, but `smoothstep`/`smootherstep` don't. In `dead-zone.ts:112`, `smoothstep(gradientStart, gradientEnd, absDelta)` is called where `gradientStart` and `gradientEnd` can converge when the dead zone `fraction` approaches 1.0.
**Fix:** Add `if (Math.abs(edge1 - edge0) < 1e-10) return 0;` guard to both functions.
File: `packages/screen-studio/src/interpolation/smoothstep.ts`
### **#3 Missing `drag` event type in MCP server and CLI validators**
`src/mcp/server.ts:120` and `src/cli/index.ts` — The Zod enum for event types lists `["click", "type", "scroll", "navigate", "screenshot", "idle"]` but `AgentEventType` in `types.ts:71` also includes `"drag"`. If an agent sends a drag event through the MCP tool or CLI, it will be rejected by validation despite the camera engine fully supporting it.
**Fix:** Add `"drag"` to both the Zod enum in `server.ts` and the `validTypes` array in `cli/index.ts`.
File: `packages/screen-studio/src/mcp/server.ts`, `packages/screen-studio/src/cli/index.ts`
### **#4 Incomplete ffmpeg drawtext escaping**
`src/mcp/ffmpeg-recorder.ts:162` — Watermark text is escaped for single quotes and colons, but ffmpeg's drawtext filter also interprets `%`, `;`, `\`, `[`, `]`, `=`, and newlines. A watermark containing `%{pts}` would be interpreted as an ffmpeg expression, and `;` could terminate the filter.
**Fix:** Escape the full set of ffmpeg drawtext special characters, or use the `textfile=` option instead of inline `text=`.
File: `packages/screen-studio/src/mcp/ffmpeg-recorder.ts`

### **#1 — Spring velocity formula has a math error**
The underdamped spring velocity computation in `stepUnderdamped` uses `displacement * dampedOmega` (i.e. `displacement * ωd`) where the correct analytical derivative requires `displacement * omega * omega / dampedOmega` (i.e. `displacement * ω²/ωd`).
These are equal only when `ζ = 0` (undamped). With the default camera preset (`ζ = 0.7`), `ωd ≈ 5.71` but `ω²/ωd ≈ 11.20` — the code is off by nearly 2x. The incorrect velocity feeds back into subsequent frames, producing a different motion profile than intended. The spring still converges (position formula is correct), but camera animations will feel noticeably different from the specified spring parameters.
**Fix:** Line 82: change `displacement * dampedOmega` to `displacement * omega * omega / dampedOmega`.
File: `packages/screen-studio/src/camera/spring.ts:82`

### **#2 — `escapeDrawtext` uses shell-level escaping but ffmpeg is invoked via `spawn` (no shell)**
The entire `escapeDrawtext` function assumes a shell interpretation layer (the comment on line 151 says "needs quadruple escaping for shell + ffmpeg"), but all `spawn("ffmpeg", args)` calls use array-based arguments without `shell: true`. There is no shell layer — args go directly to ffmpeg's filter parser.
The single-quote replacement `.replace(/'/g, "'\\\\''")` produces `'\\''` (two backslashes) when the correct spawn-context escaping is `'\''` (one backslash). Similarly, colons get `\\:` (double backslash) when they should get `\:` (single). Any watermark text with apostrophes, colons, semicolons, brackets, or equals signs will cause ffmpeg filter parse errors.
The tests encode the same incorrect expectations (e.g., line 240 of the test file asserts the wrong output).
**Fix:** Either halve all backslash escaping levels throughout the function (correct for `spawn`), or add `shell: true` to the `spawn` calls (not recommended).

### **#4 — CLI `--capture` mode is architecturally broken (hangs indefinitely)**
When `cmdStart` is called with `--capture x11grab` or `--capture avfoundation`, the `SessionManager` spawns an ffmpeg capture child process with piped stdio. The active stdio pipe listeners keep the Node.js event loop alive, so the CLI **hangs indefinitely** after printing the session ID instead of returning to the shell.

A new `@deus/screen-studio` package is introduced, implementing an end-to-end screen recording system that converts agent browser interactions into composited video frames using spring-based camera physics, intent classification, and ffmpeg processing. Includes camera engine, intent classifier, compositor, MCP server, CLI, recording session management, and comprehensive tests.
