## How Recording Works
The recording system has three layers: **rrweb capture in the browser**, **event collection in Node.js**, and **replay/viewing**.
### 1. rrweb Injection
A runtime script (`packages/browser/src/runtime/index.ts`) is compiled into an IIFE at build time and injected into every page via Playwright's `context.addInitScript()`. This exposes `globalThis.__EXPECT_RUNTIME__` with functions like `startRecording()` and `getEvents()`.
When recording starts, rrweb captures DOM mutations, mouse movements, and input changes. Events accumulate in an in-browser buffer with checkpoints every 10 seconds.
### 2. Event Collection (Node.js side)
The MCP session (`packages/browser/src/mcp/mcp-session.ts`) polls the browser every **250ms**, calling `evaluateRuntime(page, "getEvents")` which destructively drains the buffer. Events accumulate on the Node.js side.
Two modes:
- **Live view** — Events are broadcast via an SSE server (`/packages/browser/src/mcp/live-view-server.ts`) so a web viewer can show the test in real-time
- **Offline** — Events are just accumulated in memory
### 3. Saving on Session Close
When `close()` is called on the MCP session, it:
1. Does a final `collectAllEvents()` drain
2. Writes **4 files** to `.expect/replays/{planId}.*`:
   - `.ndjson` — one rrweb event per line (newline-delimited JSON)
   - `.html` — self-contained viewer with embedded rrweb-player
   - `.ndjson.js` — JS wrapper for single-file deployments
   - Playwright's native `.webm` video
3. Copies everything to `/tmp/expect-replays/` as well
### 4. Replay Viewer
The generated HTML (`packages/browser/src/replay-viewer.ts`) supports two modes:
- **SSE/live mode** — Fetches `/latest.json` for initial events, then subscribes to `/events` SSE stream. Uses `rrweb-player` with `liveMode: true` and dynamically adds events
- **NDJSON mode** — Loads the `.ndjson` file, parses line-by-line, and renders with standard playback controls
### 5. The `--replay-host` Proxy
The CLI spins up a local Hono HTTP server (`apps/cli/src/utils/replay-proxy-server.ts`) that reverse-proxies to the remote host (default: `expect.dev`). It caches events/steps locally for offline fallback and intercepts `/latest.json` and `/steps` endpoints.
### 6. Bonus: Performance Metrics
The runtime also sets up `PerformanceObserver`s to capture FCP, LCP, CLS, and INP — so the test report can include web vitals alongside functional results.
### Data Flow
```
Browser (rrweb)
  → globalThis.__EXPECT_RUNTIME__.getEvents()  [every 250ms]
  → Node.js MCP session accumulates events
  → Live: SSE broadcast to viewer
  → Close: write .ndjson + .html + .ndjson.js + .webm
  → CLI shows replay URL or file paths in results
```
