## Approach: **Split-Plane SPA**

### Philosophy

The frontend is one SPA. It talks to two planes: a **cloud plane** (auth, billing, server directory) over HTTPS, and a **data plane** (workspaces, sessions, messages) over WebSocket to the user's desktop backend -- either direct localhost or tunneled through the existing relay. No server-side rendering. No framework. Static files on a CDN. The cloud plane is a thin Hono API on Cloudflare Workers. Zero changes to the existing query protocol.

---

### Implementation Sketch

Here is the critical-path code for each layer.

**1. The web app is a static SPA on Cloudflare Pages (or any CDN).**

The existing Vite build outputs `index.html` + JS/CSS chunks. Deploy those. React Router handles URL routing. No SSR. No Workers for rendering. The CDN serves the same `index.html` for all paths (SPA fallback). This is the simplest possible deployment and gives you global edge caching for free.

**2. The frontend gains a `ConnectionMode` discriminated union that determines where data comes from.**

This is the central abstraction. In Electron mode, the WS connects to `localhost:{port}`. In web mode, the WS connects through the relay to a user's desktop. Eventually, in cloud-hosted mode, the WS connects to a cloud backend. The query protocol client is unchanged -- only the WebSocket URL changes.

**3. Cloud services (auth, billing, server directory) live in a separate Hono API on Cloudflare Workers.**

This is NOT the same as the desktop backend. It handles: OAuth login, JWT tokens, Stripe billing, team management, and the server directory (which desktop server to connect to). It stores API keys and secrets. The frontend calls this over HTTPS. It never touches the query protocol.

**4. React Router v6 with proper URL routing for the web version.**

The desktop stays on the single-route layout (BrowserRouter with `/`). The web version uses the same components but with URL-driven workspace selection.

```typescript
/**
 * Connection Mode
 *
 * Determines how the frontend connects to backend data.
 * Three modes, resolved once at startup:
 *
 * - "local": Electron desktop. WS to localhost:{port}.
 * - "relay": Web browser connected to a desktop server via cloud relay.
 * - "cloud": Web browser connected to cloud-hosted agents (future).
 *
 * The query protocol is identical in all modes. Only the WebSocket
 * URL and auth mechanism change.
 */

import { capabilities } from "@/platform/capabilities";

export type ConnectionMode =
  | { type: "local"; getWsUrl: () => Promise<string> }
  | { type: "relay"; relayUrl: string; serverId: string; deviceToken: string }
  | { type: "cloud"; apiBaseUrl: string; accessToken: string };

let cachedMode: ConnectionMode | null = null;

/**
 * Resolve connection mode. Called once on app init.
 *
 * Priority:
 * 1. Electron → always local
 * 2. URL has ?server=xxx → relay mode (deep link to a desktop server)
 * 3. Authenticated web user with cloud subscription → cloud mode (future)
 * 4. Fallback → relay mode (show server picker)
 */
export function resolveConnectionMode(): ConnectionMode {
  if (cachedMode) return cachedMode;

  // Electron: always local
  if (capabilities.ipcInvoke) {
    const { getBackendPort } = require("@/shared/config/api.config");
    cachedMode = {
      type: "local",
      getWsUrl: async () => {
        const port = await getBackendPort();
        return `ws://localhost:${port}/ws`;
      },
    };
    return cachedMode;
  }

  // Web: check for stored relay credentials
  const serverId = localStorage.getItem("opendevs_server_id");
  const deviceToken = localStorage.getItem("opendevs_device_token");
  const relayUrl = localStorage.getItem("opendevs_relay_url") || "wss://relay.rundeus.com";

  if (serverId && deviceToken) {
    cachedMode = { type: "relay", relayUrl, serverId, deviceToken };
    return cachedMode;
  }

  // No credentials yet -- will be set after server selection/pairing
  // Return relay mode with empty credentials (UI will gate on this)
  cachedMode = { type: "relay", relayUrl, serverId: "", deviceToken: "" };
  return cachedMode;
}

export function setConnectionMode(mode: ConnectionMode): void {
  cachedMode = mode;
}

export function getConnectionMode(): ConnectionMode {
  return cachedMode ?? resolveConnectionMode();
}
```

```typescript
/** Type-safe cloud API fetch. Throws on non-2xx. */
export async function cloudFetch<T>(
  path: string,
  options: RequestInit = {}
): Promise<T> {
  const token = getAccessToken();
  const headers = new Headers(options.headers);
  if (token) {
    headers.set("Authorization", `Bearer ${token}`);
  }
  headers.set("Content-Type", "application/json");

  const res = await fetch(`${CLOUD_API_URL}${path}`, {
    ...options,
    headers,
  });

  if (!res.ok) {
    const body = await res.json().catch(() => ({}));
    throw new CloudApiError(res.status, body.message ?? res.statusText);
  }

  return res.json();
}

export class CloudApiError extends Error {
  constructor(public status: number, message: string) {
    super(message);
    this.name = "CloudApiError";
  }
}
```

**`apps/web/src/platform/ws/query-protocol-client.ts`** -- Make `openSocket` use connection mode

The key change: instead of always connecting to `ws://localhost:{port}/ws`, resolve the WebSocket URL from the connection mode. For relay mode, connect to the relay WebSocket and send auth frames. The rest of the protocol is identical.

```typescript
// In openSocket(), replace the hardcoded localhost URL:

// BEFORE:
// const port = await getBackendPort();
// const url = `ws://localhost:${port}/ws`;
// ws = new WebSocket(url);

// AFTER:
import { getConnectionMode } from "@/shared/config/connection";
import { match } from "ts-pattern";

async function openSocket(): Promise<void> {
  const mode = getConnectionMode();

  const { url, onOpen } = match(mode)
    .with({ type: "local" }, (m) => ({
      url: null as string | null, // resolved async below
      getUrl: m.getWsUrl,
      onOpen: (_ws: WebSocket) => {
        // Localhost: server auto-authenticates, sends "connected" frame
      },
    }))
    .with({ type: "relay" }, (m) => ({
      url: `${m.relayUrl}/api/servers/${m.serverId}/ws`,
      getUrl: null,
      onOpen: (ws: WebSocket) => {
        // Relay: send auth token as first frame
        ws.send(JSON.stringify({ type: "authenticate", token: m.deviceToken }));
      },
    }))
    .with({ type: "cloud" }, (m) => ({
      url: `${m.apiBaseUrl.replace("https://", "wss://")}/ws`,
      getUrl: null,
      onOpen: (ws: WebSocket) => {
        ws.send(JSON.stringify({ type: "authenticate", token: m.accessToken }));
      },
    }))
    .exhaustive();

  // Resolve URL (async for local mode which needs port discovery)
  const wsUrl = url ?? (await mode.getWsUrl());

  ws = new WebSocket(wsUrl);

  ws.onopen = () => {
    onOpen(ws!);
  };

  // ... rest of onmessage/onerror/onclose stays identical
  // The "connected" frame handler already works for all modes
  // The "authenticated" frame from relay maps to the same "connected" flow
```

The critical insight: the relay already speaks the same q:* protocol. When a web client connects through `wss://relay.rundeus.com/api/servers/{id}/ws`, the relay pipes frames bidirectionally to the desktop backend. The desktop backend's `handleProtocolMessage` processes them identically. So the frontend's `subscribe()`, `sendCommand()`, `sendToolResponse()` all work unchanged.

### 3 Key Decisions and Rationale

**1. Static SPA on CDN, not SSR/TanStack Start/Workers-rendered.**

Rationale: The app is a real-time WebSocket client. Every page needs a WS connection to show data. SSR would render an empty shell anyway -- you can't server-render data that lives on someone's desktop 2000 miles away. SSR adds a runtime, a deployment surface, hydration complexity, and latency (Worker → origin → render → client → WS reconnect). A static SPA loads from edge cache in one round trip, then opens a WebSocket. Simpler, faster, fewer moving parts. The initial load shows a connection screen while WS establishes -- that's the honest UX.

**2. Two separate backends (cloud API + desktop backend), not one merged backend.**

Rationale: The desktop backend manages SQLite, git worktrees, PTY sessions, file watchers, and agent processes -- things that fundamentally require a local machine. The cloud API manages users, billing, and API keys -- things that fundamentally require a server you control. Merging them creates a chimera that's hard to deploy, hard to test, and hard to reason about. The relay already bridges them. Keep them separate. The cloud API is ~500 lines of Hono routes on Workers. The desktop backend is thousands of lines of system-level code. They share TypeScript types via `shared/`, nothing else.

**3. Connection mode resolved at startup, not per-request.**

Rationale: A request-level `if (isLocal) ... else ...` scattered across 50 call sites is a maintenance nightmare. Resolving the connection mode once and threading it through the WebSocket URL is a single decision point. The query protocol client, the HTTP client, and the auth system each check the mode in exactly one place. This matches how the app actually works: you're either connected to your local machine or you're connected through a relay. You don't switch mid-session.

---

### 1 Acknowledged Tradeoff

**HTTP API fallback for relay/web mode is degraded.** In local mode, the HTTP client hits `localhost:{port}/api` directly for operations like git diffs, file content, PR status. In relay mode, these HTTP requests need to proxy through the relay (or be converted to WS commands). The relay currently only tunnels WebSocket frames, not HTTP. Options: (a) add HTTP proxying to the relay Durable Object, (b) convert all HTTP-only operations to WS commands (the existing `q:command` framework supports this), or (c) accept that some features like diff viewing are desktop-only until we build cloud agents. Option (b) is the clean path forward and aligns with the existing architecture direction, but it's additional work per endpoint.

## Approach: **Progressive Web Shell**

### Philosophy

"Ship the simplest thing that could possibly work -- a static SPA pointed at the user's desktop backend through the existing relay tunnel -- then grow cloud capabilities one at a time. Never rewrite; always extend."

The existing architecture is remarkably well-prepared for this. The relay service, device auth, query protocol, and capabilities system already support remote web clients. The code practically tells us what the smallest valuable step is.

---

### Incremental Delivery Path (Stepping Stones)

**Stone 0 (TODAY):** The app already works as a web client connecting to a user-hosted backend via the cloud relay. `dev.sh` + `VITE_BACKEND_PORT` gives you a browser SPA. Remote pairing via relay tunnel gives you phone/tablet access. The gap: no URL routing, no public deployment, no cloud auth.

**Stone 1 (SHIP FIRST):** Static SPA on Cloudflare Pages with URL routing. Same frontend, deployed to `app.opendevs.com`. User connects to their desktop backend through the existing relay. Add React Router routes so browser navigation works. This is pure frontend work.

**Stone 2:** Cloud auth layer. Cloudflare Worker as a lightweight auth proxy. User signs in (email/magic link or OAuth), then the Worker looks up their server-id and proxies WebSocket to the relay. The frontend connects to the cloud instead of localhost.

**Stone 3:** Cloud backend for team/billing. Cloudflare Worker + D1 database for user accounts, teams, billing. The desktop backend remains the source of truth for workspaces/sessions/agents. The cloud just manages identity and routing.

**Stone 4:** Cloud-hosted agents. The cloud backend can now host its own agent-server instances. Data lives in D1 instead of local SQLite. The frontend is identical -- it speaks the same q:* protocol regardless of whether the backend is local or cloud.

**The framework question:** Do NOT start with a framework. React Router v6 is already in the dependency list. For Stones 0-2, client-side routing is sufficient. A server framework (like Next.js or Remix) only becomes useful at Stone 3+ when you need server-side auth pages, billing webhooks, and SEO. At that point you'd introduce a thin Cloudflare Worker (Hono -- already in the stack) as the edge layer, not a full React meta-framework. The frontend stays an SPA; the cloud layer is a separate service.

```typescript
  | { type: "local"; port: number }        // Electron or dev mode
  | { type: "relay"; serverId: string };    // Web app via cloud relay

/**
 * Resolve how this frontend should connect to a backend.
 *
 * Priority:
 * 1. Electron IPC → local port from main process
 * 2. VITE_BACKEND_PORT → local dev mode
 * 3. URL param or localStorage → relay connection
 * 4. /connect page (user enters server ID)

export function getConnectionMode(): ConnectionMode | null {
  return connectionMode;
}

export function setConnectionMode(mode: ConnectionMode): void {
  connectionMode = mode;
  // Persist relay server ID for reconnect on refresh
  if (mode.type === "relay") {
    try {
      localStorage.setItem("opendevs_server_id", mode.serverId);
    } catch { /* storage unavailable */ }
  }

/**
 * Get the WebSocket URL based on connection mode.
 * Local: ws://localhost:{port}/ws
 * Relay: wss://{RELAY_HOST}/api/servers/{serverId}/client
 */
export async function getWebSocketUrl(): Promise<string> {
  const mode = connectionMode;

  if (!mode) {
    // Fall back to legacy local port resolution for backward compat
    const port = await getBackendPort();
    return `ws://localhost:${port}/ws`;
  }

  if (mode.type === "local") {
    return `ws://localhost:${mode.port}/ws`;
  }

  // Relay mode: connect through cloud
  const relayHost = import.meta.env.VITE_RELAY_URL || "https://relay.opendevs.com";
  const token = getStoredToken();
  return `${relayHost.replace("https://", "wss://").replace("http://", "ws://")}/api/servers/${mode.serverId}/client?token=${encodeURIComponent(token || "")}`;
}
```

```typescript
async function openSocket(): Promise<void> {
  const url = await getWebSocketUrl();

  ws = new WebSocket(url);

  ws.onopen = () => {
    // For relay connections, the relay sends "authenticated" or "auth_failed"
    // For localhost, the backend sends "connected" immediately
    // Both are handled in onmessage below
  };

  ws.onmessage = (evt) => {
    let msg: Record<string, unknown>;
    try {
      msg = JSON.parse(evt.data as string);
    } catch {
      return;
    }

    const type = typeof msg.type === "string" ? msg.type : "";

    match(type)
      .with("connected", () => {
        // Direct local connection — already authenticated
        _connectionId = msg.connectionId as string;
        connected = true;
        connecting = false;
        reconnectAttempt = 0;
        resubscribeAll();
        notifyConnectionChange(true);
        connectResolve?.();
        connectResolve = null;
        connectReject = null;
      })
      .with("authenticated", () => {
        // Relay connection — authenticated through cloud
        connected = true;
        connecting = false;
        reconnectAttempt = 0;
        resubscribeAll();
        notifyConnectionChange(true);
        connectResolve?.();
        connectResolve = null;
        connectReject = null;
      })
      .with("auth_failed", () => {
        // Relay auth rejected — token invalid or server offline
        connecting = false;
        connectReject?.(new Error(msg.message as string || "Authentication failed"));
        connectResolve = null;
        connectReject = null;
      })
      .with("server_offline", () => {
        // Relay says desktop server is offline
        connecting = false;
        connectReject?.(new Error("Desktop server is offline"));
        connectResolve = null;
        connectReject = null;
      })
      // ... rest of existing handlers unchanged (ping, q:snapshot, q:delta, etc.)
```

### 3 Key Decisions and Rationale

**1. No server framework to start.** React Router v6 client-side routing gives us everything Stone 1 needs: deep links, back/forward, bookmarks. Adding Next.js or Remix now would mean rewriting the entire frontend mount path, build pipeline, and deployment model -- all for server-side capabilities we don't need yet. The framework question resolves itself at Stone 3 (cloud auth + billing) and the answer is likely "a separate Hono Worker on Cloudflare" rather than a React meta-framework, because the frontend is already a rich SPA with its own data protocol.

**2. Reuse the relay tunnel as-is.** The `relay.service.ts` + cloud Durable Object already handles the hard problem: tunneling WebSocket traffic from a web client to a desktop backend through a cloud intermediary. The web SPA doesn't need its own backend at all for Stone 1 -- it just needs to connect to the relay instead of localhost. The entire connection-mode abstraction is ~50 lines of code.

**3. URL routing via store sync, not page-level data loading.** The existing architecture loads all data through WebSocket subscriptions and TanStack Query, not through route loaders. Adding React Router loaders would fight the existing pattern. Instead, we sync the URL to the Zustand workspace store and let `useQuerySubscription` do what it already does. The URL becomes a bookmark/share mechanism, not a data-fetching trigger.

---

### 1 Acknowledged Tradeoff

**No server-side rendering or SEO for Stone 1.** The entire app is a client-side SPA. There are no public-facing pages that benefit from SSR (it's a tool for authenticated users, not a content site). If marketing pages or a public workspace viewer are needed later, those would be separate pages on the Cloudflare Worker edge -- not a reason to rewrite the app shell.

## Approach Name: **Injected Cloud Seams**

## Philosophy Statement

"Keep the existing desktop path untouched. Introduce cloud capabilities as new dependencies injected at the boundary, never by modifying what already works. The frontend doesn't know where its data comes from -- it talks to an `ApiTarget` that resolves to either a local backend or a cloud backend. The seam is the URL and the auth header. Everything else is protocol."

---

## Analysis of the Current System

Here is what I found after reading every key file:

**The existing architecture already has most of the seams you need.** Specifically:

1. **`api.config.ts`** resolves a backend URL at runtime (Electron IPC port, `VITE_BACKEND_PORT`, dev middleware). This is the single choke point for "where does the frontend talk."

2. **`query-protocol-client.ts`** builds a `ws://localhost:{port}/ws` URL from `getBackendPort()`. It's a singleton that reconnects and re-subscribes. The protocol is already transport-agnostic -- relay clients go through the cloud relay and arrive at the same `handleProtocolMessage` dispatcher.

3. **`relay.service.ts`** already implements desktop-to-cloud tunneling. Remote web clients connect to a Cloudflare relay, which tunnels frames to the desktop backend. The backend creates "virtual" `WsSendable` connections that look identical to local WebSocket connections.

4. **`capabilities.ts`** checks features by capability name, not platform identity. Good.

5. **`useAuth.ts`** already has a `needsRemoteAuth()` gate -- non-localhost, non-Electron connections need a token. But currently the only auth is device-pairing tokens. Cloud auth (OAuth, sessions) would be a new auth provider.

6. **`client.ts` (ApiClient)** injects a Bearer token for remote connections. The `getBaseURL()` call is async and already supports dynamic URLs.

**The key insight**: Today, "web mode" means "a browser pointed at a user's desktop backend, tunneled through a relay." The new "web product" means "a browser pointed at a cloud backend that owns the data." These are two different `ApiTarget` configurations, not two different apps.

## Implementation Sketch (Real Code)

### Layer 1: ApiTarget -- The Central Seam

The single most important abstraction. Everything downstream (HTTP client, WebSocket client, auth headers) derives from this.

```typescript
// apps/web/src/shared/config/api-target.ts
//
// The seam between "where does data come from?"
// Desktop: local backend on dynamic port
// Web (relay): local backend tunneled through cloud relay  
// Web (cloud): cloud-hosted backend with its own auth

export interface ApiTarget {
  /** Base URL for HTTP REST calls (e.g., "http://localhost:52341/api" or "https://api.opendevs.com/api") */
  getBaseURL(): Promise<string>;
  
  /** WebSocket URL for query protocol (e.g., "ws://localhost:52341/ws" or "wss://api.opendevs.com/ws") */
  getWsURL(): Promise<string>;
  
  /** Auth headers to attach to every HTTP request. Empty object = no auth needed. */
  getAuthHeaders(): Promise<Record<string, string>>;
  
  /** Auth message to send as first WS frame after connect (null = auto-authenticated). */
  getWsAuthFrame(): Promise<Record<string, unknown> | null>;
}
```

Three implementations -- one for each deployment scenario:


```typescript
// apps/web/src/shared/config/targets/relay.target.ts
//

// This is what exists today (device-token pairing).

import { getStoredToken } from "@/features/auth";

export function createRelayTarget(relayUrl: string, serverId: string): ApiTarget {
  return {
    async getBaseURL() {
      // Relay doesn't support HTTP -- all data goes through WS.
      // But we still need an HTTP URL for fallback REST calls.
      // Route them through the relay's HTTP proxy endpoint.
      return `${relayUrl}/api/servers/${serverId}/proxy`;
    },
    async getWsURL() {
      const wsScheme = relayUrl.startsWith("https") ? "wss" : "ws";
      const host = relayUrl.replace(/^https?:\/\//, "");
      return `${wsScheme}://${host}/api/servers/${serverId}/ws`;
    },
    async getAuthHeaders() {
      const token = getStoredToken();
      return token ? { Authorization: `Bearer ${token}` } : {};
    },
    async getWsAuthFrame() {
      const token = getStoredToken();
      return token ? { type: "authenticate", token } : null;
    },
  };
}
```

### Layer 2: Target Resolution

Decide which target to use at app startup:

```typescript
// apps/web/src/shared/config/resolve-target.ts
//
// Determines the ApiTarget based on environment.
// Decision is made ONCE at startup, then cached.

import type { ApiTarget } from "./api-target";
import { createLocalTarget } from "./targets/local.target";
import { createRelayTarget } from "./targets/relay.target";
import { createCloudTarget } from "./targets/cloud.target";
import { isElectronEnv } from "@/platform/electron/invoke";

let _target: ApiTarget | null = null;

export function resolveApiTarget(): ApiTarget {
  if (_target) return _target;

  // 1. Electron desktop -- always local
  if (isElectronEnv) {
    _target = createLocalTarget();
    return _target;
  }

  // 2. Cloud web product (VITE_CLOUD_API_URL is set at build time)
  const cloudUrl = import.meta.env.VITE_CLOUD_API_URL as string | undefined;
  if (cloudUrl) {
    _target = createCloudTarget(cloudUrl, async () => {
      // Cloud auth session provider -- injected later via setCloudSessionProvider
      return _cloudSessionProvider?.() ?? null;
    });
    return _target;
  }

  // 3. Web dev mode with local backend (VITE_BACKEND_PORT set by dev.sh)
  if (import.meta.env.VITE_BACKEND_PORT) {
    _target = createLocalTarget();
    return _target;
  }

  // 4. Relay mode (connecting to someone's desktop through cloud relay)
  // URL params: ?relay=<url>&server=<serverId>
  const params = new URLSearchParams(window.location.search);
  const relayUrl = params.get("relay");
  const serverId = params.get("server");
  if (relayUrl && serverId) {
    _target = createRelayTarget(relayUrl, serverId);
    return _target;
  }

  // 5. Fallback: local dev
  _target = createLocalTarget();
  return _target;
}

// ---- Cloud session injection point ----
// Set by the cloud auth feature when it initializes.
// This is the seam: cloud auth is a plugin, not a hard dependency.

type CloudSessionProvider = () => Promise<{ accessToken: string } | null>;
let _cloudSessionProvider: CloudSessionProvider | null = null;

export function setCloudSessionProvider(provider: CloudSessionProvider): void {
  _cloudSessionProvider = provider;
}
```

```typescript
  const navigate = useNavigate();
  const selectedWorkspaceId = useWorkspaceStore((s) => s.selectedWorkspaceId);
  const selectWorkspace = useWorkspaceStore((s) => s.selectWorkspace);

  // Desktop: no URL sync needed

  // Web: URL → store (on mount or URL change)
  useEffect(() => {
    if (params.workspaceId && params.workspaceId !== selectedWorkspaceId) {

    const targetPath = `/w/${selectedWorkspaceId}`;
    if (currentPath !== targetPath) {
      navigate(targetPath, { replace: true });
    }
  }, [selectedWorkspaceId, navigate]);
}
```

## 3 Key Decisions and Rationale

### 1. ApiTarget interface as the ONLY seam

**Decision**: One interface with 4 methods (`getBaseURL`, `getWsURL`, `getAuthHeaders`, `getWsAuthFrame`) is the entire boundary between "local" and "cloud."

**Rationale**: The existing system already routes all data through two transports: HTTP (ApiClient) and WebSocket (query-protocol-client). Both already resolve their URLs asynchronously. The seam already exists at the URL resolution layer -- we're just naming it and making it injectable. This means zero changes to: the query protocol wire format, the subscription model, TanStack Query hooks, Zustand stores, the entire backend, or any feature-level code.

### 2. No new backend for V1 of web deployment

**Decision**: The web product's first version can still connect to a user's desktop backend through the existing relay tunnel. Cloud-hosted agents come later.

**Rationale**: The relay path already works. Building a cloud backend (database, agent orchestration, billing) is a large project. The `ApiTarget` seam means we can ship the web product with relay-mode first, then swap to cloud-mode by setting `VITE_CLOUD_API_URL` when the cloud backend is ready. No frontend changes needed -- just a new target implementation and the build-time env var.

### 3. URL routing is additive, not transformative

**Decision**: Add routes (`/w/:workspaceId`, `/settings`) alongside the existing `/` route. Desktop mode continues using `/` exclusively. The `useRouteSync` hook bridges URL params to the workspace store only in web mode.

**Rationale**: The entire MainLayout, sidebar, and workspace selection flow works today via Zustand store state. Adding URL routing means syncing that store state to/from the URL -- not rewriting navigation. The hook is ~35 lines, capability-gated, and completely invisible to desktop mode. Back/forward, bookmarks, and deep links all work because the store is the source of truth and the URL is a reflection of it.

---

## 1 Acknowledged Tradeoff

**The relay target's HTTP path is a workaround.** Today the relay only handles WebSocket frames -- there's no HTTP proxy. The `createRelayTarget().getBaseURL()` returns a relay HTTP proxy URL that doesn't exist yet. For V1, this means REST calls (diff stats, file changes, PR status) would need to either: (a) be converted to WS commands (more protocol work), or (b) a small HTTP proxy added to the relay (more infrastructure). I chose to define the interface correctly and flag the gap rather than pretend it doesn't exist. The WS query protocol already handles the heavy-traffic data (workspaces, sessions, messages) -- the HTTP-only calls are low-frequency git operations that can be added to the command protocol incrementally.

## Approach: **Dual-Origin Data Layer**

### Philosophy

The right question isn't "how do we make the frontend talk to two backends" -- it's "what does a connection *to data* actually look like, and how does the frontend stay ignorant of where that data lives?"

The existing architecture already solved the hard problem. The relay service proves it: a `WsSendable` interface abstracts away whether data comes from a local WebSocket or a cloud-tunneled virtual connection. The frontend's `query-protocol-client.ts` doesn't know or care about the transport underneath -- it sends `q:subscribe` frames and gets `q:snapshot`/`q:delta` back.

The web product is the same problem at a different layer. Instead of the *backend* abstracting transport (local WS vs relay tunnel), the *frontend* needs to abstract its data origin (local desktop backend vs cloud API). The data structure that represents "where am I getting my data from" is the critical design decision. Everything else follows.

### The Core Data Structure

The fundamental type the whole system hinges on:

```typescript
// shared/types/data-origin.ts
//
// A DataOrigin is the frontend's answer to: "where does my data come from?"
// This is a discriminated union because the connection parameters, auth flow,
// error states, and capabilities are fundamentally different per origin.
// Trying to unify them into a single shape with optional fields would be
// the "bad taste" Linus warns about — special cases everywhere.

/** Desktop: frontend connects directly to a local backend on localhost. */
interface DesktopOrigin {
  readonly kind: "desktop";
  /** Dynamic port, resolved via Electron IPC or Vite dev middleware. */
  readonly port: number;
  /** WebSocket URL derived from port. Always ws://localhost:{port}/ws */
  readonly wsUrl: string;
  /** HTTP base URL. Always http://localhost:{port}/api */
  readonly httpUrl: string;
  /** No auth token needed — localhost connections are auto-authenticated. */
  readonly token: null;
}

/** Cloud: frontend connects to the cloud API which proxies to a user's desktop. */
interface CloudOrigin {
  readonly kind: "cloud";
  /** Cloud API base URL (e.g., https://app.opendevs.com) */
  readonly apiUrl: string;
  /** WebSocket URL (e.g., wss://app.opendevs.com/ws) */
  readonly wsUrl: string;
  /** HTTP base URL (e.g., https://app.opendevs.com/api) */
  readonly httpUrl: string;
  /** JWT or session token for cloud auth. */
  readonly token: string;
  /** Which server (desktop) we're connected to, if any. */
  readonly serverId: string | null;
}

type DataOrigin = DesktopOrigin | CloudOrigin;
```

Why a discriminated union and not a config bag with optional fields? Because the error states are *different*. A desktop origin can fail because the backend hasn't started yet (port not available). A cloud origin can fail because the user's session expired, or their desktop is offline, or they haven't selected a server yet. You can't handle "desktop offline" and "JWT expired" with the same recovery path. The union makes the compiler enforce that you handle each case.

### Connection State Machine

This is the second critical data structure. The frontend needs to know: can I show data? Why not? What should I show the user?

```typescript
// shared/types/connection-state.ts

/**
 * Connection state machine.
 *
 * States are ordered by progression. The frontend gates rendering on these —
 * you can't show workspaces if you're not connected.
 *
 * Desktop path:  resolving_port → connecting → connected
 * Cloud path:    authenticating → selecting_server → connecting → connected
 *
 * Any state can transition to `error` or `disconnected`.
 * `reconnecting` is a transient state with retry semantics.
 */
type ConnectionState =
  | { status: "authenticating" }
  | { status: "selecting_server"; availableServers: ServerInfo[] }
  | { status: "resolving_port" }
  | { status: "connecting" }
  | { status: "connected"; origin: DataOrigin }
  | { status: "reconnecting"; attempt: number; nextRetryMs: number; lastOrigin: DataOrigin }
  | { status: "disconnected"; reason: DisconnectReason }
  | { status: "error"; error: ConnectionError };

interface ServerInfo {
  serverId: string;
  serverName: string;
  online: boolean;
  lastSeen: string;
}

type DisconnectReason =
  | "server_offline"
  | "auth_expired"
  | "user_logout"
  | "backend_stopped";

type ConnectionError =
  | { code: "PORT_UNAVAILABLE"; message: string }
  | { code: "AUTH_FAILED"; message: string }
  | { code: "SERVER_UNREACHABLE"; message: string }
  | { code: "PROTOCOL_ERROR"; message: string };
```

### Implementation Sketch: The Transport Adapter

The existing `query-protocol-client.ts` is a module-level singleton with `let ws: WebSocket | null`. That's fine for desktop-only. For dual-origin, we need an interface that both desktop and cloud transports implement, and a single instance the React app consumes.

```typescript
// apps/web/src/platform/transport/transport.ts

import type { QueryResource, CommandName } from "@shared/types/query-protocol";

type SnapshotCb = (data: unknown) => void;
type DeltaCb = (upserted?: unknown[], removed?: string[], cursor?: number) => void;
type EventCb = (event: string, data: unknown) => void;
type ConnectionCb = (connected: boolean) => void;

/**
 * Transport — the interface between the React app and its data source.
 *
 * Both desktop (direct WS to localhost backend) and cloud (WS to cloud API)
 * implement this. The React app never imports transport internals directly;
 * it uses this interface through a context provider.
 *
 * Why an interface and not just "swap the WS URL"?
 * 1. Auth is different (none vs JWT)
 * 2. HTTP base URL is different (localhost:port vs cloud domain)
 * 3. Reconnect semantics differ (desktop retries port discovery; cloud retries with token refresh)
 * 4. Cloud needs server selection before connecting (desktop doesn't)
 */
interface Transport {
  /** Connect to the data source. Resolves when the first "connected" frame arrives. */
  connect(): Promise<void>;

  /** Disconnect gracefully. */
  disconnect(): void;

  /** Force reconnect (e.g., backend restart, token refresh). */
  forceReconnect(): void;

  /** Subscribe to a query resource. Returns unsubscribe function. */
  subscribe(
    resource: QueryResource,
    params: Record<string, unknown> | undefined,
    onSnapshot: SnapshotCb,
    onDelta?: DeltaCb,
  ): () => void;

  /** Send an async command. */
  sendCommand(
    command: CommandName,
    params: Record<string, unknown>,
  ): Promise<{ accepted: boolean; commandId?: string; error?: string }>;

  /** Send a tool response. */
  sendToolResponse(requestId: string, result?: unknown, error?: string): boolean;

  /** Listen for ephemeral events. Returns unregister function. */
  onEvent(cb: EventCb): () => void;

  /** Listen for connection state changes. Returns unregister function. */
  onConnectionChange(cb: ConnectionCb): () => void;

  /** Current connection status. */
  isConnected(): boolean;

  /**
   * HTTP base URL for REST calls.
   * Desktop: http://localhost:{port}/api
   * Cloud: https://app.opendevs.com/api
   */
  getHttpBaseUrl(): Promise<string>;
}
```

The existing `query-protocol-client.ts` already implements 90% of this. The refactor is mechanical — extract the interface, wrap the existing module functions:

```typescript
/**
 * Cloud transport — connects to the cloud API's WebSocket endpoint.
 *
 * Key differences from desktop:
 * 1. Auth: sends JWT in the WS upgrade request (cookie or query param)
 * 2. Server selection: the cloud multiplexes multiple desktop backends;
 *    the connect() flow includes picking which server to talk to
 * 3. Reconnect: on auth failure, refreshes the token before retrying
 *
 * The q:* protocol frames are IDENTICAL. The cloud backend proxies them
 * to the user's desktop backend via the existing relay tunnel. The frontend
 * doesn't know the difference — it sends q:subscribe and gets q:snapshot back.
 */
export function createCloudTransport(config: {
  apiUrl: string;
  getToken: () => Promise<string>;
  serverId: string;
}): Transport {
  let ws: WebSocket | null = null;
  let connected = false;
  // ... (mirrors query-protocol-client.ts structure, but with cloud URL + auth)

  return {
    async connect() {
      const token = await config.getToken();
      const wsUrl = `${config.apiUrl.replace("https://", "wss://")}/ws?server=${config.serverId}`;
      ws = new WebSocket(wsUrl, ["opendevs-v1"]);

      // After open, send auth frame (same pattern as relay client_connected)
      ws.onopen = () => {
        ws!.send(JSON.stringify({ type: "authenticate", token }));
      };

      // Wait for "connected" frame — identical to desktop
      // ... rest follows the same pattern as query-protocol-client.ts
    },

    async getHttpBaseUrl() {
      return `${config.apiUrl}/api/servers/${config.serverId}`;
    },

    // ... other methods mirror desktop transport
  };
}
```

```typescript

 *
 * This is where the DataOrigin decision is made — once, at app boot.
 * Everything below this in the tree is origin-agnostic.
 */
export function TransportProvider({ children }: { children: React.ReactNode }) {
  const transport = useMemo(() => {
    // Desktop: Electron app or localhost dev mode
    if (capabilities.ipcInvoke || isLocalhostDev()) {
      return createDesktopTransport();
    }

    // Cloud: deployed web app
    return createCloudTransport({
      apiUrl: import.meta.env.VITE_CLOUD_API_URL || "https://app.opendevs.com",
      getToken: () => getCloudAuthToken(), // from cloud auth module
      serverId: getSelectedServerId(),     // from URL or Zustand
    });
  }, []);

  return (
    <TransportContext.Provider value={transport}>
      {children}
    </TransportContext.Provider>
  );
}

function isLocalhostDev(): boolean {
  return import.meta.env.DEV && (
    window.location.hostname === "localhost" ||
    window.location.hostname === "127.0.0.1"
  );
}
```

### Cloud Backend: What Actually Needs to Exist

The cloud backend is a thin proxy + auth layer. It does NOT replicate the desktop backend. This is the key architectural insight: the existing relay infrastructure already tunnels `q:*` frames between web clients and desktop backends. The cloud backend adds:

1. **Auth (users, sessions, billing):** Standard server-side auth. Cookie-based sessions, OAuth providers. This is a separate Hono app deployed on Cloudflare Workers.

2. **Server directory:** Which desktop backends does this user have? Are they online? This replaces the pairing code flow with a first-class server list.

3. **WebSocket proxy:** Cloud WebSocket endpoint that:
   - Authenticates the user (JWT)
   - Routes `q:*` frames to the correct desktop backend via the existing relay tunnel
   - Returns `server_offline` if the desktop is down

4. **HTTP proxy:** Cloud HTTP endpoints that proxy REST calls to the selected desktop backend, also via the relay tunnel.

```
Cloud Backend (Cloudflare Workers + Durable Objects)
├── /api/auth/*         User auth (login, register, OAuth, billing)
├── /api/servers        Server directory (list user's desktop backends)
├── /api/servers/:id/*  Proxied REST → relay tunnel → desktop backend
├── /ws                 WebSocket → relay tunnel → desktop backend q:* protocol
└── /api/billing/*      Stripe integration (future)
```

The cloud backend does NOT have SQLite, does NOT have agent-server, does NOT run queries. It's a proxy. All data lives on the user's desktop.

### 3 Key Decisions and Rationale

**1. Transport interface, not URL swapping.**

The naive approach is "just change the WebSocket URL." That breaks because auth, reconnect, error handling, and HTTP endpoints are all different between desktop and cloud. The Transport interface makes these differences explicit at the type level while keeping the consumer code (React hooks, API client) identical. The existing `query-protocol-client.ts` becomes one implementation of the interface, not "the" client.

**2. Cloud backend as proxy, not replica.**

Replicating the desktop backend in the cloud (SQLite, agent-server, query-engine) would be a multi-month effort that duplicates every feature and every bug. The relay tunnel already solves the "get data from desktop to web client" problem. The cloud backend is auth + proxy + server directory. When cloud-native agents arrive later, you add a new `DataOrigin` variant and a new transport — you don't retrofit the proxy into a full backend.

**3. DataOrigin as discriminated union, not config bag.**

A config object like `{ url: string, token?: string, serverId?: string }` would work initially but invites bugs: "what happens when token is null but we're in cloud mode?" The discriminated union on `kind` makes illegal states unrepresentable. `match(origin).with({ kind: "desktop" }, ...).with({ kind: "cloud" }, ...).exhaustive()` catches every case at compile time. When cloud-native agents arrive, adding `{ kind: "cloud-native" }` to the union triggers compile errors everywhere you haven't handled it.

### 1 Acknowledged Tradeoff

**Latency for cloud users.** Every `q:*` frame takes a round trip: cloud WS -> relay DO -> desktop backend -> relay DO -> cloud WS. For messages and workspace status this adds 50-150ms depending on geography. For chat (where the user is waiting for an AI agent anyway), this is imperceptible. For sidebar updates and diff stats, the existing stale-while-revalidate pattern in TanStack Query absorbs it. The real pain point will be tool relay — agent requests a browser tool, the request traverses cloud->desktop->cloud->user browser->cloud->desktop->cloud->agent. That's 6 hops. This is acceptable for the MVP but will need optimization (edge caching, request batching) when cloud-native agents eliminate the desktop hop entirely.

### Estimated Complexity and Risk

**Complexity: Medium.** The hardest part (reactive query protocol, relay tunnel, virtual connections) already exists and works. The new work is:
- Transport interface + 2 implementations: ~400 lines
- TransportProvider + context plumbing: ~100 lines

# **TWO BACKENDS, ONE WIRE**

## Philosophy Statement

The cloud backend and the desktop backend are independent services serving independent concerns. The frontend should not know which one it talks to. The q:* wire protocol is already transport-agnostic; we extend that property upward: the frontend connects to *a* backend endpoint, and the backend behind it fulfills queries. Desktop: local SQLite + sidecar. Cloud: Postgres + hosted agents. The frontend code is identical.

---

## Implementation Sketch

### The Core Insight

After reading every key file, I see that the system is already 80% of the way there. The architecture has three clean seams:

1. **`api.config.ts`** -- single point where backend URL is resolved
2. **`query-protocol-client.ts`** -- transport-agnostic q:* pub/sub (talks to whatever WebSocket endpoint you give it)
3. **`capabilities.ts`** -- feature detection decoupled from platform identity

The mistake to avoid: do NOT put auth/billing/cloud services INTO the data backend. The data backend speaks q:*. Cloud services speak REST. These are different concerns. Keep them on different axes.

#### 1. `apps/web/src/shared/config/backend.config.ts` (NEW)

The single seam that determines desktop-vs-cloud. Replaces port-centric thinking with origin-centric thinking.

```typescript
/**
 * Backend Configuration
 *
 * Resolves the data backend origin. This is the SINGLE seam that determines
 * whether the frontend talks to a local desktop backend or a cloud gateway.
 *
 * Three deployment modes:
 *   1. Electron desktop  -> ws://localhost:{port} (IPC-resolved dynamic port)
 *   2. Web dev (dev.sh)  -> ws://localhost:{port} (VITE_BACKEND_PORT env var)
 *   3. Web production    -> wss://app.opendevs.com (same origin as SPA)
 *
 * The cloud gateway handles auth, billing, server directory. It proxies
 * q:* WebSocket traffic to the user's desktop backend (via existing relay)
 * or a cloud-hosted backend (future).
 */

import { isElectronEnv } from "@/platform/electron/invoke";

// ---- Types ----

export interface BackendEndpoints {
  /** HTTP origin for REST (e.g. "http://localhost:52341" or "https://app.opendevs.com") */
  httpOrigin: string;
  /** WebSocket URL for q:* protocol (e.g. "ws://localhost:52341/ws") */
  wsUrl: string;
  /** API base with /api prefix */
  apiBase: string;
}

// ---- Deployment Mode ----

export type DeploymentMode = "electron" | "web-dev" | "web-production";

export function detectDeploymentMode(): DeploymentMode {
  if (isElectronEnv) return "electron";
  if (import.meta.env.DEV) return "web-dev";
  return "web-production";
}

export const deploymentMode: DeploymentMode = detectDeploymentMode();

// ---- Resolution ----

let cachedEndpoints: BackendEndpoints | null = null;
let resolvePromise: Promise<BackendEndpoints> | null = null;

export async function resolveBackendEndpoints(): Promise<BackendEndpoints> {
  if (cachedEndpoints) return cachedEndpoints;
  if (resolvePromise) return resolvePromise;

  resolvePromise = (async (): Promise<BackendEndpoints> => {
    if (deploymentMode === "web-production") {
      // Same-origin: SPA served from Cloudflare Pages alongside gateway Worker.
      // Zero config -- the URL is wherever you deployed.
      const origin = import.meta.env.VITE_CLOUD_GATEWAY_URL as string
        || window.location.origin;
      const wsOrigin = origin.replace(/^http/, origin.startsWith("https") ? "wss" : "ws");
      cachedEndpoints = {
        httpOrigin: origin,
        wsUrl: `${wsOrigin}/ws`,
        apiBase: `${origin}/api`,
      };
      return cachedEndpoints;
    }

    // Electron / web-dev: resolve localhost port
    const { getBackendPort } = await import("./api.config");
    const port = await getBackendPort();
    cachedEndpoints = {
      httpOrigin: `http://localhost:${port}`,
      wsUrl: `ws://localhost:${port}/ws`,
      apiBase: `http://localhost:${port}/api`,
    };
    return cachedEndpoints;
  })();

  resolvePromise.finally(() => { resolvePromise = null; });
  return resolvePromise;
}

export function resetBackendEndpoints(): void {
  cachedEndpoints = null;
  resolvePromise = null;
}

export function isCloudDeployment(): boolean {
  return deploymentMode === "web-production";
}
```

```typescript
// apps/web/src/shared/hooks/useRouteSync.ts (NEW)

import { useEffect } from "react";
import { useParams, useNavigate } from "react-router-dom";
import { useWorkspaceStore } from "@/features/workspace/store";

/**
 * Two-way sync between URL params and Zustand workspace selection.
 * URL is the source of truth in web mode. Zustand is the source in Electron.
 *
 * Web: /w/:workspaceId -> selectWorkspace(workspaceId)
 * Electron: selectWorkspace(id) -> no URL change (single-page, no routing)
 */
export function useRouteSync(): void {
  const { workspaceId } = useParams<{ workspaceId?: string }>();
  const navigate = useNavigate();
  const selectedId = useWorkspaceStore((s) => s.selectedWorkspaceId);
  const selectWorkspace = useWorkspaceStore((s) => s.selectWorkspace);

  // URL -> Store: when URL has a workspace ID, select it
  useEffect(() => {
    if (workspaceId && workspaceId !== selectedId) {
      selectWorkspace(workspaceId);
    }
  }, [workspaceId, selectedId, selectWorkspace]);

  // Store -> URL: when store selection changes, update URL (web mode only)
  useEffect(() => {
    if (!isCloudDeployment()) return; // Electron: no URL routing
    if (selectedId && selectedId !== workspaceId) {
      navigate(`/w/${selectedId}`, { replace: true });
    } else if (!selectedId && workspaceId) {
      navigate("/", { replace: true });
    }
  }, [selectedId, workspaceId, navigate]);
}
```

## 3 Key Decisions and Rationale

### 1. Cloud gateway is a separate service, not an extension of the desktop backend

**Rationale**: Auth, billing, and API key storage require a server with secrets. The desktop backend runs on user machines -- you cannot put Stripe keys or OAuth client secrets there. These are fundamentally different trust domains.

Putting them in the same Hono app would require conditionally loading middleware, route guards that check "am I in cloud mode?", and configuration matrices. That is the definition of complecting.

Instead: two separate services, each doing one thing. The desktop backend handles data. The cloud gateway handles identity and routing. They share the q:* wire protocol as their interface.

### 2. Same-origin serving (SPA + gateway on same domain)

**Rationale**: By serving the SPA from Cloudflare Pages and the gateway API from a Cloudflare Worker on the same domain (e.g., `app.opendevs.com`), we get:
- No CORS configuration needed
- Session cookies work automatically (same-origin, HttpOnly, Secure)
- `window.location.origin` resolves the gateway URL with zero configuration
- WebSocket connections inherit the cookie for authentication

This avoids the entire class of problems around cross-origin auth, token management in localStorage, and CORS preflight overhead.

### 3. URL routing via useRouteSync, not by refactoring navigation

**Rationale**: The existing app uses Zustand for workspace selection. This works perfectly for Electron where there is no URL bar. For web, we need URL routing, but refactoring all navigation to be router-first would touch dozens of files and change the control flow.

Instead, a thin `useRouteSync` hook provides two-way binding between URL params and the Zustand store. In web mode, URL is the source of truth (user can bookmark, use back/forward). In Electron, Zustand is the source of truth (no URL changes). The hook is 30 lines. The rest of the app is unchanged.

## Acknowledged Tradeoff

**The cloud gateway adds a WebSocket hop for relay-connected users.** Today, a remote browser connects to the relay, which forwards frames to the desktop backend -- one hop. With the cloud gateway, it becomes: browser -> gateway -> relay -> desktop backend -- two hops.

This is acceptable because:
- The gateway and relay are both on Cloudflare's edge network (sub-millisecond between them)
- The alternative (browser connects directly to relay, bypassing gateway) means auth has to live in the relay Durable Object, which conflates relay concerns with auth concerns
- For cloud-hosted backends (future), there is only one hop (browser -> gateway -> cloud backend)
- The gateway can be optimized later to hold persistent connections to relay endpoints

## Report: Gateway Architecture for app.opendevs.com

### Approach: **Separate Gateway (Cloud Gateway Worker)**

**Philosophy:** The relay DO is a state machine purpose-built for one job: multiplexing WebSocket connections between a desktop server and its clients. Its data structures (storage keys, tags, alarm scheduling) are tightly coupled to that job. Extending it with HTTP tunneling, OAuth sessions, and billing state would violate the single-responsibility that makes the current code clean. The relay already has exactly the right data model for what it does. A separate gateway is a new concern with its own data structures.

### 1. Separate Gateway vs. Extend the Relay

**Answer: Separate gateway. Do not touch the relay.**

The relay DO is a well-designed state machine with five clean storage keys (`relayToken`, `tunnelRegistered`, `serverDisconnectedAt`, `serverName`, `pending:<clientId>`) and three WebSocket tag groups (`tunnel`, `client`, `pairer`). Its alarm system is tuned to auth timeouts and heartbeats.

HTTP-over-WS tunneling would pollute every one of those concerns:
- Storage needs `pending_http:<requestId>` entries with timeouts
- Alarm scheduling needs to account for HTTP request deadlines (30s) alongside auth deadlines (5s)
- The `handleClientMessage` path, which currently just forwards `data` frames, would need to branch on whether the payload is a q:* frame or an HTTP envelope
- The relay's contract with the desktop backend (`relay.service.ts`) would need to understand a completely new frame type

A separate Cloudflare Worker (the "gateway") sits in front. It has its own Durable Object for session state, uses the relay purely as a transport pipe, and handles all cloud-native concerns (OAuth, billing, server directory) that don't belong in a relay.

### 2. HTTP Through the WS-Only Relay — Data Structure

The key insight: the relay already tunnels arbitrary `data` frames bidirectionally. The relay's `handleClientMessage` (line 471-479 of `relay-do.ts`) forwards any JSON from an authenticated client to the server as `{ type: "data", clientId, payload }`. The server's `relay.service.ts` (line 272-283) parses the inner payload and routes `q:*` frames to the query engine.

HTTP requests are just another payload type inside that existing pipe. The data structure:

```typescript
// ---- HTTP-over-WS Envelope ----
// Tunneled inside the relay's existing `data` frame as the inner payload.
// The gateway serializes HTTP requests into this shape before sending via WS.
// The desktop backend deserializes, dispatches to Hono, and returns the response.

/** Gateway → Desktop (tunneled via relay data frame) */
export interface HttpRequestEnvelope {
  type: "http:request";
  requestId: string;           // UUID, correlates request ↔ response
  method: string;              // "GET" | "POST" | "PATCH" | "DELETE"
  path: string;                // "/api/workspaces/abc123/diff-stats"
  headers: Record<string, string>;  // Subset: content-type, accept
  body: string | null;         // JSON-stringified body, null for GET
}

/** Desktop → Gateway (tunneled via relay data frame) */
export interface HttpResponseEnvelope {
  type: "http:response";
  requestId: string;           // Matches the request
  status: number;              // 200, 404, 500, etc.
  headers: Record<string, string>;  // content-type, cache-control
  body: string | null;         // JSON-stringified response body
}
```

Why this shape:
- `requestId` is the critical field. The gateway holds a `Map<string, { resolve, reject, timer }>` of in-flight HTTP requests. When the response envelope arrives, it resolves the promise by `requestId`. This is the same pattern as the relay's `pending:<clientId>` storage — a correlation map with timeouts.
- `type` prefix (`http:`) distinguishes these from `q:*` frames in the existing `handleProtocolMessage` dispatcher. One new `.with("http:request", ...)` arm in the `relay.service.ts` match block.
- Headers are a flat record, not an array. HTTP headers allow duplicates, but we control both ends and only tunnel headers we need (content-type, authorization is stripped at gateway).
- Body is always string (JSON.stringify). Binary endpoints (file downloads) are out of scope — this is an IDE management API, not a file server.

**On the desktop side**, the handler is ~30 lines in `relay.service.ts`:

```typescript

  } catch (err) {
    sendToRelay({
      type: "data",
      clientId: f.clientId,

        type: "http:response",
        requestId: f.requestId,
```

### 4. Error States — The Critical Path

**Error State 1: Gateway auth succeeds but relay auth fails**

This happens when: cloud session is valid, but the stored device token has been revoked on the desktop (user deleted the device from paired_devices).

```typescript

// Data structure for tracking this:

interface RelayConnectionState {
  status: "connecting" | "authenticating" | "authenticated" | "auth_failed" | "server_offline";
  failedAt?: number;
  failureReason?: string;
  retryCount: number;

// 1. Mark the LinkedServer as stale: `staleSince: Date.now()`
// 2. Return 407 (Proxy Authentication Required) to the web client with:
//    { error: "device_token_revoked", serverId, serverName }
// 3. Web frontend shows "Re-pair required" UI with the pairing code flow
// 4. Do NOT auto-delete the LinkedServer — user might re-pair and want history
```

**Error State 2: Desktop is offline**

The relay already handles this elegantly (line 456-467 of `relay-do.ts`). When a client sends `authenticate` and the tunnel is down, the relay extends the pending deadline to `OFFLINE_WAIT_MS` (5 minutes) and sends `server_reconnecting`. The gateway must surface this to HTTP clients:

```typescript
// For WS clients through the gateway: relay sends `server_reconnecting` →
// gateway forwards to web client. Web client shows "Waiting for desktop..." UI.

// For HTTP requests through the gateway: the gateway holds the fetch() promise
// with a 30s timeout. If the server comes back within 30s, the request completes.
// If not: 504 Gateway Timeout with structured error:
//   { error: "server_offline", serverName, disconnectedAt, retryAfter: 5 }

interface PendingHttpRequest {
  requestId: string;
  resolve: (envelope: HttpResponseEnvelope) => void;
  reject: (error: Error) => void;
  timer: ReturnType<typeof setTimeout>;  // 30s deadline
  createdAt: number;
}

// Map lives in the gateway's WS connection handler, NOT in the DO storage.
// These are ephemeral — if the gateway Worker restarts, pending requests fail.
// That's correct: HTTP requests shouldn't survive process boundaries.
const pendingRequests = new Map<string, PendingHttpRequest>();
```

**Error State 3: Token expires mid-session**

Two tokens can expire:
- **Cloud session** (30-day cookie): Gateway middleware checks `expiresAt` on every request. Returns 401 with `{ error: "session_expired" }`. Web frontend redirects to OAuth flow. All in-flight WS subscriptions are lost — the frontend's React Query cache handles refetch on reconnect.
- **Relay device token**: Can't expire (no TTL in `paired_devices` table), but can be revoked. The gateway learns this only when the relay sends `auth_failed`. At that point, the gateway's WS connection to the relay for that server is dead. All in-flight HTTP requests for that server get `502`.

```typescript
// Gateway's relay WS connection close handler:

  for (const [reqId, pending] of pendingRequests) {
    clearTimeout(pending.timer);
    pending.reject(new Error("relay_connection_lost"));
    pendingRequests.delete(reqId);

  // Mark server as disconnected in gateway state
  serverConnections.delete(serverId);

  // If close code is 4003 (auth rejected), mark LinkedServer as stale
  if (event.code === 4003) {
    // Async — update DO storage
    markServerStale(userId, serverId);
```

### 3 Key Decisions

1. **HTTP dispatches via `app.fetch()`, not a second HTTP server.** The Hono app already exists. Constructing a `Request` object and calling `app.fetch()` reuses every route handler, middleware, and error handler without running a second listener or making a loopback TCP connection. The auth middleware sees `x-relay-connection-id` and trusts the connection is pre-authenticated (since the relay + `relay.service.ts` already validated the device token).

2. **Gateway establishes its own WS connection to the relay as a `client`.** The gateway doesn't get special treatment — it's just another relay client that authenticates with a device token. This means the existing relay code handles it with zero changes. The gateway's connection is long-lived (pool per user-server pair), and HTTP requests are multiplexed over it via `requestId` correlation.

3. **Cloud sessions and server links live in a Gateway Durable Object, separate from the Relay DO.** The relay knows nothing about OAuth, users, or billing. The gateway DO stores `CloudSession` and `LinkedServer` data. This keeps the relay's storage model (5 keys) unchanged and means a relay bug can never leak cloud auth data.

### 1 Tradeoff

**HTTP request latency: +2 WS hops.** Every HTTP request from the web client goes: `Web → Gateway Worker → Relay DO → Desktop Backend`. That's two WS round-trips instead of one direct HTTP call. For a management API (list workspaces, check diff stats), the added ~20-50ms is acceptable. For anything latency-sensitive (streaming agent output), the web client should use the WS subscription path which is already optimized for push. The tradeoff is: keep the relay untouched and accept slightly higher HTTP latency, versus coupling HTTP routing into the relay and saving one hop.

# Architecture Report: Cloud Gateway for OpenDevs Web Product

## Approach Name: **"Gateway Worker — HTTP-over-WS Bridge"**

## Philosophy

**Separate gateway Worker, thin relay.** The relay DO does one thing well: multiplex WebSocket tunnels. The gateway does a different thing: authenticate cloud users, proxy HTTP to desktops, manage billing. These are different responsibilities with different lifecycles, so they should be different Workers. "Make the change easy, then make the easy change."

---

## 1. Separate Gateway Worker or Extend the Relay?

**Separate Gateway Worker.** Here is why:

The relay DO (`RelayDO`) has a clean, narrow responsibility: it holds a server tunnel WebSocket and N client WebSockets, forwards frames between them, manages auth delegation to the desktop, and handles hibernation/alarms. It has zero knowledge of HTTP semantics, cloud auth, billing, or user identity.

The gateway needs to:
- Terminate cloud OAuth (Google, GitHub)
- Map cloud user IDs to desktop server IDs
- Proxy 55 HTTP REST endpoints through a WS-only tunnel
- Manage billing state, server directories, usage metering
- Rate limit by cloud identity (not by IP)

Putting this into the relay DO would violate single responsibility and make the relay untestable in isolation. The relay is a production Durable Object that must hibernate efficiently -- adding D1/KV/R2 bindings and OAuth state machines to it would balloon its surface area.

**The seam:** The relay's responsibility ends at bidirectional frame forwarding. The gateway's responsibility begins at interpreting what those frames mean to a cloud user.

---

## 2. How to Proxy HTTP REST Through a WS-Only Relay

The key insight from reading the code: the relay already forwards arbitrary JSON payloads bidirectionally via `{ type: "data", clientId, payload }` frames. The desktop backend's `ws.service.ts` already routes `q:*` frames through `handleProtocolMessage`. The query engine already handles `q:request` for one-shot data fetches.

But HTTP REST endpoints (55 routes) use Hono handlers that expect `Request` objects and return `Response` objects. We need a bridge.

**The design: HTTP-over-WS request/response envelope.**

A new frame type in the relay protocol:

```typescript
// New frame types added to the relay envelope
// Gateway → Desktop (via relay tunnel)
type HttpRequestFrame = {
  type: "http_request";
  requestId: string;
  method: string;       // GET, POST, PUT, PATCH, DELETE
  path: string;         // /api/workspaces/by-repo?state=ready
  headers: Record<string, string>;
  body?: string;        // JSON stringified
};

// Desktop → Gateway (via relay tunnel)  
type HttpResponseFrame = {
  type: "http_response";
  requestId: string;
  status: number;
  headers: Record<string, string>;
  body?: string;
};
```

On the desktop backend side, a new handler in `relay.service.ts` receives `http_request` frames, constructs a local `fetch()` against `http://localhost:{port}/api/...`, and sends the response back as an `http_response` frame. This reuses **all 55 existing Hono route handlers** with zero changes to route code.

On the gateway side, the Worker receives HTTP requests from the web app at `app.opendevs.com/api/*`, wraps them into `http_request` frames, sends them through the relay, and awaits the `http_response` frame.

**Why not extend the query protocol instead?** Because:
1. The 55 REST routes already exist and work. Rewriting them as `q:*` handlers would be massive, error-prone, and create two code paths for the same operations.
2. HTTP semantics (status codes, headers, content-type negotiation) matter for the web frontend's `ApiClient` -- it expects standard HTTP responses.
3. The desktop frontend's `ApiClient` already makes HTTP calls to `localhost`. The web frontend should use the same `ApiClient` class, just pointed at a different base URL (`app.opendevs.com/api` instead of `localhost:PORT/api`).

---

## 3. Cloud OAuth Integration with Relay Device-Token Auth

Currently, the relay uses **device tokens** (64-char hex strings) that the desktop backend generates and validates. Web clients pair via a 6-digit code displayed on the desktop, get a device token, and use it for all subsequent relay connections.

For the cloud product, we need:
1. **Cloud identity** (Google/GitHub OAuth) managed by the gateway Worker, stored in D1/KV.
2. **Server binding** -- a cloud user claims a desktop server by pairing (same code flow).
3. **Request authentication** -- the gateway validates the cloud session (JWT/cookie), looks up which servers the user has access to, and forwards requests through the relay using the stored device token.

**The key decision: the gateway holds device tokens, not the browser.**

In the current direct-relay flow, the browser holds the device token and sends it to the relay for auth delegation. In the cloud flow, the browser holds a cloud session (JWT cookie from OAuth), and the gateway holds the device token server-side. The browser never sees the device token.

```
Current (direct relay):
  Browser --[device_token]--> Relay --[device_token]--> Desktop

Cloud (gateway):  
  Browser --[cloud_session]--> Gateway --[device_token from DB]--> Relay --> Desktop
```

This is more secure (device tokens never leave the server side) and enables the gateway to manage multiple servers per user, revocation, and billing enforcement.

**`apps/cloud-gateway/src/services/http-bridge.ts`** -- Bridges HTTP to WS relay

```typescript
/**
 * HTTP-over-WS Bridge.
 *
 * Opens a temporary WebSocket to the relay DO, authenticates with a device
 * token, sends an HTTP request envelope, and waits for the response envelope.
 *
 * Each request opens a fresh connection. For high-throughput scenarios,
 * this could be pooled per relay server, but the simplicity of one-shot
 * connections is preferred until profiling proves otherwise.
 * "Duplication is far cheaper than the wrong abstraction."
 */

export interface HttpRequest {
  method: string;
  path: string;
  headers: Record<string, string>;
  body?: string;
}

export interface HttpResponse {
  status: number;
  headers: Record<string, string>;
  body: string | null;
}

const REQUEST_TIMEOUT_MS = 30_000;

export class HttpBridge {
  constructor(
    private relayNs: DurableObjectNamespace,
    private serverId: string,
    private deviceToken: string
  ) {}

  async fetch(req: HttpRequest): Promise<HttpResponse> {
    const doId = this.relayNs.idFromName(this.serverId);
    const stub = this.relayNs.get(doId);

    // Connect to the relay DO as a client
    const wsResponse = await stub.fetch(
      new Request("https://relay/connect", {
        headers: { Upgrade: "websocket" },
      })
    );

    const ws = wsResponse.webSocket;
    if (!ws) throw new Error("server_offline");
    ws.accept();

    return new Promise<HttpResponse>((resolve, reject) => {
      const requestId = crypto.randomUUID();
      let settled = false;

      const timer = setTimeout(() => {
        if (!settled) {
          settled = true;
          ws.close();
          reject(new Error("Request timed out"));
        }
      }, REQUEST_TIMEOUT_MS);

      ws.addEventListener("message", (evt) => {
        let msg: Record<string, unknown>;
        try {
          msg = JSON.parse(evt.data as string);
        } catch {
          return;
        }

        if (msg.type === "authenticated") {
          // Auth succeeded -- send the HTTP request envelope
          ws.send(
            JSON.stringify({
              type: "q:http_request",
              id: requestId,
              method: req.method,
              path: req.path,
              headers: req.headers,
              body: req.body,
            })
          );
        } else if (msg.type === "auth_failed") {
          settled = true;
          clearTimeout(timer);
          ws.close();
          reject(new Error("Device token rejected"));
        } else if (msg.type === "q:http_response" && msg.id === requestId) {
          settled = true;
          clearTimeout(timer);
          ws.close();
          resolve({
            status: msg.status as number,
            headers: (msg.headers as Record<string, string>) ?? {},
            body: (msg.body as string) ?? null,
          });
        } else if (msg.type === "server_reconnecting" || msg.type === "server_offline") {
          settled = true;
          clearTimeout(timer);
          ws.close();
          reject(new Error("server_offline"));
        }
      });

      ws.addEventListener("close", () => {
        if (!settled) {
          settled = true;
          clearTimeout(timer);
          reject(new Error("server_offline"));
        }
      });

      // Authenticate with the device token
      ws.send(JSON.stringify({ type: "authenticate", token: this.deviceToken }));
    });
  }
}
```

## 7. Three Key Decisions

### Decision 1: Self-fetch for HTTP proxy (not rewriting routes)

The desktop backend does a `fetch("http://127.0.0.1:{port}/api/...")` against itself to handle proxied HTTP requests. This means all 55 existing Hono routes work with zero changes. The alternative -- rewriting each route as a `q:*` handler -- would be a 2000+ line change with a parallel code path to maintain forever.

**Trade-off:** Each proxied HTTP request has an extra localhost hop (~0.5ms). Negligible compared to the relay's cross-internet latency.

### Decision 2: Gateway holds device tokens, browser holds cloud session

The browser never touches device tokens. The gateway's D1 database maps `(user_id, server_id) -> device_token`. This means:
- Revoking a user's access is a single DB delete, no browser cooperation needed
- Device tokens stay server-side (more secure)
- Cloud OAuth and desktop pairing are decoupled -- you pair once, the gateway remembers

### Decision 3: Separate Worker, not an extended relay DO

The relay DO stays unchanged. The gateway is a new Worker (`apps/cloud-gateway/`) with its own wrangler.toml, deployed to `app.opendevs.com`. It references the relay's DO namespace as a service binding. This means:
- Relay can be deployed independently (no breaking cloud gateway)
- Gateway can be deployed independently (no breaking relay)
- Testing is isolated -- relay tests don't need OAuth mocks, gateway tests don't need DO storage mocks

---

## 8. One Key Trade-off

**Per-request WebSocket connections for HTTP proxy vs. connection pooling.**

The `HttpBridge` opens a fresh WebSocket to the relay for every HTTP request, authenticates, sends the request, gets the response, and closes. This is simple and stateless but means every HTTP request incurs WebSocket setup overhead (~50-100ms for DO fetch + auth handshake).

For the MVP, this is fine -- the web product's HTTP calls are infrequent (git diffs at 5s intervals, one-off workspace creation). The WebSocket subscription channel handles the high-frequency data (messages, status changes).

If profiling shows this is too slow, the gateway can maintain a pool of authenticated relay connections per server, reusing them across requests. The `HttpBridge` class is already dependency-injected, so swapping in a pooled implementation requires changing one constructor call.

---

## 9. Complexity and Risk Assessment

**Complexity: Medium.** The hardest part is the `HttpBridge` -- managing WebSocket lifecycle for request/response semantics. Everything else (OAuth, D1 queries, WS proxying) is straightforward CF Worker patterns.

**Risks:**
1. **Relay DO cold starts** -- If the DO has hibernated, the first `HttpBridge.fetch()` pays a cold start penalty (~200-500ms). Mitigation: the WS subscription channel keeps the DO warm for active servers.
2. **Request timeout** -- Desktop backend processing time + relay latency must fit within CF's 30s Worker execution limit. All current backend routes return in <1s, so this is safe.
3. **Device token rotation** -- If the desktop regenerates credentials, the gateway's stored token becomes invalid. Mitigation: the gateway catches `auth_failed` from the relay and surfaces it as a "re-pair required" error to the web UI.

# Cloud Gateway Design: HTTP-over-WS Tunneling

## Approach Name: **"Tunnel Everything"**

**Philosophy:** The relay WebSocket tunnel already carries bidirectional JSON frames between web clients and the desktop backend. HTTP REST calls are just request/response pairs. Instead of building a second transport, a second connection, or converting 26 endpoints, we wrap HTTP semantics in WS frames and unwrap them on the desktop backend. One tunnel. One auth model. One code path.

## 1. Architecture Decision: Extend the Relay DO (NOT a separate Worker)

**Answer to Q1: Extend the existing relay DO.**

The relay DO already holds the server tunnel WebSocket, client auth state, and connection lifecycle. A separate Worker would need its own connection to the desktop backend or a way to reach the relay DO's tunnel -- which means either a second tunnel (wasted resources, complexity) or Worker-to-DO RPC (which is just calling the DO anyway). The relay DO is the natural place because it already owns the pipe.

What we add to the cloud layer is a **separate Cloudflare Worker** (`apps/cloud-gateway/`) for cloud-native concerns (OAuth, billing, server directory). This gateway calls the relay DO for tunneling but owns its own state (D1 database for users, subscriptions, server registry). The relay DO gets a small addition for HTTP-over-WS. The gateway Worker is new, separate, clean.

## 2. HTTP Tunneling: HTTP-over-WS (Option A)

**Answer to Q2: Option (a) -- HTTP-over-WS tunneling through the relay.**

Why not the others:
- **(b) Convert HTTP to WS commands**: Requires rewriting 26 endpoints, the query engine, and the frontend HTTP client. Enormous blast radius. The endpoints use Hono middleware, path params, query strings, multipart -- WS commands can't replicate this without reinventing HTTP.
- **(c) Gateway maintains its own connection**: Now the desktop backend needs to accept connections from both the relay AND the gateway. Two tunnels, two auth flows, two reconnect loops. Doubles complexity for no gain.
- **(d) Other**: There is no other. The WS tunnel is the only pipe to the desktop.

The HTTP-over-WS approach is mechanically simple: the cloud gateway receives `POST /api/workspaces` from a web client, wraps it in a JSON envelope `{ type: "http_request", id, method, path, headers, body }`, sends it down the existing tunnel, the desktop backend unwraps it, runs it through the Hono router, wraps the response in `{ type: "http_response", id, status, headers, body }`, sends it back up. The gateway reassembles the HTTP response and returns it to the web client.

## 3. Auth Integration

**Answer to Q3:** Two-layer auth. The cloud gateway validates OAuth sessions (JWT from Cloudflare Access, Google, GitHub -- whatever you pick). Once validated, the gateway looks up which server(s) this user owns in its D1 database, then uses the existing relay token + device token model to authenticate to the relay DO. The web client never sees relay tokens. OAuth session -> gateway -> relay DO -> desktop backend. Device tokens become an internal transport credential, not a user-facing auth mechanism.

## 4. Security Concerns

**Answer to Q4:**
1. **Request size limits**: HTTP-over-WS payloads must be capped (e.g., 1 MB). A malicious client could send enormous bodies through the tunnel. The gateway must enforce Content-Length limits before wrapping.
2. **Path traversal in tunnel**: The desktop backend must validate that tunneled HTTP requests only hit `/api/*` routes. The gateway should strip/reject anything targeting `/ws`, internal paths, or the filesystem.
3. **Timeout amplification**: An HTTP request tunneled over WS has two hops. If the desktop backend is slow (git diff on a huge repo), the gateway must enforce its own timeout and not hold the Cloudflare Worker subrequest forever. Workers have a 30s wall-clock limit anyway, but explicit timeouts prevent resource leaks.
4. **Replay/MITM**: The relay tunnel is WSS (TLS to Cloudflare edge), but the desktop backend connects over `wss://relay.rundeus.com`. The tunnel is encrypted in transit. Internal frames are not signed, so a compromised relay DO could tamper. This is acceptable -- Cloudflare is the trust boundary.
5. **Rate limiting**: The gateway must rate-limit per-user HTTP requests to prevent abuse of the desktop backend's resources (each tunneled request spawns real work on someone's laptop).

```typescript
private async handleHttpTunnel(request: Request): Promise<Response> {
  const tunnel = this.getTunnel();
  const isRegistered = await this.ctx.storage.get<boolean>("tunnelRegistered");
  if (!tunnel || !isRegistered) {
    return Response.json({ error: "server_offline" }, { status: 503 });
  }

  const body = await request.json() as {
    requestId: string;
    method: string;
    path: string;
    headers: Record<string, string>;
    body: string | null;

  this.safeSend(tunnel, {
    type: "http_request",
    requestId: body.requestId,
    method: body.method,
    path: body.path,
    headers: body.headers,
    body: body.body,
  });

  // Wait for http_response from the server (with timeout)
  const response = await this.waitForHttpResponse(body.requestId, 25_000);
  if (!response) {

  return new Response(response.body, {
    status: response.status,
    headers: response.headers,
  });
}
```

The `waitForHttpResponse` uses a promise stored in a Map, resolved when `handleServerMessage` receives an `http_response` frame:

```typescript
// New state on the DO (transient, not in storage -- HTTP requests don't survive hibernation)
private httpPending = new Map<string, {
  resolve: (resp: HttpResponseFrame) => void;
  timer: ReturnType<typeof setTimeout>;
}>();

private waitForHttpResponse(requestId: string, timeoutMs: number): Promise<HttpResponseFrame | null> {
  return new Promise((resolve) => {
    const timer = setTimeout(() => {
      this.httpPending.delete(requestId);
      resolve(null);
    }, timeoutMs);
    this.httpPending.set(requestId, { resolve, timer });
  });
}

// In handleServerMessage, add a case:
case "http_response": {
  if (!isRegistered) break;
  const pending = this.httpPending.get(frame.requestId);
  if (pending) {
    clearTimeout(pending.timer);
    this.httpPending.delete(frame.requestId);
    pending.resolve(frame);
  }
  break;
}
```

**Critical detail:** The DO's `fetch()` for `/http` is called by the gateway Worker via `stub.fetch()`. Cloudflare Workers have a 30s wall-clock limit, so 25s timeout gives margin. The pending map is in-memory (transient) -- if the DO hibernates mid-request, the request times out. This is acceptable for HTTP semantics.

```typescript
// New function:
async function handleTunneledHttp(frame: HttpRequestFrame): Promise<HttpResponseFrame> {
  try {
    // Build a Request object from the frame
    const url = `http://localhost${frame.path}`;
    const init: RequestInit = {
      method: frame.method,
      headers: {
        ...frame.headers,
        // Mark as tunneled so middleware knows this is authenticated
        "x-relay-tunneled": "true",
      },
    };
    if (frame.body && frame.method !== "GET" && frame.method !== "HEAD") {
      init.body = frame.body;
    }

    const request = new Request(url, init);
    
    // Run through the Hono app directly (no network hop)
    const response = await honoApp.fetch(request);
    
    const responseBody = await response.text();
    const responseHeaders: Record<string, string> = {};
    response.headers.forEach((v, k) => { responseHeaders[k] = v; });

    return {
      type: "http_response",
      requestId: frame.requestId,
      status: response.status,
      headers: responseHeaders,
      body: responseBody,
    };
  } catch (err) {
    return {
      type: "http_response",
      requestId: frame.requestId,
      status: 500,
      headers: { "content-type": "application/json" },
      body: JSON.stringify({ error: "Internal tunnel error" }),
    };
  }
}
```

## 3 Key Decisions

1. **HTTP-over-WS through the existing tunnel, not a second connection.** One pipe. The relay DO already manages the tunnel lifecycle, reconnection, hibernation. Adding a second pipe doubles the state machine complexity. Wrapping HTTP in JSON frames is ~100 lines of code on each side. Converting 26 HTTP endpoints to WS commands would be 10x more code and break the Hono middleware chain (auth, workspace-loader, CORS, error handling).

2. **Separate gateway Worker, extended relay DO.** Cloud concerns (OAuth, billing, server directory, D1 database) belong in their own Worker with their own deployment lifecycle. But the relay DO gets the HTTP tunnel addition because it already holds the server WebSocket. The gateway calls `stub.fetch("/http", ...)` to tunnel -- clean separation. Gateway = cloud business logic. Relay = transport.

3. **`honoApp.fetch()` for in-process request execution.** The desktop backend already has a full Hono app with middleware, routes, and error handling. Instead of reimplementing dispatch logic for tunneled requests, we construct a synthetic `Request` and feed it to the Hono app. All middleware (auth, workspace-loader, CORS, error handler) runs exactly as it would for a direct HTTP request. Zero duplication.

## 1 Tradeoff

**Latency.** A tunneled HTTP request has 3 hops: web client -> gateway Worker (Cloudflare edge) -> relay DO -> desktop backend, then the reverse. Minimum added latency is ~50-100ms per request vs direct connection. For git diff endpoints polled at 5s intervals, this is negligible. For user-initiated actions (create workspace, send message), 100ms extra is imperceptible. For large responses (file content, full diffs), the body serialization through JSON adds overhead. If this becomes a problem, the tunnel could support binary frames, but YAGNI -- JSON is fine for now.

## Complexity / Risk Estimate

- **Relay DO changes**: ~100 lines. Low risk. The DO already handles multiple frame types; `http_request`/`http_response` is one more match arm. The pending-response map is transient (not in durable storage), so hibernation just means timeouts, not corruption.
- **Desktop backend changes**: ~80 lines. Low risk. `handleTunneledHttp` is a pure function that constructs a Request and feeds it to Hono. The `x-relay-tunneled` header gating is a 3-line middleware change.
- **Gateway Worker**: ~300 lines. Medium risk. This is new infrastructure (OAuth, D1, JWT validation). But it's isolated -- failures in the gateway don't affect the relay or desktop. OAuth integration depends on which provider you choose.
- **Frontend changes**: ~20 lines. Low risk. Just `getBaseURL()` returning a different origin in cloud mode.
- **Total new code**: ~500 lines across 4 files.
- **Risk hotspot**: The DO's in-memory `httpPending` map. If the DO hibernates between sending `http_request` and receiving `http_response`, the pending promise never resolves (the timeout fires and returns 504). This is correct behavior -- the client retries. But if hibernation is aggressive (Cloudflare can hibernate after ~10s of no WS activity), legitimate slow requests might false-timeout. Mitigation: the heartbeat ping every 30s keeps the DO alive, and the DO won't hibernate while a `fetch()` subrequest is pending.

## DESIGN REPORT: Cloud Gateway Architecture

### Approach Name: **"Three Concerns, Three Workers"**

### Philosophy

*"Every time I complect things, I regret it."* -- Rich Hickey, paraphrased.

The relay (`RelayDO`) is a wire. It multiplexes WebSocket connections between desktop backends and remote clients. That is its one job. OAuth, billing, server directories, and HTTP bridging are fundamentally different concerns that change for different reasons, at different rates, by different people. Mixing them into the relay would produce a 2000-line DO that is impossible to reason about and dangerous to modify.

Instead: three orthogonal systems, each with one purpose.

---

### Answer to Each Question

#### 1. Should the cloud gateway be SEPARATE from the relay, or extend the relay DO?

**Separate.** Unequivocally.

The relay is a **transport primitive** -- it moves bytes. The gateway is a **policy enforcement point** -- it decides who can move bytes, what bytes they can move, and whether they've paid for the privilege. These are fundamentally different concerns.

Concrete proof the relay is already well-bounded: `RelayDO` at 633 lines has zero knowledge of what the `payload` string contains. It does not parse `q:*` frames. It does not know about workspaces, sessions, or messages. It just forwards strings. This is excellent design. Embedding gateway logic would destroy this property.

The gateway is a new Cloudflare Worker (`cloud-gateway`) with its own Durable Objects for user sessions. It sits in front of the relay and uses the relay's existing API surface (`/connect`, `/tunnel`, `/pair`, `/status`) as a downstream service.

#### 2. How should HTTP REST reach the desktop through the WS-only relay?

**HTTP-over-WS bridge in the gateway.** The relay already tunnels arbitrary JSON payloads. The backend already handles `q:request` frames for one-shot query-response patterns. The bridge is thin:

1. Browser sends `GET /api/servers/{serverId}/workspaces/{id}/diff-stats` to the gateway
2. Gateway validates OAuth session, checks billing
3. Gateway wraps the request as a `q:request` frame with a new `"http"` resource type:
   ```json
   { "type": "q:request", "id": "req_1", "resource": "http", "params": { "method": "GET", "path": "/api/workspaces/{id}/diff-stats" } }
   ```
4. Gateway sends this through its relay client connection (it maintains a WS to the relay as an authenticated client)
5. Backend query engine receives the frame, dispatches to the route handler, returns `q:response`
6. Gateway unwraps the response and returns it as HTTPS

This requires one change to the backend: add an `"http"` resource handler to the query engine that internally dispatches to the Hono app. This is ~30 lines of code.

For writes: POST/PUT/DELETE operations use `q:command` (already exists) with a new `"http:mutate"` command, or the existing mutation/command protocol directly.

#### 3. How do cloud OAuth sessions integrate with the relay's device-token auth?

**The gateway translates between identity systems.** The relay only understands device tokens. The gateway only issues device tokens.

Flow:
1. User authenticates via GitHub OAuth at `app.opendevs.com`
2. Gateway creates a cloud session (JWT in cookie or header)
3. When user connects to a server, the gateway checks: does this cloud user have a device token for server X?
4. If yes: gateway injects the device token into the relay WS connection transparently
5. If no: gateway initiates the pairing flow (via the relay's existing `/pair` endpoint) and stores the resulting device token in D1, associated with the cloud user + server

The relay never sees OAuth tokens. The backend never sees OAuth tokens. Only the gateway knows about OAuth. The relay continues to see device tokens exactly as it does today.

#### 4. Are we complecting concerns by mixing auth/billing with relay?

**Yes, absolutely.** The relay's concern is transport reliability (reconnection, heartbeat, multiplexing). Auth's concern is identity verification. Billing's concern is entitlement checking. These change independently:

- Relay changes: new frame types, hibernation optimization, reconnect logic
- Auth changes: add Google login, switch JWT libraries, add MFA
- Billing changes: new plans, usage metering, trial expiration

Mixing them means a billing bug can take down WebSocket tunneling. That is the definition of complecting.

```typescript
//
// The bridge maintains a pooled WS connection per server. Each inbound HTTP
// request becomes a q:request frame with resource "http". The backend's query
// engine dispatches it to the Hono app internally and returns a q:response.
//
// Request correlation: each request gets a unique ID. The bridge awaits the
// matching q:response or q:error frame, with a timeout.

import type { GatewayBindings } from "../types";

const BRIDGE_TIMEOUT_MS = 30_000;

interface PendingRequest {
  resolve: (result: BridgeResponse) => void;
  reject: (err: Error) => void;
  timer: ReturnType<typeof setTimeout>;
}

interface BridgeResponse {
  status: number;
  data: unknown;
}

let requestCounter = 0;

export function createHttpBridge(env: GatewayBindings, serverId: string) {
  // Get or create a relay client connection for this server.
  // In production, this would be pooled per server in a Durable Object
  // to avoid creating a new WS per HTTP request.
  return {
    async request(method: string, path: string, body?: string): Promise<BridgeResponse> {
      const relayDOId = env.RELAY.idFromName(serverId);
      const relayStub = env.RELAY.get(relayDOId);

      // Use the relay's /connect endpoint to get a WS to the desktop backend.
      // For efficiency, a production implementation would pool this connection
      // in a GatewayServerDO (one per server), keeping it alive across requests.
      // For the critical path, we show the one-shot version.

      const requestId = `bridge_${++requestCounter}_${Date.now()}`;

      // Connect to relay as a client
      const connectUrl = new URL(`https://relay.internal/connect`);
      const wsResp = await relayStub.fetch(new Request(connectUrl, {
        headers: { Upgrade: "websocket" },
      }));

      const ws = wsResp.webSocket;
      if (!ws) {
        throw new Error("Failed to establish relay connection");
      }
      ws.accept();

      return new Promise<BridgeResponse>((resolve, reject) => {
        const timer = setTimeout(() => {
          ws.close(1000, "Bridge timeout");
          reject(new Error(`Bridge request timed out after ${BRIDGE_TIMEOUT_MS}ms`));
        }, BRIDGE_TIMEOUT_MS);

        let authenticated = false;

        ws.addEventListener("message", (evt) => {
          let msg: Record<string, unknown>;
          try {
            msg = JSON.parse(evt.data as string);
          } catch {
            return;
          }

          // Phase 1: Authenticate with device token
          if (!authenticated && msg.type === "authenticated") {
            authenticated = true;
            // Send the HTTP-over-WS request as a q:request frame
            ws.send(JSON.stringify({
              type: "q:request",
              id: requestId,
              resource: "http",
              params: { method, path, body },
            }));
            return;
          }

          // Phase 2: Await response
          if (msg.type === "q:response" && msg.id === requestId) {
            clearTimeout(timer);
            ws.close(1000, "Request complete");
            resolve({
              status: 200,
              data: msg.data,
            });
          }

          if (msg.type === "q:error" && msg.id === requestId) {
            clearTimeout(timer);
            ws.close(1000, "Request failed");
            resolve({
              status: (msg.code as string) === "NOT_FOUND" ? 404 : 500,
              data: { error: msg.message },
            });
          }
        });

        ws.addEventListener("close", () => {
          clearTimeout(timer);
          if (!authenticated) {
            reject(new Error("Relay connection closed before authentication"));
          }
        });

        // Start auth with the device token for this server+user
        // (looked up from D1 by the gateway before calling bridge.request)
        const deviceToken = env.DEVICE_TOKEN; // Injected by middleware
        ws.send(JSON.stringify({ type: "authenticate", token: deviceToken }));
      });
    },
  };
}
```

```typescript
      "X-Bridge-Internal": "1",
    },
    ...(body && method !== "GET" ? { body } : {}),
  });

  const contentType = response.headers.get("content-type") || "";
  const data = contentType.includes("application/json")
    ? await response.json()
    : await response.text();

  return { status: response.status, data };
})
```

This is the most elegant approach: zero network hops. The WS frame arrives, the query engine calls the Hono app in-process, gets the response, sends it back as `q:response`. The frontend's `apiClient` doesn't even need to change -- the gateway proxies transparently.

```typescript
// apps/cloud-gateway/src/bridge/ws-proxy.ts
// Proxies WebSocket connections from browser clients through the gateway
// to the relay. The gateway injects the device token so browsers never
// need to know about device-level auth.

import type { Context } from "hono";
import type { GatewayBindings } from "../types";

export async function handleWsProxy(c: Context<{ Bindings: GatewayBindings }>): Promise<Response> {
  const serverId = c.req.param("serverId");
  const user = c.get("user"); // Set by sessionMiddleware

  // Look up device token for this user + server
  const deviceToken = await getDeviceToken(c.env, user.id, serverId);
  if (!deviceToken) {
    return c.json({ error: "Not paired with this server" }, 403);
  }

  // Create two WebSocket pairs:
  // 1. Browser <-> Gateway (the response we return)
  // 2. Gateway <-> Relay (our upstream connection)

  const browserPair = new WebSocketPair();
  const [browserClient, browserServer] = Object.values(browserPair) as [WebSocket, WebSocket];

  // Connect to relay
  const relayDOId = c.env.RELAY.idFromName(serverId);
  const relayStub = c.env.RELAY.get(relayDOId);
  const relayResp = await relayStub.fetch(
    new Request(new URL("https://relay.internal/connect"), {
      headers: { Upgrade: "websocket" },
    })
  );

  const relayWs = relayResp.webSocket;
  if (!relayWs) {
    return c.json({ error: "Server unavailable" }, 503);
  }
  relayWs.accept();
  browserServer.accept();

  let relayAuthenticated = false;

  // Relay -> Browser: forward data frames, handle auth handshake
  relayWs.addEventListener("message", (evt) => {
    try {
      const msg = JSON.parse(evt.data as string);

      if (!relayAuthenticated) {
        if (msg.type === "authenticated") {
          relayAuthenticated = true;
          // Tell browser they're connected (same protocol as direct WS)
          browserServer.send(JSON.stringify({ type: "connected", connectionId: "cloud" }));
        } else if (msg.type === "auth_failed") {
          browserServer.send(JSON.stringify({ type: "error", message: "Server auth failed" }));
          browserServer.close(4003, "Auth failed");
        }
        // Don't forward relay control frames to browser
        return;
      }

      // Forward all data frames to browser (q:snapshot, q:delta, q:event, etc.)
      browserServer.send(evt.data as string);
    } catch {
      // Forward raw if can't parse
      browserServer.send(evt.data as string);
    }
  });

  // Browser -> Relay: forward all frames (q:subscribe, q:command, etc.)
  browserServer.addEventListener("message", (evt) => {
    if (!relayAuthenticated) return; // Buffer or drop until authenticated
    relayWs.send(evt.data as string);
  });

  // Lifecycle: close one when the other closes
  relayWs.addEventListener("close", () => {
    try { browserServer.close(1001, "Upstream closed"); } catch {}
  });
  browserServer.addEventListener("close", () => {
    try { relayWs.close(1001, "Client disconnected"); } catch {}
  });

  // Initiate auth with the relay using the stored device token
  relayWs.send(JSON.stringify({ type: "authenticate", token: deviceToken }));

  return new Response(null, { status: 101, webSocket: browserClient });
}

async function getDeviceToken(
  env: GatewayBindings,
  userId: string,
  serverId: string
): Promise<string | null> {
  // Look up from D1: cloud_user_id + server_id -> device_token
  const result = await env.DB.prepare(
    "SELECT device_token FROM server_access WHERE user_id = ? AND server_id = ?"
  ).bind(userId, serverId).first<{ device_token: string }>();
  return result?.device_token ?? null;
}
```

### 3 Key Decisions

1. **Relay stays untouched.** Zero changes to `RelayDO`. The gateway uses its existing API surface (`/connect`, `/pair`, `/status`) as a downstream service. This means the relay's 452+ tests continue to pass, the production tunnel keeps working, and we can deploy the gateway independently.

2. **HTTP-over-WS via in-process Hono dispatch.** Rather than adding 26 new command types to the WS protocol, we add ONE new query resource (`"http"`) that internally dispatches to the existing Hono app. This means every new REST endpoint automatically works over the relay with zero additional code. The bridge is at the backend side, not the relay side -- keeping the relay dumb.

3. **Gateway translates OAuth to device tokens.** The relay's auth model (device tokens) is simple and proven. Rather than teaching the relay about OAuth, the gateway translates between the two systems. This means the relay's security model doesn't change, the backend's auth middleware doesn't change, and OAuth complexity is contained in one place.

### 1 Tradeoff

**Latency for HTTP-over-WS requests.** Every REST call from cloud users takes an extra hop: browser -> gateway -> relay DO -> desktop backend (and back). For the one-shot connection model shown above, this means a fresh WebSocket per HTTP request, which adds ~100-200ms of connection overhead per request.

The mitigation is **connection pooling**: the gateway maintains a persistent WS connection to each active server via a `GatewayServerDO` (one Durable Object per server). HTTP requests reuse this connection with request-response correlation. This brings latency down to ~50ms over the relay (one WS message round-trip). The tradeoff is DO memory usage -- one persistent connection per active server per gateway region.

For the first version, the one-shot approach is fine. Pooling is an optimization to add when latency matters (which it will for diff stats polling at 5s intervals -- those should become WS subscriptions anyway).

### Complexity Assessment

- **New code:** ~870 lines across the gateway worker
- **Changed code:** ~40 lines in the backend (query engine + events), ~20 lines in frontend config
- **Risk:** Low for relay (zero changes). Medium for gateway (new service, but stateless workers are easy to deploy/rollback). Low for backend (adding one match arm to query engine).
- **Testing:** Gateway needs integration tests with miniflare (mock relay DO). Backend change needs one test for the `"http"` resource. Existing relay tests unchanged.

## Report: Cloud Web Gateway -- Incremental Architecture

### Approach Name: **"Relay-Native Web Client"**

### Philosophy

The relay already does the hard work. The backend already treats relay-tunneled clients identically to local ones. The smallest step is making the **frontend** connect through the relay in cloud mode. No new services, no HTTP proxying, no gateway.

---

### What Already Works (Don't Break This)

1. **RelayDO** (`apps/cloud-relay/src/relay-do.ts`) -- Production Durable Object. Handles server tunnel registration, client auth delegation, bidirectional data forwarding, device pairing, heartbeat, reconnection. Robust, 633 lines, well-tested.

2. **relay.service.ts** (`apps/backend/src/services/relay.service.ts`) -- Desktop backend creates virtual `WsSendable` connections for relay clients that feed into the same `ws.service` + `query-engine` pipeline as local WS clients. **This is the critical design decision that makes everything work** -- relay clients are first-class query protocol citizens.

3. **query-protocol-client.ts** (`apps/web/src/platform/ws/query-protocol-client.ts`) -- Frontend WS client handles `q:subscribe`, `q:snapshot`, `q:delta`, `q:command`, `q:command_ack`, `q:event`, `q:tool_response`. Reconnects with backoff, re-subscribes on reconnect.

4. **Auth system** -- `useAuth` / `needsRemoteAuth` / `getStoredToken` in the frontend, `remote-auth.ts` middleware + `remote-gate.ts` in the backend, device token pairing flow in the relay.

### What's Missing for Cloud Web MVP

**One thing**: The frontend WS client (`query-protocol-client.ts`) always connects to `ws://localhost:{port}/ws`. For cloud web clients, it must connect to `wss://relay.rundeus.com/api/servers/{serverId}/connect` and send a `{ type: "authenticate", token: "..." }` frame after open.

The relay then forwards this to the backend via `client_connected`, the backend validates the device token, sends `auth_response: allowed`, and the relay sends `authenticated` to the client. After that, **all q:\* frames flow identically** through the relay's opaque `data` forwarding.

### Answering the 4 Questions

**Q1: Separate gateway or extend relay?**
**Extend the relay -- minimally.** The relay already does bidirectional WS forwarding. A separate gateway duplicates auth, connections, deployment. For HTTP proxying (later), add a new frame type to the existing relay protocol.

**Q2: HTTP proxying through WS relay?**
**Defer it.** For MVP, convert the boot-critical HTTP calls to WS query protocol frames. The app already has `q:subscribe` for workspaces/sessions/messages/stats. Settings (`GET /settings`) is the only HTTP call needed at boot time -- add `"settings"` as a query resource. Workspace creation, diff stats, etc. can be converted to WS commands/mutations incrementally. HTTP-over-WS proxying is overengineered for what we need now.

**Q3: Cloud OAuth + relay device tokens?**

### 3 Key Decisions

1. **No HTTP-over-WS proxy for MVP.** The WS query protocol already handles all real-time data. Converting the few remaining HTTP calls to WS resources/commands is simpler, more performant, and leverages existing infrastructure. HTTP proxying would require new frame types in the relay, request/response correlation, timeout handling, streaming support -- none of which is needed when the data fits in a WS frame.

2. **Extend the existing WS client, don't create a new one.** The `query-protocol-client.ts` already handles reconnection, subscription management, and protocol dispatch. Adding cloud-mode awareness (different URL + auth frame) is a minimal diff. Creating a separate "relay client" would duplicate all that logic and create a maintenance burden.

3. **Keep device token auth for MVP, defer cloud OAuth.** The pairing flow works today. A cloud web user visits `app.rundeus.com/s/{serverId}`, sees the pairing gate, enters the code from their desktop app, gets a device token, and the full query protocol works. Cloud OAuth is a separate feature that maps cloud identities to device tokens -- it builds ON TOP of what we're shipping, not instead of it.

### 1 Tradeoff

**HTTP endpoints are unavailable in cloud mode until converted to WS.** This means features like diff stats, file content viewing, PR status, and workspace creation won't work in the cloud web MVP. They'll show errors or be disabled. This is acceptable because:
- The core value (viewing chat sessions, seeing workspace status, monitoring agent activity) works entirely through WS subscriptions
- Each HTTP endpoint can be converted to a WS command/resource independently, in priority order
- Trying to proxy all 26 HTTP endpoints through the relay for launch would take weeks and add significant complexity

The conversion priority queue would be:
1. Settings (boot-critical) -- convert first
2. Workspace creation (`POST /workspaces`) -- add as WS command
3. Diff stats / file changes -- add as WS query resources with session-gated polling
4. Repos, agent config -- lower priority, settings-like

### Complexity & Risk

**Complexity: Low.** 1 new file (~80 lines), 3 edited files (~50 lines total), 2 backend additions (~10 lines). The relay, backend query engine, and relay.service.ts require zero changes.

**Risk: Low.** All local-mode behavior is behind an `isCloudMode()` check that returns `false` for localhost and Electron. The relay's data forwarding has been in production. The main risk is the relay auth handshake timing -- between `ws.onopen` and the `authenticated` response, there's a brief unauthenticated window where `subscribe()` calls queue up. The existing `resubscribeAll()` on authentication resolves this naturally.

## Review Results

### **#1 — Security: Auth bypass via spoofable `x-relay-tunneled` header**

Both `authMiddleware` and `remoteGateMiddleware` skip all authentication when the request includes `x-relay-tunneled: true`. This header is intended to be set only by the in-process `app.fetch()` call in the HTTP bridge handler (`server.ts`), but nothing prevents an external HTTP client from sending this header directly. The backend binds to `0.0.0.0` (server.ts line 115), so any network client can bypass auth:

```bash
curl -H "x-relay-tunneled: true" http://<target-ip>:<port>/api/workspaces
```

This gives full unauthenticated access to every API endpoint.

**Fix options:** (a) Strip `x-relay-tunneled` from all incoming HTTP requests in an early middleware, ensuring only the in-process `app.fetch()` path can set it; (b) Use a Hono context variable (`c.set('relayTunneled', true)`) instead of a header — context vars can't be spoofed by external clients; (c) Use a per-request nonce shared only with the bridge handler.

**Files:** `apps/backend/src/middleware/remote-auth.ts:29`, `apps/backend/src/middleware/remote-gate.ts:34`, `apps/backend/src/server.ts:50-53`

### **#2 — Bug: HTTP requests silently sent to localhost:3333 in relay mode when WS disconnects**

In `client.ts`, when `isRelayMode()` is true but `isConnected()` is false (temporary WS disconnect), `request()` falls through to `requestViaFetch()`. In web-production mode, `getBackendPort()` has no relay awareness — it falls through all checks (no `VITE_BACKEND_PORT`, no `window.electronAPI`, not dev mode) to the hardcoded fallback port `3333`. The result: HTTP requests go to `http://localhost:3333/api/...`, which will fail with cryptic network errors in a user's browser.

The code should either throw an explicit "WebSocket disconnected" error in relay mode, or queue/retry requests until the WS reconnects.

**File:** `apps/web/src/shared/api/client.ts:84-91`

## Report

### Approach: "Expand the Protocol, Delete the Bridge"

### Philosophy

The HTTP bridge is a category error. It complects transport (HTTP) with information flow (request-response). Your `q:*` protocol already defines three clean information flow shapes:

- **`q:request/q:response`** -- ask a question, get an answer (one-shot reads)
- **`q:mutate/q:mutate_result`** -- change something, confirm it changed (sync writes)
- **`q:command/q:command_ack`** -- kick off something async, acknowledge receipt

Every one of your 26 HTTP endpoints is exactly one of these shapes. The HTTP bridge exists not because the protocol lacks a concept, but because `QUERY_RESOURCES` has 5 entries when it needs ~20, and `MUTATION_NAMES` has 2 entries when it needs ~12. The fix is enumeration, not a new abstraction.

Tunneling raw HTTP over WebSocket (`http:request/http:response`) brings along method strings, URL path parsing, HTTP headers, status codes, and content-type negotiation -- none of which carry meaning through the relay pipe. They are incidental complexity from a transport layer that doesn't exist in this path.

### Three Decisions

**1. One-shot reads use `q:request`, not `q:subscribe`.** Git diffs, file content, PR status -- these are point-in-time queries that the client polls on its own schedule (5s for working sessions, 30s for PR status). Making them subscribable would require the query engine to track and push-invalidate for resources that aren't backed by DB changes (git state is external). `q:request` is the right tool: ask, receive, done. The existing `q:subscribe` resources (workspaces, sessions, messages, stats) remain subscribable because they are backed by DB writes that trigger `invalidate()`.

**2. Async mutations that trigger background pipelines use `q:command`, not `q:mutate`.** `POST /workspaces` fires a background init pipeline and returns immediately. The caller doesn't need to wait for the pipeline to finish -- it gets the workspace in "initializing" state and subscribes to workspace updates via `q:subscribe("workspaces")`. This is the semantic of `q:command`: "I asked the system to do something; it acknowledged." `q:mutate` is for operations that are done when the response arrives.

**3. HTTP routes survive for desktop (localhost) connections.** The Electron renderer can keep using fetch to `localhost:PORT/api/*`. That path is direct, fast, and doesn't go through the relay. The protocol expansion is **required** for relay mode (web client) and **optional** for desktop. This means zero risk to the working desktop app -- the HTTP routes are still there, still tested, still serving the same service functions. The query engine just calls the same functions via a different dispatch path.

### One Tradeoff

**The query engine becomes larger.** Today it is ~620 lines with 5 query resources and 2 mutations. With the expansion it will be ~900-1000 lines with ~20 query resources and ~12 mutations. Each match arm is small (3-15 lines calling an existing service function), but the file grows. The alternative -- keeping the HTTP bridge -- means the query engine stays small but the system carries two protocol stacks, the bridge handler in `server.ts`, the `HttpRequestFrame/HttpResponseFrame` types, the `pendingHttpRequests` map in the client, and the dual-path routing in `ApiClient`. That distributed complexity is harder to reason about than a larger but self-contained dispatch function. The tradeoff is: one big function vs. two small systems that must stay in sync. Hickey would choose the one thing every time.

If the query engine gets unwieldy later, the `runQuery` function decomposes naturally: extract a `runGitQuery`, `runConfigQuery`, `runFileQuery` that each handle their resource subset. The match remains exhaustive; the bodies move to focused modules. But that's a future concern.

### Complexity and Risk

**Complexity: Low.** No new abstractions. No new frame types. No protocol changes. The relay is untouched. The wire format is untouched. It is strictly additive: more entries in arrays, more arms in match statements, calling the same service functions that already exist and work.

**Risk: Low-to-medium.** The main risk is in the transition: ensuring every frontend service call switches from `apiClient.get(endpoint)` to `queryClient.request(resource, params)` without dropping functionality. This is mechanical but wide (50+ call sites across ~10 service files). A safe strategy: migrate one resource at a time, behind the existing `isRelayMode()` check. Desktop stays on HTTP. Web switches to `q:request`. When all resources are migrated and tested, delete the bridge.

**Async query resources** introduce a second risk. Some match arms (`prStatus`, `ghStatus`) need to call `async` functions (gh CLI). The current `runQuery` is synchronous. Either make it async (changing `handleRequest` to await) or split async resources into commands. The cleanest path: make `runQuery` return `Promise<unknown>` and `handleRequest` await it. The overhead is negligible (a resolved promise for sync resources). This is the one structural change beyond pure enumeration.

### Testing

- **Integration test:** Send `q:request` frames over a test WebSocket for each new resource, verify correct `q:response` shape. Same for `q:mutate` frames.

## Protocol Design Report

### Approach: **"Promote, Don't Unify"**

**Philosophy:** The existing q: protocol has the right bones. The HTTP bridge exists because 26 endpoints needed coverage fast. Rather than build one mega-protocol or keep the bridge forever, promote the HTTP endpoints into the protocol categories they already belong to -- `q:request` for reads, `q:command` for writes that spawn processes, `q:mutate` for synchronous DB writes. The bridge is scaffolding; the protocol is the building.

### Current State Analysis

Here is what exists today, transport by transport:

**WebSocket q: protocol (5 resources, 2 mutations, 11 commands):**
- `q:subscribe`/`q:snapshot`/`q:delta` -- reactive: workspaces, stats, sessions, session, messages
- `q:request`/`q:response` -- **defined in types but never used on frontend**
- `q:mutate` -- archiveWorkspace, updateWorkspaceTitle
- `q:command` -- sendMessage, stopSession, pty:*, fs:*, browser-server:*, git:clone

**HTTP endpoints (13 route modules, ~30 endpoints):**
- **Pure reads (no side effects):** diff-stats, diff-files, diff-file-content, file-content, file-scan, settings, agent-config/*, repos, sessions, messages, stats, onboarding, gh-status, pr-status, pen-files, workspace manifest, setup-logs, workspace/:id
- **Writes that change DB:** POST workspaces, POST sessions, PATCH workspaces, POST repos, POST settings, POST agent-config/*, DELETE agent-config/*
- **Writes that spawn processes:** POST workspace retry-setup, POST open-pen-file, POST tasks/run

**HTTP bridge (bolt-on):** Wraps all HTTP calls as `http:request`/`http:response` over WS for relay mode. Implemented in `server.ts` by calling `app.fetch()` in-process. Works but is structurally wrong -- you are shipping an entire HTTP protocol (methods, headers, paths, status codes, content-type negotiation) inside a protocol that already has typed request/response patterns.

### The Problem with the Bridge

The HTTP bridge works, but it has three costs:

1. **Double serialization.** Every HTTP bridge call: JSON body -> stringified into `http:request.body` -> parsed by Hono -> JSON response -> stringified into `http:response.body` -> parsed by frontend. That is 4 JSON parse/stringify cycles for one round trip.

2. **Untypeable.** The frontend `ApiClient` class sends `string` method + `string` path + `string` body and gets back `string` body. No compile-time safety. Every new endpoint is invisible to the type system until someone writes a service function to call it.

3. **Wrong seam for testing.** To test a relay scenario, you must stand up a full Hono app instance. The q: protocol handlers are plain functions that take `(connectionId, params)` -- trivially testable in isolation.

### The Right Design

**Do not unify.** The existing protocol already has four verbs. Use them:

| What it does | Protocol verb | Examples |
|---|---|---|
| Read data, one-shot | `q:request` | diff-stats, settings, agent-config, file-content, repos, onboarding, gh-status, pr-status |
| Read data, live | `q:subscribe` | workspaces, stats, sessions, session, messages |
| Write data, synchronous | `q:mutate` | archiveWorkspace, updateTitle, saveSettings, saveMcpServers, createRepo |
| Write data + side effects | `q:command` | createWorkspace, sendMessage, stopSession, retrySetup |

The protocol types already define `q:request`/`q:response`. The query engine already handles `q:request` and dispatches to `runQuery()`. The only work is:

1. Widen `runQuery()` to accept more resource names (or add `runRequest()` for non-subscribable reads).
2. Widen `runMutation()` for new sync writes.
3. Move process-spawning endpoints into `runCommand()`.
4. Add a frontend `sendRequest()` function (the `subscribe()` pattern minus the subscription).

```typescript
function handleRequest(connectionId: string, msg: ResourceFrameInput): void {
  const { id, resource, params } = msg;

  try {
    // Try subscribable resource first, then one-shot
    let data: unknown;
    if (isQueryResource(resource)) {
      data = runQuery(toQueryResource(resource), params);
    } else if (isRequestResource(resource)) {
      data = runRequest(toRequestResource(resource), params);
    } else {
      throw new Error(`Unknown resource: ${resource}`);
    }

    // Handle async resources (gh-status, pr-status)
    if (data instanceof Promise) {
      data.then((resolved) => {
        sendFrame(connectionId, { type: "q:response", id, data: resolved });
      }).catch((err) => {
        sendFrame(connectionId, {
          type: "q:error", id, code: "QUERY_ERROR",
          message: err instanceof Error ? err.message : "Request failed",
        });
      });
      return;
    }

    sendFrame(connectionId, { type: "q:response", id, data });
  } catch (err) {
    sendFrame(connectionId, {
      type: "q:error", id, code: "QUERY_ERROR",
      message: err instanceof Error ? err.message : "Query failed",
    });
  }
}
```

```typescript
      pendingRequests.delete(id);
      reject(new Error("WebSocket not connected"));
    }
```

## Approach: "Absorb, Don't Unify" -- Incremental Query Protocol Expansion

### Philosophy

Kent Beck's "Make it work, make it right, make it fast" tells us exactly what to do here. The HTTP bridge **works**. The question is whether it's **right**. My answer: the HTTP bridge is a perfectly correct interim solution, and the right move is to absorb the HTTP endpoints into the q:* protocol incrementally, one natural tier at a time, without a Big Rewrite.

The key insight: **you already have two well-separated protocol categories that map to real semantic differences**:

1. **Reactive data** (subscriptions, deltas, push) -- the q:* protocol handles this well
2. **Request/response operations** (RPC calls with typed params/results) -- these are the HTTP endpoints tunneled through the bridge

The HTTP bridge works because it exploits an existing, well-tested Hono app. The cost is: double serialization (JSON frame -> HTTP Request -> Hono -> HTTP Response -> JSON frame), no streaming, opaque to the protocol layer (you can't see what's being called from the WS logs), and no type safety across the wire.

### What NOT to Do

Do NOT create a "unified protocol" that tries to be everything. The q:* protocol already has clean frame types (subscribe/snapshot/delta for reactive, command/ack for async, mutate/result for sync writes). The HTTP bridge endpoints fall cleanly into these existing categories. The protocol is already unified -- you just haven't migrated the endpoints into it yet.

### The Situation

The q:* protocol is well-designed. Five frame pairs cover every interaction pattern:

| Pattern | Client Frame | Server Frame | Semantics |
|---------|-------------|-------------|-----------|
| Live subscription | `q:subscribe` | `q:snapshot` / `q:delta` | Reactive push |
| One-shot read | `q:request` | `q:response` | Request/response |
| Sync write | `q:mutate` | `q:mutate_result` | Immediate result |
| Async action | `q:command` | `q:command_ack` | Fire-and-ACK |
| Ephemeral push | — | `q:event` | Server-initiated |

The HTTP bridge (`http:request`/`http:response` in `server.ts` lines 46-98) works by constructing an in-process `Request` object and routing it through the full Hono middleware stack. For diff-stats polled every 5s across N workspaces, that is a lot of wasted work.

More importantly, the bridge is untyped. The frontend constructs `{ method: "GET", path: "/api/workspaces/abc/diff-stats" }` as raw strings. There is no compile-time check that the path exists or that the response shape matches expectations.

## Approach: "Typed Dual Protocol"

### Philosophy

The data structures already tell you the answer. You have two fundamentally different data access patterns:

1. **Reactive state** -- small, bounded resources (5 of them) where the server knows when data changes and pushes it. The `q:` protocol with `subscribe/snapshot/delta` is the correct structure for this. It has cursor-based deltas for messages, targeted invalidation with session ID context, and per-connection subscription tracking.

2. **Request/response operations** -- 50+ endpoints with complex URL patterns, HTTP semantics (status codes, content-type negotiation, headers), file content, git process spawning, and external service proxies. These are inherently one-shot, parameterized, and often expensive.

The mistake would be unifying these into one protocol. The HTTP bridge is not a hack -- it's the correct acknowledgment that these are different data patterns. What's wrong is that the HTTP bridge is **untyped, uncancellable, and lacks back-pressure**.

### The Data Model

Here are the three protocol layers, each with a clear invariant:

### Changes to `shared/types/http-bridge.ts`

```typescript
// shared/types/http-bridge.ts
// HTTP-over-WebSocket bridge types.
//
// Design rationale: The q: protocol handles reactive data (subscriptions,
// push, deltas) and lightweight RPC (commands, mutations). The HTTP bridge
// handles everything else: parameterized reads with complex URL patterns
// (diffs, file content), multi-step writes (workspace creation with git
// worktree + DB coordination + progress events), and external service
// proxies (GitHub PR status via gh CLI).
//
// The relay doesn't understand either protocol — it forwards opaque
// payloads. The type discriminator is for the endpoints, not the pipe.

/** HTTP method subset used by our API. */
export type HttpBridgeMethod = "GET" | "POST" | "PUT" | "PATCH" | "DELETE";

/** Sent by the web frontend to request an HTTP endpoint via the WS tunnel. */
export interface HttpRequestFrame {
  type: "http:request";
  requestId: string;
  method: HttpBridgeMethod;
  path: string;
  headers: Record<string, string>;
  body: string | null;
}

/** Client cancels a pending HTTP bridge request. Backend aborts in-flight
 *  Hono dispatch (AbortController) and suppresses the http:response.
 *
 *  Use case: user navigates away from diff view while a large diff-file
 *  request is in flight. Without cancellation, the response still gets
 *  serialized, sent through the relay, and parsed — wasting bandwidth
 *  on data nobody will read. */
export interface HttpCancelFrame {
  type: "http:cancel";
  requestId: string;
}

/** Sent by the desktop backend in response to an http:request. */
export interface HttpResponseFrame {
  type: "http:response";
  requestId: string;
  status: number;
  headers: Record<string, string>;
  body: string | null;
}

/** All HTTP bridge frames sent by the client. */
export type HttpBridgeClientFrame = HttpRequestFrame | HttpCancelFrame;

/** All HTTP bridge frames sent by the server. */
export type HttpBridgeServerFrame = HttpResponseFrame;
```

```typescript
const inFlightBridgeRequests = new Map<string, AbortController>();

setProtocolHandlers({
  onHttpRequest: (connectionId, msg) => {
    const frame = msg as unknown as HttpRequestFrame;
    const conn = getConnection(connectionId);
    if (!conn) return;

    const controller = new AbortController();
    inFlightBridgeRequests.set(frame.requestId, controller);

    const headers: Record<string, string> = { ...frame.headers };
    const request = new Request(`http://internal${frame.path}`, {
      method: frame.method,
      headers,
      body: frame.method !== "GET" && frame.method !== "HEAD" ? frame.body : undefined,
      signal: controller.signal,
    });

      .then(async (response: Response) => {
        // Cancelled — don't send response
        if (controller.signal.aborted) return;

        const responseHeaders: Record<string, string> = {};
        response.headers.forEach((value: string, key: string) => {
          responseHeaders[key] = value;
        });

        const body = await response.text();
        const responseFrame: HttpResponseFrame = {
          type: "http:response",
          requestId: frame.requestId,
          status: response.status,
          headers: responseHeaders,
          body: body || null,
        };

        conn.ws.send(JSON.stringify(responseFrame));
      })
      .catch((err: Error) => {
        if (controller.signal.aborted) return;
        // ... existing error handling
      })
      .finally(() => {
        inFlightBridgeRequests.delete(frame.requestId);
      });

    const controller = inFlightBridgeRequests.get(requestId);

      controller.abort();
```

### 3 Key Decisions

**1. Don't promote HTTP endpoints to q: resources.** The temptation is to turn `GET /workspaces/:id/diff-stats` into `q:request { resource: "diff-stats" }`. Wrong, because:
- The query engine dispatcher (`runQuery`) would need to handle 50+ arms with complex parameter validation, workspace loading middleware, git subprocess spawning, and error handling that Hono routes already do
- The HTTP bridge already uses the Hono router via `app.fetch()` -- all middleware, error handling, and response serialization works automatically
- Diff stats require workspace path resolution, git process spawning with timeouts, and binary detection -- this is not "query a database table"

**2. Remove `QInvalidateFrame`.** It's defined in the type system, exported, but never sent. The backend does push-based invalidation: when data changes, it re-runs queries for active subscribers and sends `q:snapshot`/`q:delta` directly. `QInvalidateFrame` was a "tell the client to refetch" pattern that was never implemented because push is strictly better for this architecture.

**3. Add request cancellation to the HTTP bridge, not to q: protocol.** Subscriptions are inherently long-lived and already have `q:unsubscribe`. The rule is simple: if the data is in `QUERY_RESOURCES` / `COMMAND_NAMES` / `MUTATION_NAMES`, use q:. Everything else goes through the HTTP bridge.
