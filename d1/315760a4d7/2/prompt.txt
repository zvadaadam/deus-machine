Done. Here’s what the Agnt cloud codebase adds to the picture:

Key New Patterns from Agnt

1. Raw Event Passthrough — The biggest insight. Your cloud sidecar is a dumb pipe: BaseAgent.execute() yields raw SDK events unchanged. The backend (AgentSession DO) runs them through an Adapter<TEvent> + EventTransformer pipeline that maps Claude SDK events → typed Parts (TEXT, REASONING, TOOL, STEP_FINISH). This means the sidecar stays thin and stable while the transformation logic evolves in the backend.

2. GeneratorSession + EventTap — Cleaner than your desktop sidecar’s session management. The GeneratorSession keeps a long-lived SDK query alive. Each sendMessage() returns an EventTap { events, detach() }. Detach ≠ terminate — the session stays idle for reuse. waitForIdle() with a FIFO waiter queue prevents concurrent turns naturally.

3. Permission Bridge via PendingRequestStore — Your cloud version already solved external tool approval: canUseTool callback → sidecar emits permission:request → WS → DO → client → user decides → permission.response → WS → sidecar resolves PendingRequestStore<T>. Same pattern for MCP questions and hooks.

4. Turn Lifecycle in Server — The AgentSession DO owns turn creation, state machine, and message persistence — not the sidecar. The sidecar just sends agent:status (running/completed/failed).

5. Session Channel Protocol — Clean bidirectional typed WS protocol: events flow sidecar→DO, commands flow DO→sidecar.

The updated design doc is at .context/agent-server-design.md with all four codebases compared side-by-sid

```typescript
// event-handler.ts

import { match } from "ts-pattern";
import type { AgentEvent } from "../../../shared/agent-events";
import type { QueryResource } from "../../../shared/types/query-protocol";
import { invalidate } from "../query-engine";
import { relay } from "./tool-relay";
import * as persist from "./persistence";

// ---- Invalidation table ----
// Declares which query resources to push after each event type.
// Kept here (not in persistence) because invalidation is a concern of
// the event handler, not the DB writer.

const SESSION_RESOURCES: QueryResource[] = ["workspaces", "sessions", "session", "stats"];
const MESSAGE_RESOURCES: QueryResource[] = ["messages", "session"];

type RespondToAgentFn = (params: {
  sessionId: string;
  requestId: string;
  result: unknown;
}) => Promise<void>;

export type AgentEventHandler = (event: AgentEvent) => void;

export function createAgentEventHandler(deps: {
  respondToAgent: RespondToAgentFn;
}): AgentEventHandler {
  const { respondToAgent } = deps;

  return function handleAgentEvent(event: AgentEvent): void {
    match(event)
      // ── Session lifecycle ───────────────────────────────────────
      .with({ type: "session.started" }, (e) => {
        persistAndInvalidate(persist.persistSessionStarted(e), SESSION_RESOURCES, e.sessionId);
      })
      .with({ type: "session.idle" }, (e) => {
        persistAndInvalidate(persist.persistSessionIdle(e), SESSION_RESOURCES, e.sessionId);
      })
      .with({ type: "session.error" }, (e) => {
        persistAndInvalidate(persist.persistSessionError(e), SESSION_RESOURCES, e.sessionId);
      })
      .with({ type: "session.cancelled" }, (e) => {
        persistAndInvalidate(persist.persistSessionCancelled(e), SESSION_RESOURCES, e.sessionId);
      })

      // ── Messages ────────────────────────────────────────────────
      .with({ type: "message.assistant" }, (e) => {
        persistAndInvalidate(persist.persistAssistantMessage(e), MESSAGE_RESOURCES, e.sessionId);
      })
      .with({ type: "message.tool_result" }, (e) => {
        persistAndInvalidate(persist.persistToolResultMessage(e), MESSAGE_RESOURCES, e.sessionId);
      })
      .with({ type: "message.result" }, (e) => {
        persist.persistMessageResult(e);
      })
      .with({ type: "message.cancelled" }, (e) => {
        persistAndInvalidate(
          persist.persistMessageCancelled(e),
          ["messages", "sessions", "session", "stats"],
          e.sessionId,
        );
      })

      // ── Interaction requests (log only) ─────────────────────────
      .with({ type: "request.opened" }, (e) => {
        console.log(`[AgentEvent] request.opened: session=${e.sessionId} requestId=${e.requestId} type=${e.requestType}`);
      })
      .with({ type: "request.resolved" }, (e) => {
        console.log(`[AgentEvent] request.resolved: session=${e.sessionId} requestId=${e.requestId}`);
      })

      // ── Tool relay ──────────────────────────────────────────────
      .with({ type: "tool.request" }, (e) => {
        console.log(`[AgentEvent] tool.request: session=${e.sessionId} method=${e.method} requestId=${e.requestId}`);
        void relayToolRequest(e, respondToAgent);
      })

      // ── Metadata ────────────────────────────────────────────────
      .with({ type: "agent.session_id" }, (e) => {
        console.log(`[AgentEvent] agent.session_id: session=${e.sessionId} agentSessionId=${e.agentSessionId}`);
        persist.persistAgentSessionId(e);
      })

      .exhaustive();
  };
}

// ---- Helpers ----

function persistAndInvalidate(
  result: persist.WriteResult<unknown>,
  resources: QueryResource[],
  sessionId: string,
): void {
  if (result.ok) {
    invalidate(resources, { sessionIds: [sessionId] });
  }
}

async function relayToolRequest(
  event: AgentEvent & { type: "tool.request" },
  respondToAgent: RespondToAgentFn,
): Promise<void> {
  const { sessionId, requestId, method } = event;

  try {
    const result = await relay(event);
    await respondToAgent({ sessionId, requestId, result });
    console.log(`[AgentEvent] tool.request resolved: requestId=${requestId} method=${method}`);
  } catch (err) {
    console.error(`[AgentEvent] tool.request failed: requestId=${requestId} method=${method}`, err);
    try {
      await respondToAgent({
        sessionId,
        requestId,
        result: { error: err instanceof Error ? err.message : "Tool relay failed" },
      });
    } catch (respondErr) {
      console.error(`[AgentEvent] Failed to send error response to agent: requestId=${requestId}`, respondErr);
    }
  }
}
```

## 3 Key Decisions and Rationale

**1. Directory, not abstraction.** The problem was navigability, not extensibility. Six files among twelve is hard to scan. A directory boundary with a barrel export solves this without adding any runtime abstraction, any indirection, any new concept to learn. The cheapest solution that removes the pain.

**2. Putting it inside `agent/` would conflate the WS query protocol layer with the agent communication layer. ## Philosophy

The data structures are right. The dependency graph is acyclic. The exhaustive pattern matching gives compile-time extension safety. Six flat files with an `agent-` prefix namespace is the correct structure for 1,268 lines. The impulse to restructure is a solution looking for a problem.

### Core Data Structures (all correct)

- **`WriteResult<T> = { ok: true; value: T } | { ok: false; error: string }`** in agent-persistence.ts -- callers decide whether to invalidate based on `result.ok`.
- **`AgentEvent` discriminated union** (12 variants) in shared/agent-events.ts with Zod schema -- validated at the boundary in agent-client.ts `dispatchEvent()`, then dispatched via `match(event).exhaustive()`.

### One Real Bug

**`/Users/zvada/Developer/agent/box-ide/.conductor/dhaka-v6/backend/src/server.ts` line 99** references `agentClient?.disconnect()` but `agentClient` is not defined anywhere in that file. ## 3 Key Decisions and Rationale

1. **Do not introduce subdirectories.** Six files with an `agent-` prefix in a flat `services/` directory have zero naming collisions and a single import depth. A subdirectory adds `../agent/client` import paths for no structural benefit. Reorganize at 15+ files, not 6.

2. **Do not introduce DI container or classes.** The factory pattern already solves the one dependency inversion needed. Module-level singletons with `init()/shutdown()` are the simplest correct pattern for a process that boots once. The existing test suite already handles this with `vi.mock()`.

3. **Do not split one-file-per-command.** Two commands, 157 lines, `match().exhaustive()` for compile-time safety. Split at 8+ commands with 50+ lines each.

## 1 Acknowledged Tradeoff

The module-level singleton pattern (`let client: AgentClient | null = null`) means you can't create two independent AgentService instances in one test process. This is the right tradeoff for a desktop app's single backend process, and the existing tests already handle it correctly with `vi.hoisted()` + `vi.mock()`.

- **Complexity:** One line change.
- **Risk:** Very low. The fix is straightforward -- `agentService.shutdown()` calls `client?.disconnect()` which calls `ws.close(1000)`. Without this fix, the sidecar sees an abrupt TCP drop instead of a clean WebSocket close on every app shutdown.
