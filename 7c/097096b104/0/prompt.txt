## Code Review: `zvadaadam/explore-agent-browser`

### Overview

This branch replaces the browser automation architecture from a complex **frontend-RPC relay** chain (sidecar -> backend -> frontend -> evalWithResult in embedded BrowserView) to a direct **CLI-based approach** where the sidecar executes `agent-browser` CLI commands via `child_process.execFile`.

**Net effect:** +203 / -3,172 lines. Massive simplification that removes 4 frontend files (~2,090 lines), 1 test file (~398 lines), and 15 EventBroadcaster browser RPC methods (~164 lines).

### Architecture Change

| Before | After |
|--------|-------|
| Agent tool -> EventBroadcaster RPC -> backend -> WS q:event -> frontend `useBrowserRpcHandler` -> `evalWithResult` -> BrowserView `executeJavaScript` -> JS IIFE template -> response back through the chain | Agent tool -> `execAgentBrowser(sessionId, args)` -> child_process.execFile -> agent-browser CLI -> CDP -> Chrome |

The frontend BrowserPanel still manages the user-facing browser tabs (navigation, cookies, devtools, screenshots). Only the agent automation path changed.

### Issues Found

After thorough review and validation, **no high-signal bugs were found**. Here's what was investigated and the outcomes:

**1. `formatSnapshotResponse` with undefined snapshot -- NOT A BUG**

Initially appeared that `response.snapshot` (type `string | undefined`) could crash `formatSnapshotResponse`. However, validation confirmed the function signature is `snapshot: string | undefined` and has an explicit guard at line 58:
```ts
if (!snapshot) {
    sections.push("- Page Snapshot: (not available)");
    return sections.join("\n");
}
```

**2. Binary path resolution in `agent-browser-client.ts` -- DEAD CODE**

Line 19: `resolve(require.resolve("agent-browser"), "../../bin/agent-browser.js")` has incorrect path math (`../../` should be `../`). However, `require.resolve("agent-browser")` always throws because the package has no `main`/`exports` field, so the `catch` block runs and falls back to `"agent-browser"` (PATH lookup). The incorrect path never executes.

**3. `execWithSnapshot` doesn't check `snapResult.success` -- SILENT FAILURE**

In `agent-browser-client.ts` line 106, after a successful action, the snapshot sub-command's success status is not checked. If the snapshot fails, the agent receives "Page Snapshot: (not available)" instead of a meaningful error. This is poor error propagation but not a crash or incorrect behavior.
