Agentic Apps Protocol (AAP)

Deus is the first host. The protocol is portable — any IDE, any agent runtime can adopt it.

The protocol for agentic apps — regular web apps augmented so an AI agent can see them, operate them, and collaborate inside them alongside you. One URL, one tool surface, one lifecycle.

01The pitch

Deus today has simulator-specific panels, simulator-specific state, simulator-specific tools. Every new preview (mobile, DB, web, 3D) would mean writing another panel, another state machine, another tool set. AAP inverts this: the app ships its own viewer, its own tools, and its own skill — Deus becomes the launcher, the scheduler, and the bridge.

We call these agentic apps: normal web apps — React dashboards, PDF viewers, 3D canvases, DB browsers — augmented with a small contract so an AI can operate them the way you do. The app keeps its UI, keeps its users, keeps its own repo. AAP is just the handshake.

~2,300
lines of simulator-specific UI & state deletable
~1,500
lines of new generic apps plumbing
1
JSON file to add a new app
Design stance
Not a spec for the ecosystem. An internal contract that could become one. Keep it boring, keep it small, keep it localhost-only until real usage pulls it outward.

02Before & after

Before — simulator-specific

Deus owns the preview.

  • apps/web/src/features/simulator/ui/ — 1,636 LOC
  • apps/backend/src/services/simulator-context.ts — 795 LOC
  • Typed tools hand-registered in deus-tools/simulator.ts
  • Adding a DB browser means a whole new feature folder
  • Cross-cutting concerns (lifecycle, state, stream) reinvented per preview

After — AAP-driven

The app owns the preview.

  • Simulator UI & stream live in packages/device-use
  • Deus renders its URL in a browser tab — same tab infra for every app
  • Tools registered dynamically from the app's own MCP endpoint
  • New app = one agentic-app.json + its own viewer
  • Lifecycle, storage, events are shared plumbing once, not per app

03What you can build

Text-out chat is great for Q&A. It's the wrong UI for anything spatial, temporal, or stateful — a song wants to be heard, a graph wants to be walked, a map wants to be dragged. Agentic apps let the agent step into a real medium and be useful there. Here's what that unlocks.

Flagship use case

Gen-AI node-graph studio

A ComfyUI-style canvas where every node is an AI model — a video model, a music model, an image model, an LLM, a ControlNet, a custom Python cell. You describe what you want; the agent drags nodes in, wires them, and hits run. You watch the graph assemble, frames render, waveforms appear.

When a flow works, you save the graph. When you want a variant, you ask. When you want to remix, you grab a node and drop it elsewhere — the agent adapts the wiring around your change. You stop writing prompts and start composing with an AI that can see your work.

Why this is the killer app. ComfyUI exists today, but the agent lives outside it with no shared state. With AAP, the agent sees node positions, edits graphs by reference, explains why a flow produces the look it does, and can propose three alternatives you actually see rendered.

agent → sees graph JSON, node params, render status
agent → calls add_node, connect, set_param, run
user  → drags nodes, rewires, previews outputs
both  → share graph state in real time
// the AAP manifest is ~30 lines.
// the rest is just a web app that speaks MCP.

The pattern
Chat is a medium for words. Agentic apps give the agent a medium for anything — canvases, timelines, maps, graphs, scenes, boards — and let you be in there with it.
The 80/20 insight
Most of these apps already exist as open-source localhost tools — ComfyUI, Drizzle Studio, Observable, CesiumJS, three.js-editor, DaVinci-lite forks. AAP's leverage is not inventing these apps. It's giving them a shared way to be agent-drivable. Ship the protocol; the ecosystem writes the apps.

04How an app plugs in

Three processes, one manifest. Deus reads the manifest, spawns the app, proxies agent tool calls to it, and points the UI at the app's URL.

The app is just a process. It serves three HTTP endpoints (viewer, MCP, health) and owns whatever it needs underneath — a Swift binary, a Python CLI, a headless Chrome, doesn't matter. Deus stays ignorant of the internals.

05The manifest, annotated

Everything about an app is declared in agentic-app.json, shipped at the root of the app package. Deus discovers it by path reference from its registry.

packages/device-use/agentic-app.json AAP v1 
{
  "$schema": "https://agenticapps.dev/schema/v1.json",
  "protocolVersion": "1",

  "id": "deus.mobile-use",
  "name": "Mobile Use",
  "description": "iOS simulator with agent-drivable tap/type/inspect.",
  "version": "0.1.0",
  "icon": "./assets/icon.svg",
Identity Reverse-DNS id is the permanent handle — never rename it. version is the app's version; protocolVersion is the AAP version the app targets (the key future-proofing lever). 
  "launch": {
    "command": "device-use",
    "args": ["serve", "--port", "{port}"],
    "cwd": "{workspace}",
    "env": { "DEUS_STORAGE": "{storage.workspace}" },
    "ready": {
      "type": "http",
      "path": "/health",
      "timeoutMs": 30000
    }
  },
Launch Template vars: {port}, {workspace}, {storage.workspace}, {storage.global}, {userData}, {sessionId}. Explicit ready probe: http, tcp, or stdout-pattern. No more "process alive but not serving." 
  "ui": {
    "url": "http://localhost:{port}/viewer",
    "surface": "browser-tab",
    "preferredSize": { "w": 400, "h": 800 },
    "theme": "inherit"
  },
UI Just a URL — no HTML ships through AAP. App owns its viewer in whatever stack it wants. surface is reserved for future panel/modal modes. v1 is always a browser tab. 
  "agent": {
    "tools": {
      "type": "mcp-http",
      "url": "http://localhost:{port}/mcp"
    },
    "bootstrap": "Call help for usage. Always snapshot before tapping."
  },
Agent surface Tools come from the app's own MCP server. Deus registers it dynamically via the Claude Agent SDK's setMcpServers — no hand-written proxy. Tools become mcp__{server-name}__{tool-name} automatically. bootstrap is a one-liner returned from launch_app. The full skill lives inside the app's own MCP (expose a help tool) — Deus doesn't handle skill files. 
  "storage": {
    "workspace": "{workspace}/.deus/apps/{id}",
    "global":    "{userData}/apps/{id}"
  },
Storage Two scopes. Deus guarantees dirs exist, gitignores them, cleans on delete. App picks its own format (SQLite, JSON, files). Workspace = per-repo state; global = per-user prefs. 
  "lifecycle": {
    "scope": "workspace",
    "dedupe": "by-workspace",
    "stopSignal": "SIGTERM",
    "stopTimeoutMs": 5000
  },
Lifecycle scope = when does it die (workspace | session | global). dedupe = reuse instances or spawn fresh. Stop = SIGTERM → wait → SIGKILL. 
  "commands": [
    {
      "name": "snapshot",
      "path": "/commands/snapshot",
      "description": "Capture the UI tree",
      "palette": true
    },
    {
      "name": "tap",
      "path": "/commands/tap",
      "description": "Tap a UI element by @ref",
      "palette": true
    }
  ],
Commands (Pi-inspired) User-facing slash commands — appear in Deus's command palette and become typed tools the agent can call. Same HTTP handler, two access paths. Lets the user invoke app actions without going through the agent. 
  "requires": [
    {
      "type": "cli",
      "name": "xcrun",
      "install": "Install Xcode from the App Store"
    },
    { "type": "platform", "os": "darwin" }
  ],
Requires (Pi-inspired) Declared prerequisites. Deus checks them before launch and surfaces a clear error with the install hint — no mystery process crashes. Types: cli, platform, env, port. 
  "events": {
    "channel": "ws",
    "url": "ws://localhost:{port}/deus-events"
  },

  "capabilities": [
    "filesystem:workspace",
    "process:spawn",
    "network:local"
  ]
}
Events & capabilities events is optional — a back-channel for state-changed, notification, request-attention. capabilities is documentary in v1 (all apps are first-party); becomes a consent prompt in v2 when workspace-local manifests land.

06Launch contract

command
string, required
Binary or resolvable name. Deus looks up on PATH, then in the app package's bin/.
args
string[]
Template substitution happens per-arg. Unknown vars throw at launch.
cwd
string
Working dir. Defaults to the app package root, not the workspace — override with {workspace} if the app needs it.
env
Record<string, string>
Templated. Deus always injects DEUS_APP_ID, DEUS_WORKSPACE_ID, DEUS_SESSION_ID, DEUS_PORT.
ready.type
"http" | "tcp" | "stdout-pattern"
How Deus decides the app is serving. Defaults to tcp on the allocated port. Pick http when the server has a real startup phase (compilation) before it's usable.
ready.path
string (http only)
Path to GET. Any 2xx response = ready.
ready.timeoutMs
number, default 30000
After this, Deus kills the process and surfaces a launch error to the user and the agent.
requires
Requirement[], optional (Pi-inspired)
External prerequisites Deus validates before spawn. Missing deps surface as actionable errors with install hints — no cryptic exit-1 crashes. Types: cli (binary on PATH), platform (os / arch match), env (env var must be set), port (specific port available).

07UI contract

Deliberately thin. One URL, one optional size hint. The app owns its viewer stack.

Constraints in v1

  • URL must be http://localhost:{port}/* — no remote origins.
  • Desktop renders via the existing BrowserView infra (same as the browser feature) — cookie partitioning, inspect-mode, devtools all inherit for free.
  • Web renders via <iframe> with sandbox="allow-scripts allow-same-origin". Apps that need MJPEG / streaming must host the stream inside an HTML viewer the app controls, not expose the raw stream URL.
  • Mobile web: apps opt into a mobileSupported: true flag. Un-opted apps are hidden on the mobile layout.
Gotcha — MJPEG & Safari
Native MJPEG decoding is unreliable in Safari. Any app that streams frames must embed them inside an HTML page it serves (canvas or img polling), not rely on the browser's MJPEG support.

08Agent surface

How the agent discovers, launches, and uses agentic apps. Two layers: Deus-level lifecycle tools (always present) and per-app tools (registered dynamically when an app launches).

Deus-level lifecycle tools

Four tools live in the built-in deus MCP server. Always visible to the agent, regardless of what's installed or running.

list_apps()
discovery
Returns all installed agentic apps with id, name, description, and current status.
launch_app(id)
lifecycle
Spawns the app (if not already running) and registers its MCP server with the Claude SDK via setMcpServers. Returns { url, bootstrap, tools: [...] }. The app's own help/skill tool becomes available in the returned tool list. Agent calls it if it needs full usage guidance.
stop_app(id)
lifecycle
Unregisters the MCP server, sends SIGTERM, waits stopTimeoutMs, SIGKILL on timeout.
app_status(id)
introspection
Current state + URL + last error (if crashed). Lets the agent diagnose without reading the UI.

Per-app tools — SDK-native dynamic registration

Each running app hosts an MCP server (HTTP transport). On launch_app, Deus calls the Claude Agent SDK's setMcpServers with the app's endpoint added. Claude sees the app's tools appear, namespaced automatically as mcp__{server}__{tool}. 

// agent-server side — per-app MCP registration
await session.setMcpServers({
  deus: deusServerInstance,                       // lifecycle tools (always)
  deus_mobile_use: {                              // newly launched app
    type: "http",
    url: "http://localhost:47831/mcp",
  },
});
// Claude now sees: mcp__deus_mobile_use__tap, mcp__deus_mobile_use__snapshot, ...
// On stop_app → omit the entry, tools disappear.

Tool naming

Two rules, strictly enforced at manifest-registration time:

  1. Server name = manifest id with . and - replaced by _. (deus.mobile-use → deus_mobile_use.)
  2. Final tool name = SDK-generated mcp__{server}__{tool}. Matches Claude Desktop's convention exactly. We don't hand-concatenate.

Validate total length ≤ 64 chars at manifest load — fail fast if an app's id + longest tool name would exceed the Claude tool-name limit.

Three tool transports

The manifest's agent.tools.type picks how the app exposes its MCP server.

mcp-http (preferred)
HTTP on the app's port
App hosts an MCP server at an HTTP path. Deus adds it via setMcpServers({ type: "http", url }). Used by any app that runs an HTTP server anyway (most of them — the viewer already needs HTTP).
mcp-stdio
separate process
Deus spawns a second child process whose stdin/stdout speaks MCP. For apps that can't host HTTP.
cli
fallback
App ships only a CLI; agent drives via Bash. Zero integration work — suitable for bringing in existing tooling (drizzle-kit, supabase) before they add MCP.

Skills — deferred, owned by the app

The agent's full usage guide (mental model, workflows, common pitfalls) is not injected into the system prompt. Instead, the app's own MCP server exposes a help tool. Agent pulls the guide on demand.

Why deferred, not eager:

  • Context-compaction-safe. If the conversation compacts and the skill falls out, the agent can help again.
  • Relaunch-safe. Stop + start re-exposes the tool; no state to re-inject.
  • Flexible. Each app owns its own skill format — Deus doesn't need to know about markdown, YAML frontmatter, or Claude skill conventions.
  • Respects SDK constraints. The Claude Agent SDK's systemPrompt is fixed at session start; we can't dynamically inject anyway.

The bootstrap field in the manifest is a short always-on hint (returned from launch_app as text). That's the one-liner; help is the full doc.

Slash commands — palette dual surface Pi-inspired

The manifest's commands field produces entries that are both typed MCP tools (for the agent) and entries in Deus's existing command palette (apps/web/src/features/command-palette/). One handler in the app, two invocation paths.

User invokes

/mobile-use.snapshot

  • Palette entry added to staticCommands per running app
  • Format: /{app-short-name}.{command} — avoids collisions
  • cmd+K palette search resolves it
  • Handler calls POST {app-url}/commands/snapshot

Agent invokes

mcp__deus_mobile_use__snapshot

  • Registered as MCP tool by the app's own server
  • Same HTTP handler fires
  • Typed schema from MCP tools/list
  • No duplicate code in the app

09Storage

Two scopes. Managed by Deus, used by the app.

storage.workspace
{workspace}/.deus/apps/{id}/
Per-workspace. Gitignored. Dies with the workspace. Use for: pinned simulator UDID, cached screenshots, per-project settings.
storage.global
{userData}/apps/{id}/
Per-user. Survives workspaces. Use for: user's default preferences, cached downloads, auth tokens (later).

Contract

Deus guarantees

  • Both dirs exist before launch.command runs
  • Paths are passed via env template vars
  • Workspace storage is gitignored by default
  • Dirs are cleaned on workspace-delete / app-uninstall

App agrees to

  • Write only inside its allocated paths
  • Handle either dir being empty (first-run)
  • No hardcoded absolute paths
  • Survive storage being cleared between launches

10Lifecycle

scope — when it dies

workspace
default
One instance per workspace. Dies on workspace close. Right for: mobile-use, db-browser, web-preview.
session
One per chat session. Dies when session ends. Right for: disposable sandboxes, scratch environments.
global
Long-lived singleton. Survives workspace switches. Rare. Right for: heavyweight local servers you don't want to cold-start (local LLMs, vector DBs).

Five states, enforced by the backend's apps.service.ts.

Persistence — the SQLite table

CREATE TABLE running_apps (
  id           TEXT PRIMARY KEY,        -- uuid
  app_id       TEXT NOT NULL,           -- "deus.mobile-use"
  workspace_id TEXT,                    -- null for scope=global
  session_id   TEXT,                    -- only for scope=session
  pid          INTEGER NOT NULL,
  port         INTEGER NOT NULL,
  url          TEXT NOT NULL,
  status       TEXT NOT NULL,           -- "starting" | "ready" | "stopping" | "stopped" | "crashed"
  started_at   INTEGER NOT NULL,
  last_seen_at INTEGER NOT NULL,
  exit_code    INTEGER,                 -- populated on stopped/crashed
  stderr_tail  TEXT                     -- last ~1KB of stderr on crash (for card hover)
);

CREATE INDEX idx_running_apps_workspace ON running_apps(workspace_id);
CREATE INDEX idx_running_apps_app_scope ON running_apps(app_id, workspace_id);

State transitions — every change fires three effects

Always, in parallel

  • DB write — update status and, on exit, exit_code + stderr_tail
  • WS delta — broadcast to frontend so launcher cards + sidebar dot update
  • RPC to agent-server — call setMcpServers to register or unregister the app's MCP server

Watchdog — no polling

  • Backend retains the spawned child-process reference
  • Native Node child.on("exit") is the source of truth
  • No heartbeats, no health checks — exit event ⇒ state change
  • If an app hangs without crashing (alive but frozen), the user clicks Stop

Cleanup — four triggers

Trigger 1 · Backend boot
Scan running_apps. For each row, check kill -0 pid. Any alive process orphaned by a crashed backend is killed — fresh start = clean state. Dead rows are deleted.
Trigger 2 · Workspace close / archive
Hook into workspace.service.ts. Kill all apps where workspace_id = X. SIGTERM → wait stopTimeoutMs → SIGKILL.
Trigger 3 · Explicit stop
stop_app agent tool, or the Stop button on a launcher card. Fires the same shutdown sequence.
Trigger 4 · Process crash mid-session
Watchdog catches child.on("exit") while status = "ready". Transition to crashed, capture exit_code and stderr_tail. In-flight tool calls get a structured error so the agent can relaunch or surface the failure to the user. Launcher card goes red.

11Events (optional)

A small WebSocket back-channel for apps that need to push state to Deus asynchronously. Apps that don't need it skip the events field.

App → Deus

{ "type": "state-changed",     "resource": "snapshot" }
{ "type": "notification",      "level": "info", "text": "Build finished" }
{ "type": "request-attention" }

Deus → App

{ "type": "workspace-changed", "workspaceId": "..." }
{ "type": "session-started",   "sessionId": "..." }
{ "type": "session-ended",     "sessionId": "..." }
{ "type": "theme-changed",     "theme": "dark" }
{ "type": "stop" }              // graceful shutdown request
Why not postMessage?
Two reasons: (1) MCP Apps use postMessage because they ship HTML inside the host — we don't, we point at a localhost URL. (2) Same-origin iframe → host postMessage works fine, but HTTP is easier to debug (curl, network panel, replayable). Reserved as a future transport when AAP v2 adds cross-origin apps.

Documentary in v1. Becomes a real permission model in v2 when workspace-local manifests land.

13Launch sequence

Two converging paths — agent-initiated and user-initiated — both end at the same backend method. Source of truth = backend's apps.service.ts. Agent-server and frontend are notified on completion.

Path A — agent calls launch_app("deus.mobile-use")

A1 · agent-server receives tool call
The built-in launch_app tool (in the deus MCP server) fires. Agent-server sends an RPC to the backend: LaunchAppRequest { app_id, workspace_id, session_id }.
A2 → converge
Backend runs the shared spawn method. See below.

Path B — user clicks Launch on a launcher card

B1 · frontend sends WS command
q:command { cmd: "launchApp", app_id, workspace_id } over the existing WS query protocol. Backend's command handler calls the shared spawn method.
B2 → converge
Same shared spawn. See below.

Shared spawn in apps.service.ts

1 · Resolve manifest
Look up app_id in the registry (apps/backend/src/config/apps.ts), read the manifest JSON.
2 · Dedupe check
Query running_apps WHERE app_id AND workspace_id match. If row exists and PID alive, return its URL + skip spawn (and still RPC the agent-server to setMcpServers if it wasn't already registered for this session).
3 · Validate requires
Check declared prerequisites (cli, platform, env, port). If any fail → return error with install hint. No spawn.
4 · Allocate port
Bind 0.0.0.0:0, capture assigned port, release. Accept the (rare) race — retry once on EADDRINUSE.
5 · Ensure storage
mkdir -p both storage.workspace and storage.global. Add .deus/ to the workspace's gitignore if missing.
6 · Spawn + persist
spawn(command, args, { cwd, env }). Insert row with status = "starting". Retain the child-process reference in-memory. Attach child.on("exit") watchdog immediately.
7 · Ready probe
Poll ready.path every 200ms up to timeoutMs. On success → status = "ready". On timeout → SIGKILL, mark crashed, emit error.
8 · Notify
Three parallel effects:
— WS delta to all frontend subscribers: running_apps updated.
— RPC to agent-server: call setMcpServers with the new app added. Claude sees tools appear.
— Response to caller (Path A = RPC reply, Path B = command ack): { url, bootstrap, tools: [...] }.
9 · Frontend opens the tab
On launcher-card click (Path B), the frontend opens a new Browser content-tab pointing at url. For Path A, the agent gets the URL in its tool result and decides whether to also trigger a tab-open via a follow-up action.

15In scope / out of scope for v1

In scope — v1

  • Single manifest format (agentic-app.json)
  • Hardcoded app registry in TS (1–2 apps)
  • Workspace-scoped lifecycle, SQLite-persisted
  • MCP-HTTP / MCP-stdio / CLI tool transports
  • Markdown skills, loaded-on-launch
  • Two storage scopes (workspace + global)
  • BrowserView / iframe rendering
  • Optional WebSocket events channel
  • Slash commands → command palette (Pi)
  • requires prereq validation (Pi)

Out of scope — v2+

  • Remote (non-localhost) apps
  • Third-party / workspace-local manifests
  • Permission prompts & capability enforcement
  • Secrets / Keychain integration
  • postMessage cross-origin bridge
  • Multi-instance (same app, two copies per workspace)
  • Hot reload on manifest change
  • deus app install CLI (lands with 3rd-party)
  • App marketplace / discovery UI

16Inside Deus

The protocol above is the contract agentic apps speak. This section is the part Deus itself implements — the sidebar, the naming, the file layout. A handoff spec for the PR that builds the IDE side.

Everything lives in the Browser content tab

Apps are websites. So the launcher and the apps themselves both live as Browser tabs, using Deus's existing Browser content tab and its tab-registry. No new content-tab primitive.

The flow:

  1. Sidebar Apps entry → click → activates the Browser content tab and opens/focuses an internal Apps launcher tab (served by an internal deus://apps URL or a special tab-type marker; React-rendered, not an iframe).
  2. Launcher tab shows a grid of cards (one per installed app). Click Launch → opens a new Browser tab pointing at the app's localhost URL (e.g. http://localhost:47831/viewer).
  3. The launcher tab and each app tab are peers in the Browser tab strip. Users can close any independently.

The sidebar Apps entry is therefore a navigation action, not a view: its click handler activates Browser + focuses the Apps launcher tab. It carries no launcher UI itself.

Sidebar mount: wherever the existing Repositories section lives (likely apps/web/src/app/layouts/). Insert <AppsSidebarEntry /> above the repositories section.

18Testing the protocol

How we know AAP v1 is right before writing code.

The paper-exercise rule
Draft agentic-app.json for three candidate apps on paper before merging PR 1:
1. deus.mobile-use — today's simulator
2. drizzle.studio — a DB browser
3. deus.web-preview — a Vite-app viewer

If all three fit without adding fields, ship v1. If any requires an addition, that field belongs in v1. If any requires two+ additions, the abstraction is wrong — revisit before coding.

Not negotiable in v1

  • Schema validation at launch — bad manifests fail fast
  • SQLite-backed lifecycle — in-memory will leak zombies
  • Typed MCP tools for the simulator — don't regress drivability to CLI-via-Bash just to prove the protocol works
  • Orphan cleanup on backend boot — the single biggest reliability win of formalizing lifecycle
