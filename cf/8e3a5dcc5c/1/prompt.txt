**Bug 1 — `sidecar/agents/codex/codex-handler.ts`: `emitAssistantMessage` fires on every streaming delta**

The `ts-pattern` match arm uses `P.union("item.delta", "item.completed")` and calls `EventBroadcaster.emitAssistantMessage(...)` inside that single branch. This means every streaming chunk from the Codex SDK — not just the final completed item — triggers a call to `emitAssistantMessage`. The backend's `persistAssistantMessage` in `agent-persistence.ts` performs an `INSERT` into `session_messages` on each call. A single Codex turn with 200 streaming deltas will insert 200 rows into the DB, most containing partial/intermediate text. The old code had a separate branch that only persisted `item.completed`. This is a data corruption bug.

**Fix:** Split the `P.union` into two separate `.with()` arms. The `item.delta` arm handles streaming UI updates (if any); the `item.completed` arm calls `emitAssistantMessage`.

---

**Bug 2 — `backend/src/services/query-engine.ts` ~line 1165: Silent message drop on agent-server failure**

In the `sendMessage` command handler, `forwardToAgent(...)` is called fire-and-forget after `writeUserMessage` succeeds:

```typescript
if (forwardToAgent) {
  forwardToAgent({...}).catch((err) => {
    console.error("[QueryEngine] Failed to forward to agent-server:", err);
  });
}
return { commandId: result.messageId };
```

The command ACK (`{ commandId }`) is returned to the frontend immediately regardless of whether the forward succeeds. If the agent-server is disconnected or the RPC call rejects, the error is only logged. The user message is persisted in the DB, the frontend receives the ACK and renders the message as sent, but the agent-server never receives `turn/start`. Because `session.started` is emitted by the sidecar only upon receiving `turn/start`, the session status is never updated to `working`. The frontend shows the message as sent with no error and no agent response — the message is silently dropped.

**Fix:** Either await `forwardToAgent` before returning the ACK and reject the command on failure, or — if fire-and-forget is intentional for latency reasons — emit a `session.error` event back to the frontend in the `.catch()` handler so the user sees the failure.

**Bug 3 — `sidecar/agents/query-lifecycle.ts`: `emitSessionCancelled` and `emitMessageCancelled` both called on cancellation**

`persistCancellation` now emits:

```typescript
EventBroadcaster.emitSessionCancelled(sessionId, agentType);
EventBroadcaster.emitMessageCancelled(sessionId, agentType);
```

In `agent-event-handler.ts`, the `session.cancelled` handler calls `persistSessionCancelled` (sets session status to `cancelled`) and then calls `invalidate("session", ...)`. The `message.cancelled` handler calls `persistMessageCancelled`, which is a transaction that does an `INSERT` into `session_messages` AND updates the session. Two separate events trigger two separate DB writes — the second `persistMessageCancelled` call updates the same session row that `persistSessionCancelled` just updated. Whether this causes a correctness bug depends on the order of writes and what each sets, but emitting two events from a single cancellation path creates redundant, potentially conflicting DB writes and two separate WS invalidations. The old code did this in a single atomic transaction. If `message.cancelled` already sets session status, the `session.cancelled` event is redundant and should be removed.

---

**Bug 4 — `backend/src/services/tool-relay.ts`: Timeout cleanup race on `reject`**

In `tool-relay.ts`, `reject(requestId, error)` calls `pending.delete(requestId)` but does not call `clearTimeout(timer)` before deleting the entry. The timer still holds a reference to the `requestId` and will fire after the map entry is deleted. When the timeout fires it calls `pending.get(requestId)` which returns `undefined`, so the double-fire is silently swallowed — but the timer leak persists for the full timeout duration (likely 30s). Compare `resolve` which does clear the timer. The `reject` path should mirror `resolve`:

```typescript
// Missing in reject():
clearTimeout(entry.timer);
```

This is not a crash bug but causes the timer to leak on every rejected tool request.</result>
