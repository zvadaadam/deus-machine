### **#1 Zod schema for `pair_response` allows `success: true` without `deviceToken`**

The Zod schema at `apps/cloud-relay/src/schemas.ts:24-30` defines `pair_response` with `success: z.boolean()` and `deviceToken: z.string().optional()`. This accepts `{ type: "pair_response", pairId: "x", success: true }` with no `deviceToken`.

The TypeScript type in `shared/types/relay.ts:13-14` is a proper discriminated union where `success: true` **requires** `deviceToken: string`. The `as ServerFrame` cast at `relay-do.ts:179` bridges this gap at compile time, but at runtime Zod has already let through a frame without `deviceToken`.

When this reaches `relay-do.ts:300-301`:

`frame.deviceToken` is `undefined`, and `JSON.stringify` drops it — the pairer receives `{"type":"pair_success"}` with **no token**, silently breaking the pairing flow.

**Fix:** Split the Zod schema into two discriminated branches on `success`, or add a `.refine()` ensuring `deviceToken` is present when `success` is `true`.

### **#2 `switch/case` on discriminated union instead of `ts-pattern`**

`relay-do.ts:284` dispatches on `frame.type` (the `ServerFrame` discriminated union) using a native `switch`:

```ts

  case "pair_response": { ... }
```

CLAUDE.md rule: *"Use `ts-pattern` for switch/case and if/else chains on discriminated unions. Prefer `.exhaustive()` to catch missing cases at compile time."* and *"**When to use:** Any switch/case or if/else chain dispatching on `.type`, `.status`, `.state`, or similar discriminator fields."*

The switch also has no `default` case, so adding a new `ServerFrame` variant would be silently ignored at runtime. `ts-pattern` with `.exhaustive()` would catch this at compile time. Note: `ts-pattern` is not in `apps/cloud-relay/package.json` at all.
