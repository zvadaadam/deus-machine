# SILENT HANDSHAKE

**Philosophy:** A connection between two devices should feel as natural as handing someone a key. The technology disappears; only the intent remains — "I want to use this from there."

---

## Language System

Every word was chosen to eliminate cognitive load. Technical infrastructure must never surface in the interface.

| Kill this | Replace with |
|-----------|-------------|
| Relay / Portal / Server | *(nothing — never mentioned)* |
| Server ID | *(hidden, embedded in URL/QR)* |
| Remote access | **Link your phone** or **Connect a device** |
| Pairing code | **Connection code** |
| Paired devices | **Linked devices** |
| Revoke | **Remove** |

The word "pairing" itself is borderline. People pair Bluetooth headphones. They *link* their accounts, *connect* their devices. "Link" is warmer and more familiar.

---

## The Code Format

**Format: `BEAR-4521` — single field, no change to the code itself, but fundamental change to input.**

One field. One string. The user types or pastes `BEAR-4521` as a single token. The hyphen is a visual separator, not a field boundary.

**Why keep WORD-NNNN:**
- Verbal dictation: "bear forty-five twenty-one" — four syllables for the number, one word for the anchor. Humans chunk this naturally.
- Visual memorability: a word gives the code an identity. "The bear code." Pure numbers vanish from short-term memory.
- Copy-paste: a single string copies and pastes in one action. The current two-field design is the problem, not the format.

**Input field behavior:**
- Single text input, monospaced, centered, large (20px+ font).
- Placeholder text: `WORD-0000`
- Auto-uppercases as you type.
- Accepts with or without hyphen (`BEAR4521` or `BEAR-4521` or `bear-4521`).
- On paste: strips whitespace, normalizes case, auto-submits if valid format detected.
- On submit: if code is invalid or expired, the field shakes subtly (200ms, 4px horizontal, ease-out-quart) and shows inline text below: "Code not recognized. Check the code on your desktop and try again." — never "invalid code" (accusatory) or "expired" (confusing if they just read it off screen).
- No submit button visible until the field has content matching the pattern. Then a single `Connect` button appears with a fade (opacity 0→1, 150ms).

**Validation regex:** `/^[A-Z]{3,8}-?\d{4}$/` after normalization.

## Three Key Decisions

**1. The code IS the URL path — one artifact, not two.**

The current design requires sharing a URL AND a code separately. By embedding the code directly in the URL (`/c/BEAR-4521`), every sharing mechanism collapses to one action: scan QR, click link, or say code. The backend resolves the code to the relay connection — the server ID is an implementation detail that never surfaces. This is the single most impactful change. It eliminates the entire "which piece of information goes where?" confusion.

**2. QR code is the primary, not supplementary, element.**

Phone-scanning-desktop is the most common cross-device pairing scenario (cf. WhatsApp Web, Discord, Telegram). The QR code occupies the most visual weight in the sharing card because it should be the first thing the user reaches for. The text code and copyable URL are secondary paths for scenarios where QR doesn't work (remote sharing, accessibility, verbal dictation). Hierarchy reflects frequency of use.

**3. Single input field with intelligent paste handling.**

Two fields for one code is a hostile pattern. It breaks copy-paste, it creates tab-order confusion, and it forces the user to understand the code's internal structure. A single field that accepts `BEAR-4521`, `bear4521`, `BEAR 4521`, or even a full pasted URL eliminates every friction point. The field is generous in what it accepts and precise in what it transmits.

---

## One Acknowledged Tradeoff

**The word list is only 100 words, making codes partially guessable.**

With 100 words and 4 digits, the code space is 100 x 10,000 = 1,000,000 combinations. This is small enough that a determined attacker could brute-force it within the 15-minute TTL if there were no rate limiting. The tradeoff is: short, memorable, dictatable codes vs. cryptographic strength. The mitigations (one-time use, 15-min TTL, max 5 active codes, rate limiting on the pairing endpoint — which should reject after 5 failed attempts per IP per minute) make this acceptable for the threat model of a desktop IDE tool. But it is not suitable for high-security environments. If the word list were expanded to 1,000 words or the digit portion to 6, the code space grows to 10M-1B, but dictation becomes harder. I chose human usability over theoretical security margin, accepting that rate limiting is the real security boundary.

---

## Implementation Notes

The URL structure change (`/c/CODE` instead of `/connect/{serverId}?pair=CODE`) requires:
- A route that resolves the word-code to a server ID on the backend/relay side
- The `PairGatePage` to read the code from the path segment instead of (or in addition to) the query param
- The QR generation to encode the simplified URL

The single input field requires:
- Removing the current two-field component
- A normalization function: strip whitespace, uppercase, insert hyphen if missing between alpha and numeric portions, extract code from full URL if pasted
- Validation on the normalized result, not the raw input

Both are modest changes that dramatically simplify the user's mental model. The technology recedes. The intent — "connect this to that" — is all that remains.</result>

# Direction: **Quiet Handshake**

## Philosophy

A connection between devices should feel like two people making eye contact across a room — immediate, natural, requiring no explanation. The technology disappears; what remains is the simple act of one device recognizing another.

---

## Language System

Before anything else, we establish the vocabulary. Every word a user reads shapes their mental model.

| Technical term | User-facing term |
|---|---|
| Relay server | *(never exposed)* |
| Remote access | **Access from other devices** |
| Pairing code | **Connection code** |
| Server ID | *(never exposed)* |
| Portal | *(never exposed)* |
| Paired device | **Connected device** |
| Token | *(never exposed)* |
| Revoke | **Disconnect** |

The language is deliberately domestic. "Connect" and "disconnect" — verbs everyone knows from plugging in a cable.

## The Code Format

**Decision: Single unified code. Format: `WORD-NNNN` as one atomic string.**

The code remains `WORD-NNNN` (e.g., `OAK-3847`) but is treated as a single indivisible token everywhere — one field, one clipboard entry, one thing to say aloud.

Why this format works across all three scenarios:

- **Phone scanning desktop**: QR code encodes the full URL with code embedded. No typing needed.
- **Link shared via chat**: URL is `https://app.rundeus.com/connect/OAK-3847`. The code is in the URL. Click and done.
- **Verbal dictation**: "Oak three eight four seven" — one word, four digits. Under 3 seconds to speak. The word anchors memory (people remember "oak" while their fingers find the numbers).

**Why not pure numeric?** A 6-digit number (`384721`) is harder to dictate and harder to remember mid-transfer. The word provides a cognitive anchor. "Oak" sticks in working memory while the person types the digits.

**Why not two words?** Two words from a 100-word list gives only 10,000 combinations — same as WORD-NNNN, but harder to type (spaces? hyphens? which word was first?). The hybrid format optimizes for both entropy and usability.

### The Input Field — Detailed Design

**Single field. One input. No splitting.**

This is the most critical interaction design decision. The current two-field split (word + number) breaks copy-paste and creates confusion about which field has focus. A single field eliminates this entirely.

**Field specifications:**

- `rounded-lg border border-border bg-background px-4 py-3 text-center text-lg font-mono tracking-[0.1em] uppercase`
- Placeholder: `WORD-1234` in `text-muted-foreground/40` — shows the expected format without instruction
- The field auto-uppercases input (CSS `uppercase` + JS normalization on submit)
- `maxLength={9}` (longest: 5-letter word + hyphen + 4 digits = 10, but most are 8-9)
- `inputMode="text"` on mobile (we need letters, not just numbers)
- `autocomplete="off"`, `autocorrect="off"`, `spellcheck="false"` — prevent keyboard interference

**Paste handling:**
- On paste, the field accepts the full code (`OAK-3847`), the full URL (`https://app.rundeus.com/connect/OAK-3847`), or partial strings
- The paste handler strips the URL prefix if present, trims whitespace, normalizes to uppercase
- If the pasted value looks like a complete valid code (matches `/^[A-Z]+-\d{4}$/` after normalization), auto-submit after a 300ms delay — the user sees the field populate, a brief beat, then "Connecting..."

**Typing behavior:**
- As the user types, the hyphen is NOT auto-inserted. The user types it themselves. Auto-inserting punctuation is disorienting when you don't expect it, and the dictation scenario ("oak dash three eight four seven") naturally includes it.
- Real-time validation: the field border transitions to `border-primary` when the format matches `WORD-NNNN`. This is a subtle signal — "you've typed enough, you can submit." No error messages while typing.
- The "Connect" button is `variant="default"` (primary), full-width below the input. Disabled with `opacity-50` until the input matches the code format.

**On submit (click Connect or press Enter):**
- The button text changes to a small spinner + "Connecting..." (the button doesn't disappear — layout stability)
- If the code is invalid: the input field does a subtle `translateX` shake animation (3px left, 3px right, 2x, 200ms total) and the border turns `border-destructive` briefly. Below the field, red text appears: "Code not recognized. Check the code and try again." — `text-sm text-destructive`. No field clearing — the user can edit what they typed.
- If the code is expired: same shake, but message says "This code has expired. Ask for a new one from the desktop app."
- If network error: "Unable to connect. Check your internet and try again." with a "Retry" button that appears inline.

## Three Key Decisions

### 1. QR code is the primary path, not an afterthought

The QR code occupies the visual center of the connection sheet. It's the largest element. This is intentional: the phone-scanning-desktop scenario is by far the most common real-world pairing flow (AirDrop, Spotify Connect, WhatsApp Web all validated this). The link and manual code are presented as secondary alternatives below dividers. By making QR dominant, we optimize for the 80% case while keeping the 20% cases accessible.

### 2. Single input field, auto-submit on valid paste

The current two-field design (word + number) exists because the developer who built it was thinking about validation, not about the user's hands. A person pasting a code from a chat message hits Cmd+V and expects it to work. Two fields break this completely — the paste lands in whichever field has focus, corrupting the input. A single field with smart paste handling (strip URLs, normalize, auto-submit on valid match) means the paste-from-chat scenario is literally one action: Cmd+V. Done. The 300ms delay before auto-submit gives the user a moment to see what was pasted — preventing the unsettling feeling of something happening before you understood it.

### 3. The connection sheet, not a page

Pairing is a *momentary action*, not a *destination*. A full page implies this is a place you spend time. A sheet (or dialog) layered over Settings communicates: "this takes 10 seconds, then you're done." The sheet pattern also means the desktop user doesn't lose context — Settings is still visible behind the overlay, and dismissing the sheet returns them exactly where they were. The connection success state transforms within the same sheet rather than navigating away, reinforcing that this was a quick aside, not a journey.

---

## One Acknowledged Tradeoff

**The 15-minute TTL is short enough to cause friction but necessary for security.**

A user who generates a code, gets distracted, then tries to share it 20 minutes later will hit an expired code. We mitigate this with: (a) the visible countdown timer, (b) effortless regeneration ("Generate new code" is one click, no confirmation), and (c) the remote-side expired state gracefully directs the user to ask for a new code. But we *don't* extend the TTL. One-time codes with short lifetimes are the security foundation — extending them would create a class of semi-valid codes floating in chat histories and screenshots that could be replayed. The tradeoff is: occasional re-generation friction in exchange for codes that are genuinely ephemeral. The interaction cost of regenerating (~2 seconds) is low enough that this is the right call.

# "Chromatic Handshake"

## Philosophy Statement

Pairing two devices is an act of trust between strangers — your desktop inviting an unknown screen into its world. Color should perform the emotional labor of that introduction: warm amber signals readiness, a living gradient pulse says "I'm alive and waiting for you," and a shared color convergence at the moment of connection tells both sides "we found each other" without a single word of technical jargon. The palette is the protocol.

## The Code Format: Unified Single Input

### The Problem with WORD-NNNN Split Input

The current `PairGatePage` uses two `<Input>` fields separated by a dash. This breaks three critical scenarios:

1. **Copy-paste**: Users cannot paste `BEAR-4521` into the form because the fields are separated.
2. **Verbal dictation**: Saying "bear dash four five two one" is awkward; the dash is ambiguous.
3. **QR auto-submit**: Works only because the URL param parsing reconstructs the full code.

Keep the `WORD-NNNN` generation format in the backend (no backend changes needed). Change the input to a single field that:

- Accepts the full code including the dash: `BEAR-4521`
- Auto-uppercases as you type
- Visually separates word and number using **color** instead of a physical divider
- Auto-submits when the pattern matches `/^[A-Z]+-\d{4}$/`


- **Visual chunking**: The eye naturally groups by color, making the code easier to read and dictate.
- **Error prevention**: If someone types only letters or only numbers, the missing color segment is immediately apparent.
- **Brand identity**: The amber-rose split becomes a recognizable pairing motif.

On the **remote input side**, the single input field uses the same color treatment: as the user types letters, they render in amber; once the dash appears, subsequent digits render in rose. This creates a sense of the two devices speaking the same chromatic language.

## QR Code Design

The QR code encodes: `https://app.rundeus.com/connect/{serverId}?pair={CODE}`

### The Glow

Behind the QR code, a `::before` pseudo-element creates a radial glow:

## 3 Key Decisions and Why

### Decision 1: Single Input Field Replacing Split Input

**Why:** The split input (two fields with a dash between them) is the single biggest UX friction point. It prevents copy-paste entirely. A user who receives `BEAR-4521` via text message must mentally parse it, type "BEAR" in one field, tab to the next, then type "4521." With a single field, they paste the whole string and it auto-submits. The color-segmented rendering (amber word, rose numbers) provides the same visual chunking that the split input was trying to achieve, but without fragmenting the interaction.

This requires changes only to `PairGatePage.tsx` — the backend format, validation, and relay protocol are untouched.

### Decision 2: QR Code as Hero Element (Not URL)

**Why:** The current AccessSection treats the URL as the hero (PortalCard with URL + copy/open buttons) and the pairing code as a secondary step. This is backwards. The URL is meaningful only to technical users who understand what `https://app.rundeus.com/connect/abc123` means. For the target user (semi-technical, cares about output not technology), scanning a QR code is the fastest, most intuitive action. Making the QR the visual anchor — with the amber glow, generous size, center positioning — means the most common scenario (phone scanning desktop) is served first. The URL and manual code become fallback options, still accessible but not competing for visual priority.

The QR encodes the full pairing URL including the code (`?pair=BEAR-4521`), so scanning it completes the entire pairing flow with zero typing. The current infrastructure already supports `?pair=CODE` auto-submit.

### Decision 3: Color as the State Indicator (Not Text Badges or Icons)

**Why:** The current design uses text ("Connected to relay" / "Connecting...") and a small colored dot to communicate state. This forces the user to read. In the new design, the ambient color of the pairing card — its border, glow, and background tint — shifts with state. The user does not need to read "Connected" when the entire card is radiating warm amber. They do not need to read "Paired" when the glow flashes green. Color operates below conscious attention; text requires conscious parsing. For a feature that the user interacts with rarely (pair once, forget), the color signal is faster and more memorable.

This approach uses only `opacity` and `background-color` transitions — GPU-accelerated, 60fps, no layout thrashing.

## 1 Acknowledged Tradeoff

**Color-segmented input text requires custom rendering logic.** A standard `<input>` element cannot render part of its value in amber and part in rose. Implementing the color-split display requires either:

(a) A hidden input with a visible overlay `<span>` that parses and colors segments — adds DOM complexity and requires careful cursor/selection sync.

(b) A `contentEditable` div with color spans — historically fragile on mobile browsers.

(c) Accepting a simpler version: the input text is monochrome (the standard `font-mono` treatment), and only the *display* of the code on the desktop side uses the color-split rendering. The phone input is a plain single field.

**I recommend option (c) for v1.** The emotional color journey (background gradient shifts, glow, button color) carries the chromatic weight on the phone side. The color-segmented text is a refinement for the desktop display (where it is rendered as static `<span>` elements, trivially implemented). Pursuing option (a) or (b) for the input field adds implementation risk disproportionate to its UX value, especially on mobile where custom text rendering has a long history of subtle bugs with virtual keyboards, autocorrect, and paste handling.

The phone input remains a single monochrome field with `text-center font-mono text-xl tracking-[0.15em] uppercase` — clean, generous, and functional. The color tells its story through the environment, not through the text itself.</result>

```typescript
export function validatePairCode(code: string): boolean {
  // Normalize: lowercase, collapse whitespace, treat dashes/underscores as spaces,
  // then trim to canonical "word word" format
  const normalized = code
    .toLowerCase()
    .trim()
    .replace(/[-_]+/g, " ")   // treat dashes/underscores as word separators
    .replace(/\s+/g, " ");     // collapse multiple spaces
  
  const entry = activeCodes.get(normalized);
  if (!entry) return false;
  if (entry.expiresAt <= Date.now()) {
    activeCodes.delete(normalized);
    return false;
  }
  activeCodes.delete(normalized);
  return true;
}
```

## Three Key Decisions

**1. Two lowercase words instead of WORD-NNNN.** Every pain point traces to the code format. Two words are one cognitive chunk, one keyboard mode, one speech pattern. Same entropy. Categorically better physical ergonomics.

**2. All three methods visible simultaneously, no tabs.** The QR code, text code, and copy-link fit in a single card (~200px height). Showing all three costs nothing and eliminates the cognitive overhead of choosing between tabs. The user's body already knows which method they'll use.

**3. No countdown timer.** The 15-minute TTL exists for security, but displaying it creates urgency where none belongs. Pairing takes under 15 seconds. The failure case (code expires) is handled by clicking "Generate Code" again -- a trivial 1-second recovery that doesn't justify polluting every normal pairing with a ticking clock.

---

## One Acknowledged Tradeoff

**Two-word codes from a 100-word list produce ~10,000 combinations.** With max 5 active codes and 15-minute TTL, collisions are effectively impossible at desktop-app scale. But this does not scale to thousands of concurrent pairing attempts. If OpenDevs grows to that scale, the word list must expand (500 words -> 250K combinations) or a third word must be added (1M combinations, at the cost of slightly more typing). The design deliberately optimizes for today's real usage over hypothetical future scale.

## THREE KEY DECISIONS

**1. QR code as primary, code as secondary, URL as tertiary.**

The QR code encodes the full URL with the pairing code embedded. Scanning it means zero typing, zero copy-paste, zero ambiguity. The code exists for verbal dictation and fallback. The URL exists only in the "Copy Link" button for chat-sharing. This hierarchy is reflected in the visual weight: QR is largest (180x180px in a padded container), code is medium (24px monospace), URL is never displayed as text (only copied).

**2. Single input field with intelligent normalization.**

The split input in the current `PairGatePage` (two `<Input>` components, lines 210-235) is the root cause of the UX failure. It breaks paste, confuses tab order, and adds complexity (the `handleWordKeyDown` handler on line 177 tries to intercept `-` and `Tab` to advance fields -- clever but fragile). One field that normalizes all reasonable inputs eliminates every edge case.

**3. Eager code generation on dialog open.**

The current flow requires two clicks: "Generate Code" then copy/dictate. The new flow generates a code when the dialog opens. Since codes are in-memory with 15-min TTL and max 5 active (lines 35-36 of `remote-auth.service.ts`), the cost is negligible. The user opens the dialog, the QR code and text code are immediately visible. One click, not two.

**The QR code requires a new dependency and consumes significant dialog space.**

A 180x180px QR code inside a padded container consumes roughly 250px of vertical space. On the 480px-wide dialog, this means the text code and copy buttons are pushed below the visual center. Users on shorter displays or scaled-up UI may need to scroll. I accept this because the QR path eliminates all friction for the most common scenario (phone scanning desktop screen), and because the `qrcode` package adds minimal bundle size (~15KB gzipped for SVG generation). The alternative -- making the QR small or optional -- optimizes for the less common path at the expense of the primary one.
