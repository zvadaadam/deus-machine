## Issue 2: Unconsumed redirect responses in desktop.ts

**VALID (negligible practical impact)**

Reading `apps/cli/src/desktop.ts`:

- **`downloadFile()`** (lines 226-230): On a 301/302, the code calls `download(loc)` recursively but does not call `res.resume()` or `res.destroy()` on the original response. The response body (which is typically tiny for a redirect -- just an HTML stub) sits unconsumed.
- **`fetchRelease()`** (lines 195-203): Same pattern. On redirect, `get(loc, ...)` is called but the original `res` is neither consumed nor destroyed.

In Node.js, an unconsumed `IncomingMessage` stream holds the underlying TCP socket open until it times out or the GC collects it. This is technically a resource leak.

However, the practical impact is negligible:
- This code runs once during a `deus desktop` install command, not in a long-running server.
- GitHub API redirects produce at most 1-2 redirects per request.
- The socket will be cleaned up when the process exits (which happens shortly after download completes).
- At most 2-3 sockets leak for the lifetime of a short-lived CLI command.

A senior engineer would acknowledge this as a correctness issue worth fixing (add `res.resume()` before the recursive call) but would not consider it a real bug in practice given the usage context.

## Issue 3: Shutdown kills children then immediately exits

**VALID (moderate severity)**

The `shutdown()` function at lines 90-103 of `apps/cli/src/start.ts`:

```typescript
function shutdown() {
    if (status) status.stop();
    clearServerInfo();
    // ... print goodbye message ...
    for (const child of children) {
      if (!child.process.killed) child.process.kill("SIGTERM");
    }
    process.exit(0);
}
```

It sends SIGTERM to both child processes and then immediately calls `process.exit(0)` on the very next line. The children get zero time to handle the signal.

The agent-server has explicit graceful shutdown logic (lines 123-147 of `index.ts`): it closes the HTTP server, waits for in-flight turns to drain (up to 30 seconds), and then cancels remaining sessions. By calling `process.exit(0)` immediately, the parent process exits, and while the children technically continue running momentarily as orphans, the stdout/stderr pipes from the parent are now closed. In practice, the children likely still complete their shutdown since they don't depend on the parent's pipes, but:

1. The CLI process exits before children finish, so the user's terminal returns to the prompt while child processes are still cleaning up. This is a UX issue but also a correctness issue -- cleanup may not complete.
2. If the OS reaps orphaned children aggressively (or the user runs `deus start` again quickly), the old agent-server and backend may still be holding ports.
3. The `clearServerInfo()` call happens before children exit, so `deus status` would report "not running" while processes are still alive.

A proper fix would be to send SIGTERM, then `await` child `exit` events (with a reasonable timeout like 5-10 seconds), then call `process.exit`. This is a real bug that a senior engineer would want fixed.
