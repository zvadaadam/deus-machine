2. Provider Adapter Registry

What We Have Today
Our agent handler interface is actually pretty clean:

interface AgentHandler {
  readonly agentType: AgentType;
  initialize(): { success: boolean; error?: string };
  handleQuery(sessionId, prompt, options): Promise<void>;
  handleCancel(sessionId): Promise<void>;
  handleReset(sessionId): void;
}
Registration is a simple Map<AgentType, AgentHandler>. Two implementations: ClaudeAgentHandler and CodexAgentHandler. Dispatch is getAgent(agentType) → O(1) lookup.

What OpenDevs Does Differently
T3’s ProviderAdapterShape is more granular:

interface ProviderAdapterShape {
  startSession(input): Effect<ProviderSession>;
  sendTurn(input): Effect<TurnResult>;
  interruptTurn(threadId, turnId?): Effect<void>;
  respondToRequest(threadId, requestId, decision): Effect<void>;  // approval flow
  respondToUserInput(threadId, requestId, answers): Effect<void>;  // user prompts
  stopSession(threadId): Effect<void>;
  readThread(threadId): Effect<ThreadSnapshot>;
  rollbackThread(threadId, numTurns): Effect<ThreadSnapshot>;      // undo!
  readonly capabilities: { sessionModelSwitch: boolean };           // explicit feature flags
  readonly streamEvents: Stream<ProviderRuntimeEvent>;              // unified event stream
}
Key things they have that we don’t:

Explicit capabilities — providers declare what they support at compile time
Rollback/undo — rollbackThread(threadId, numTurns) is part of the contract
Approval flow — respondToRequest for tool approvals is a first-class concept
Unified event stream — all provider events flow through one typed stream
What Would Change
Our handler interface needs two additions:

interface AgentHandler {
  readonly agentType: AgentType;
  readonly capabilities: AgentCapabilities;  // NEW: what can this agent do?
  
  initialize(): { success: boolean; error?: string };
  handleQuery(sessionId, prompt, options): Promise<void>;
  handleCancel(sessionId): Promise<void>;
  handleReset(sessionId): void;
  handleApproval?(sessionId, requestId, decision): Promise<void>;  // NEW: tool approval
}

interface AgentCapabilities {
  multiTurn: boolean;        // Claude: yes, Codex: no
  modelHotSwap: boolean;     // Claude: yes, Codex: no
  sessionResume: boolean;    // Claude: yes (agent_session_id), Codex: no
  toolApproval: boolean;     // both: yes
  rollback: boolean;         // neither currently, but door is open
}
The bigger issue is message format. Right now Codex events are mapped to Claude content blocks in the Codex handler (lines 38-121 of codex-handler.ts). This means our message schema is Claude-shaped, not provider-neutral. If we add Gemini, we’d be mapping everything to Claude’s format — which is fragile.

A cleaner approach: define a provider-neutral message envelope and let each handler emit that:

type ProviderMessage = {
  role: 'assistant' | 'tool_result';
  blocks: ContentBlock[];  // our own union, not Claude's
  model: string;
  provider: AgentType;
}
Verdict: The registry pattern itself is fine — our Map<AgentType, AgentHandler> works. What we need is explicit capabilities and a provider-neutral message format. We don’t need Effect-TS or DI containers — that’s overkill for 2-3 providers. But the interface should be richer.
